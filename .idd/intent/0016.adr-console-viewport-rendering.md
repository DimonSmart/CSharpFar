# 0016 - Viewport-bound console rendering

Type: ADR
Status: Accepted

Related:
- 0002
- 0004

## Context

The buffered rendering strategy from `0002` reduces flicker by building a full
logical frame from the application-owned UI buffer and flushing only changed
horizontal runs. Windows console resize adds a separate failure mode: the
visible viewport can move within the console screen buffer while width and
height remain unchanged.

If writes or cursor placement use stale viewport coordinates, Windows can
scroll the buffer or corrupt the bottom function-key row. A front buffer keyed
only by width and height can also claim that cells are current when the viewport
origin has moved.

## Decision

Use viewport-bound buffered rendering for physical output.

`IConsoleDriver.GetViewport()` returns a `ConsoleViewport` with `Left`, `Top`,
`Width`, and `Height`. On Windows, the snapshot comes from
`GetConsoleScreenBufferInfo.srWindow`. On Unix-like ANSI terminals, the
application viewport origin is `0,0` inside the alternate screen and size comes
from the terminal window. Size-only layout helpers may still use a width/height
view when they do not need physical origin.

`ScreenRenderer.BeginFrame()` captures one viewport snapshot for the whole
frame. The front buffer is valid only for the same viewport. If width, height,
left, or top changes, the known front buffer is invalidated and the next
successful frame is sent as a full redraw.

`FlushFrame()` checks that the current viewport still matches the frame viewport
before physical writes, uses viewport-bound physical writes, and checks again
before committing the back buffer as the new front buffer. If the viewport
changed, the frame is interrupted, the front buffer is reset, and the caller
renders again.

Pending cursor position is applied only after a successful flush. Cursor
placement is also viewport-bound and is skipped if the current viewport no
longer matches the frame viewport.

`Application` renders visible UI through a loop that retries interrupted frames
and drains stale resize events without dropping non-resize input.

The console driver does not keep parallel APIs for independent viewport origin
or absolute cursor placement. The viewport-aware contract is the single path for
physical coordinate work. Backends render the shared logical buffer; Windows may
use Win32 console APIs as an implementation detail, while Unix-like backends use
ANSI/VT cursor movement, SGR style changes, and alternate-screen mode.

## Consequences

- Frame buffers describe a specific visible viewport, not only a size.
- Origin-only viewport movement forces a full redraw.
- Writes and cursor movement cannot commit a frame into the wrong viewport.
- UI renderers continue to use logical coordinates and do not manage dirty
  regions.
- Some frames are intentionally discarded during active resize.
- Tests model viewport identity rather than only console size.
- Unix-like terminals do not provide a reliable `ReadConsoleOutput` equivalent,
  so renderer correctness is based on the application-owned buffer and emitted
  ANSI/VT output.
