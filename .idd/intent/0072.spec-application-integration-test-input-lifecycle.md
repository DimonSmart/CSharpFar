# 0072 - Application integration test input lifecycle

## Intent

CSharpFar application integration tests schedule scripted input according to UI
lifecycle boundaries. Input intended for the main application loop is not queued
while a modal dialog, progress overlay, temporary interactive surface, or other
input-owning layer can consume it.

## Related Specifications

- 0062.spec-console-ui-component-library.md
- 0067.spec-ui-layer-composition-and-modal-resize.md

## Behavior

Application integration tests may script user actions that open synchronous
dialogs, run timed modal progress UI, or trigger temporary interactive surfaces.
When a scripted action is intended for the main application loop after such UI
has completed, the test waits for an explicit main-loop readiness boundary
before queuing that action.

The readiness boundary means the application loop has finished handling the
previous top-level input, any synchronously nested modal UI created by that
input has closed, blocking temporary UI created by that command has completed,
and the application loop is about to read the next top-level input.

Tests do not infer that boundary from elapsed time, render output, absence of
modal layers, UI layer counts, disappearance of text, or extra duplicate keys.
Modal layers continue to isolate lower UI layers; test orchestration must not
depend on modal input falling through to the application.

## Durable Architecture And Constraints

The application runtime exposes an internal test-observable main-loop readiness
notification. The notification is the authoritative boundary for scheduling the
next scripted input that belongs to the main application loop.

The notification is scoped to the top-level application loop. Modal dialog
runners, editor or viewer loops, temporary interactive surface runners, and
fake console read hooks do not publish this boundary.

Production behavior does not change when no test subscriber observes the
notification, and the application does not add public test-only API for this
contract.

## Non-Goals

- Compatibility behavior for old tests that enqueue duplicate final exit keys.
- Passing unhandled modal input through to lower application layers.
- Timing-based, render-text-based, or polling-based exit synchronization.

## Acceptance Criteria

1. Scripted application integration tests can place initial dialog input before
   the application starts and defer later main-loop input until the readiness
   boundary is observed.
2. A final exit key for the main application loop is not queued while a command's
   modal progress UI can consume it.
3. The first readiness notification before the first input read does not release
   a later scripted phase while earlier queued input remains pending.
4. Tests that need mid-modal interaction may still enqueue that modal-specific
   input from deterministic modal UI evidence, but main-loop exit remains tied
   to the runtime readiness boundary.

## Verification

Automated coverage verifies that the runtime readiness notification fires before
top-level input reads and can be used to enqueue the next top-level input.

Automated coverage verifies that scripted barriers are not satisfied while
previously queued input remains pending.

End-to-end file-operation and search-progress scenarios verify that final
application exit input is consumed by the main application loop after modal work
has completed, not by progress or search modal UI.
