# 0061 - Native plugin system

Related:
- 0029
- 0035

## Goal

CSharpFar supports native .NET plugins designed around CSharpFar internal
abstractions.

CSharpFar intentionally does not emulate FarNet, Far Manager plugin APIs, or
binary compatibility with existing managed or native Far Manager plugins.

FTP and SFTP are first-party plugins that use the same native plugin API shape
as external plugins. Their network behavior and saved connection formats remain
owned by specs `0029` and `0035`.

## Scope

The native plugin system provides a small, CSharpFar-owned extension surface:

1. register a command;
2. register a menu item;
3. register a panel source or panel provider;
4. access the current application context;
5. access a settings/storage service;
6. access UI services for dialogs and messages.

The first API version exists to support FTP and SFTP migration without
redesigning their protocol logic.

Preferred terminology:

```text
Plugin
PluginHost
PluginContext
PluginDescriptor
PluginCommand
PluginMenuItem
PanelProvider
```

The API must not use FarNet, Far Manager, compatibility, shim, or adapter names.

## Behavior

Plugins initialize through a host supplied by CSharpFar. During initialization a
plugin may register command descriptors, menu item descriptors, and panel
providers.

Commands execute with a command context that exposes application state,
settings/storage, and UI services. Command registration must use stable plugin
owned ids.

Menu items reference registered commands. The main application decides where
plugin menu items are rendered and how unavailable commands are presented.

Panel providers open `IFilePanelSource` instances or the current native panel
source equivalent. Panel providers must preserve source identity so file
operations never confuse remote provider paths with local filesystem paths.

Settings/storage access is scoped by plugin id. Plugins must not write arbitrary
files under the application configuration root without going through the plugin
settings service.

UI access is intentionally small: messages, simple text input, menus, and
protocol-specific dialogs supplied by first-party plugins. A broad external UI framework
for third-party plugins is not part of the first version.

## FTP and SFTP migration

FTP and SFTP remain product functionality. They must not be deleted as part of
removing the old compatibility layer.

The target state is:

1. FTP and SFTP register a connection-manager command;
2. FTP and SFTP register Plugins menu entries;
3. FTP and SFTP register panel providers for their remote file systems;
4. FTP and SFTP use plugin-scoped settings/storage for saved connections;
5. FTP and SFTP use plugin UI services for dialogs and messages.

The migration must not rewrite FTP/SFTP network behavior, saved connection
formats, credential semantics, or file operation semantics unless the owning FTP
or SFTP spec changes.

## Non-goals

Do not implement in this spec:

1. FarNet API support;
2. Far Manager native plugin API or ABI compatibility;
3. an adapter from FarNet modules to CSharpFar plugins;
4. editor extensions;
5. viewer extensions;
6. background tasks;
7. plugin marketplace or package acquisition;
8. hot reload;
9. scripting;
10. macro compatibility.

## Acceptance criteria

1. No production project implements or references a FarNet compatibility layer.
2. No solution or project file references `CSharpFar.FarNetHost` or
   `CSharpFar.FarNetCompat`.
3. FarNet compatibility tests and FarNet sample modules are absent.
4. A native plugin abstractions project or equivalent neutral abstraction layer
   exists with command, menu item, panel provider, context, settings, and UI
   extension points.
5. FTP and SFTP source directories remain in the repository.
6. If FTP/SFTP registration is temporarily disabled during migration, the
   reason is explicit: re-enable after the native CSharpFar plugin API is
   introduced.
7. Documentation does not promise FarNet or Far Manager plugin compatibility.

## Verification

Automated checks:

1. Project graph contains no `CSharpFar.FarNetHost` or
   `CSharpFar.FarNetCompat` references.
2. Repository search for FarNet/FarmNet terms finds no production code or
   compatibility tests.
3. Plugin abstraction types compile and expose only native CSharpFar names.
4. Existing FTP/SFTP tests remain enabled unless a test is explicitly skipped
   for native plugin API migration.

Manual checks:

1. Open the Plugins menu and verify first-party FTP/SFTP entries are the only
   network plugin entries until external native plugin loading is implemented.
2. Verify README and docs describe native CSharpFar plugins and no FarNet
   compatibility.
