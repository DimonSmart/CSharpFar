## Goal

Application-owned console UI is rendered through one shared composition host so
that a viewport change redraws the active full-screen surface and every modal
overlay as one current frame. Closing a layer rebuilds the UI from state rather
than restoring pixels captured from an earlier viewport.

## Layer model

- A **surface** is an opaque full-screen UI: the main application screen,
  `HiddenCommandLine` surface, viewer, editor, or full-screen help.
- Surface lifecycle and layer identity are separate roles. A surface lifecycle
  owns `BeginFrame` and `CompleteFrame`; the associated `IUiLayer` owns
  rendering, committed frames, interaction frames, and routed input. A surface
  object that also implements `IUiLayer` may use itself for both roles; otherwise
  infrastructure supplies an explicit layer or a render-only adapter.
- Surfaces and overlays share one ordered layer stack. The last surface in that
  stack is active; earlier surfaces and every overlay below that active surface
  are not drawn.
- An **overlay** is drawn only when it is above the active surface in the same
  stack. Eligible overlays are rendered in stack order and support nesting.
- Surface and overlay lifetimes are explicit disposable LIFO scopes. Repeated
  disposal is safe; invalid out-of-order disposal is rejected deterministically.
- A layer is a render participant and may also expose a routed-input contract.
  Existing surfaces and overlays that have not been migrated to interactive
  behavior may participate through render-only adapters with no input policy.
- Visual kind and input policy are separate concepts. Surface or overlay
  placement controls composition order; input policy controls whether a layer is
  skipped, bubbles unhandled input downward, or blocks lower layers as modal.
- Application-owned transient overlays above the main application surface are
  persistent overlay layers in the shared stack. Command-line completion is
  below panel quick search; panel quick search is below the top menu; modal
  dialogs and other temporary overlays opened later are above those application
  overlays. Rendering order and routed-input priority come from this one stack.

## Composition and frames

- The application owns one shared composition lifecycle and one associated modal
  lifecycle. They are created during bootstrap and passed through normal
  dependencies; UI components do not use static service lookup or local hosts.
- The shared composition lifecycle exclusively owns stable-frame retry, viewport
  change detection, resize-event consumption, and rendering the active surface
  followed by the visible overlays. It rebuilds the complete composition when a
  frame is interrupted and records a viewport only after a stable frame.
- Layers derive layout from the viewport supplied by the current frame. They do
  not open nested frames or reread console size for the current layout. Mouse
  hit testing uses layout from the most recent stable render.
- Application overlay hit testing uses the committed frame of the layer that
  receives the input. Overlay input handlers do not recompute input geometry
  from current console size, prior renderer bounds, or stale application UI
  state.
- The topmost rendered layer determines cursor state. On overlay close the
  composition redraw restores the cursor from the remaining top layer.
- Editor cursor ownership travels through committed focus metadata. Ordinary
  cursor placement is applied by the composition frame, not directly by editor
  rendering. When the editor cursor is over a wide character, the physical
  cursor is hidden and the renderer draws the composed cursor cell; blink
  changes are scheduled with the interactive surface host timed-wake contract
  only while such a composed cursor is needed. A wake proposes a new blink phase
  and requests rendering, and the phase becomes input-visible only after the
  frame commits. User input restores the visible phase and restarts the next
  blink deadline. Modal dialogs above the editor own the top cursor while they
  are open, and closing them restores the editor cursor through the next
  composition render.
- Interactive layers publish frame-dependent input metadata only through stable
  frame commit. Each interactive layer exposes an immutable committed
  interaction frame for the last successful render attempt. The committed focus
  frame is the focus portion of that same interaction frame, and cursor metadata
  travels through the committed focus entries.
- Surface-level state that depends on a rendered frame is synchronized through
  `UiLayer<TFrame>.OnFrameCommitted` for the owning layer. Interactive surface
  runners do not provide an alternate committed-frame callback, so initial
  renders, ordinary post-input renders, automatic resize recovery renders,
  viewport recovery without an explicit resize event, rejected-attempt retries,
  and renders initiated by another composition participant all publish state
  through the same stable-frame commit lifecycle.
- A rejected render attempt does not change the committed layer frame, focus
  frame, interaction frame, hit regions, cursor metadata, committed callbacks,
  or any input-visible frame state.
- For interactive form layers, row-owned dropdown component frames are
  input-visible frame state. A successful render commit publishes the dropdown
  field geometry, popup geometry, content geometry, scrollbar geometry, visible
  row count, selected index, scroll position, open-or-closed state, and the
  cancel baseline for any temporary popup selection together with the form
  frame.
  A rejected render attempt preserves the previous committed dropdown component
  frame and must not alter routed dropdown geometry, list state, open state,
  cancel baseline, or scrollbar drag lifecycle. The next input after a rejected
  render uses exactly the previously committed dropdown lifecycle state for
  every input target in the form layer, including outside clicks, other rows,
  footer buttons, focus traversal, and programmatic focus requests. Mutable
  component open state, temporary selection, and drag state from a rejected
  render attempt are not sources of routing ownership, outside-click, focus, or
  submit decisions.
- The built-in editor publishes an immutable committed editor frame after each
  accepted render. That frame contains the viewport, screen size, header,
  content, status-bar and function-key bounds, content dimensions, committed
  top line and left column, visible line range, scrollbar bounds and scroll
  state when interactive, rendered function-key actions and hit regions, cursor
  placement metadata, syntax diagnostics for the visible range, and redraw-event
  data. Editor mouse hit testing, text-coordinate mapping, scrollbar
  interaction, and function-key clicks use only this committed frame.

## Focus and routed input

- Each interactive layer instance owns one independent focus scope. The same
  interactive layer instance cannot be registered more than once in the active
  composition stack. Registration uniqueness is evaluated against the real
  `IUiLayer` identity, so the same layer cannot be registered twice as a
  surface, as both surface and overlay, or through multiple lifecycle wrappers.
  The layer is the only owner that commits its focus frame. Child components
  publish focus and cursor metadata through their interaction frame and never
  own a fallback focus scope.
  Focus targets are stable only within their owning layer; there is no global
  target registry.
- Focus traversal is layer-local and driven by the layer's committed focus
  frame. Input handlers request focus changes through routed input results; the
  composition host applies those requests to the source layer's focus scope.
- A semantic handler may return a post-dispatch focus request without mutating
  the focus scope. The owning layer keeps it pending until the next stable
  interaction-frame commit. A rejected render does not consume it; an absent or
  disabled target is ignored; a successful form target request makes that
  target visible.
- Interactive form focus transitions cancel row-owned transient overlays before
  the new focus is committed. A row-owned overlay may remain open only while
  its row owns focus; committed interaction frames must not publish hidden open
  popup or overlay-scrollbar targets for unfocused rows. For form dropdown rows,
  focus-transition cancellation is evaluated after the row has been restored to
  the lifecycle recorded in the committed form frame, so a rejected selection or
  rejected close cannot leak into the new focused target.
- Routed input uses only the active composition: the topmost visible overlay is
  routed first, lower visible overlays follow in reverse visual order, and the
  active surface is last. Earlier surfaces and overlays below the active surface
  do not receive routed input.
- A layer with no input policy is skipped and does not block routing. A bubble
  layer can stop routing by handling input. A modal layer stops routing to lower
  layers even when it reports input as unhandled.
- Rendering and input pumping cannot be started from inside routed dispatch.
  Domain dialog actions that may open, close, render, or read layers run only
  after routed dispatch has completed.
- Modal sessions and interactive temporary surface sessions use one shared
  composition packet pump for blocking and polling reads. The pump first
  consumes any pending routed packet for the owner, then reads semantic input
  through the shared composition lifecycle, dispatches it through routed input,
  and checks the owner packet again before rendering. If dispatch creates the
  owner's packet, the packet is returned without an intermediate render; if no
  owner packet appears but dispatch requested invalidation, the pump renders the
  composition before reading more input. Invalidation requested by a higher
  layer therefore is not lost while a lower modal or temporary surface session
  is waiting for its own packet.
- Timed interactive modal sessions use the same composition packet pump. The
  pump returns an already-formed routed packet before considering a wake
  deadline. When a wake deadline occurs without an owner packet, the modal
  runner invokes the dialog wake handler outside routed dispatch and outside a
  composition render transaction, passing the last committed modal frame. A
  wake may request no change, request one render, or complete the modal
  lifecycle without a synthetic input event. Wake-prepared frame-dependent state
  becomes input-visible only through the same accepted stable-frame commit path
  used by initial render, post-input render, resize recovery, and retry after a
  rejected frame. A rejected wake-triggered render publishes none of its pending
  frame state.
- Ordinary, interactive, and timed modal sessions, and interactive temporary
  surfaces, share this input-pump lifecycle for blocking read, non-blocking
  read, cancellation, and timed or external wake. A wake is a lifecycle signal,
  not synthetic console input, and cannot consume unread input intended for the
  composition that becomes active next.
- Routed input is separate from console input reading and resize recovery.
  Dialog and application layers may accept input during routed dispatch, but
  domain command processing runs only after the dispatch has completed.
- The main application surface is an interactive bubble layer. It accepts
  keyboard, modifier, and mouse input for the committed application frame, stores
  at most one pending routed packet, and does not execute application commands,
  legacy routers, terminal transitions, modal actions, or rendering from
  `RouteInput`.
- The main application surface publishes the command line as a stable local
  target named `application.command-line` in both `Panels` and
  `HiddenCommandLine` workspace modes. The target's hit region, cursor metadata, and
  text-coordinate mapping come from the same committed application frame that
  was rendered to the user.
- Application keyboard and modifier input is delivered only as an application
  routed packet from the main application surface. The routed packet records
  the selected local target, route kind, and committed application frame before
  any application command or domain transition runs.
- Command-line mouse input is routed to the command-line target when it hits the
  committed command-line row. Command-line hit testing and cursor placement use
  only committed frame geometry and do not reread console size or recompute a
  separate render layout.
- Command-line selection dragging uses composition mouse capture owned by the
  command-line target. Captured movement and release are routed back to that
  target, including pointer coordinates outside the command-line bounds; those
  captured events do not fall through to panel, scrollbar, directory shortcut,
  or function-key handling.
- The production runtime never calls application keyboard handling directly
  with raw `ConsoleKeyInfo`. Keyboard, modifier, and mouse input for the main
  application surface are handled after routed dispatch by consuming the
  application routed packet. An unhandled routed result is not an application
  fallback signal. Input handled by an overlay or accepted by a modal layer
  does not reach application keyboard or mouse handling.
- The production runtime preserves the whole application routed packet until
  post-dispatch handling. The selected target, route kind, and committed frame
  are not reduced to raw console input before application-level dispatch.
- After a layer is selected, its committed interaction frame resolves one local
  route: captured mouse targets take priority only for mouse input, ordinary
  mouse selects the topmost matching hit target, keyboard and modifier input
  selects the committed keyboard target when one is published, keyboard and
  modifier input without an explicit keyboard target falls back to the committed
  focused target, and all other cases use a layer route. Target identity is
  local to its layer instance; targets are never resolved across layers.
- Hit testing uses only the last committed interaction frame. A mouse hit
  chooses an input recipient but does not change focus; a focus request made by
  the handler applies only after the current event has been routed.
- Interaction hit regions are ordered by visual stacking order and describe
  only controls rendered in the committed frame. A control may publish focus
  metadata without a mouse hit region when it has no visible active area.
- The main application surface publishes committed mouse targets for the
  command line, both file panel sides in `Panels`, interactive panel scrollbars,
  rendered function-key actions, and rendered directory-shortcut slots.
  `HiddenCommandLine` publishes no panel, panel scrollbar, function-key, or
  directory-shortcut targets. Passive quick-view preview sides, non-interactive
  scrollbars, and empty bar slots do not publish hit targets.
- The main application surface publishes one explicit keyboard target for the
  application workspace in both `Panels` and `HiddenCommandLine`. Command-line
  cursor metadata remains visual focus metadata on the command-line target, but
  keyboard and modifier routed packets for the application surface target the
  workspace keyboard route. Panel targets remain mouse hit targets and do not
  become a second independent active-side source of truth.
- Application keyboard ownership is resolved in one application-level policy
  using the committed application frame. Command-line editing keys, printable
  text, selection, command completion, and command history belong to command-line
  handling. Panel navigation belongs to the active rendered panel when
  command-line state does not own the key. Global shortcuts, function keys, and
  directory shortcuts are handled before owner-specific keyboard handling.
  Ownership decisions use the committed workspace mode, active side,
  command-line text and selection presence, active-panel search state, and
  committed panel frames. They do not reread mutable active-side state,
  command-line state, panel search state, screen coordinates, current viewport,
  or recalculated layout.
- Application panel item hit mapping is part of the committed application
  frame. Panel mouse handlers use the committed item hit map and verify the
  stored item identity before performing item-specific actions, so a stale
  routed packet cannot open, select, or toggle a different current item.
- Application panel keyboard movement uses panel geometry from the committed
  application frame, including visible rows and any multi-column navigation
  metadata. Keyboard handlers do not reread console size or recompute
  input-sensitive panel layout.
- Application panel scrollbar dragging uses composition mouse capture owned by
  the corresponding panel-scrollbar target. Captured movement and release route
  back to that target even outside the scrollbar bounds. A committed geometry or
  list-dimension change rebases the local drag state when the same interactive
  scrollbar target remains available; disappearance or loss of interactivity
  clears the local drag state and capture eligibility.
- Application function-key and directory-shortcut mouse handlers execute after
  routed dispatch and choose their command or shortcut only from the committed
  frame that created the routed packet. Later binding or settings changes do
  not change the semantic target of that packet. A committed directory-shortcut
  mouse target includes the resolved path shown in that frame; keyboard
  directory-shortcut input continues to resolve the shortcut number against the
  current settings.
- The former keyboard compatibility routing path is not part of the runtime
  contract. It may survive only as a small internal coordinator whose input is
  an application routed packet with a committed frame, not raw runtime input.
- Editor routed input publishes stable local targets for keyboard, visible text
  content, the interactive vertical scrollbar, and rendered active function-key
  actions. Keyboard and modifier input are accepted only through the committed
  keyboard target. Text clicks, double-clicks, wheel input, selection drags,
  scrollbar clicks and drags, and function-key clicks are converted to compact
  editor semantic packets during routed dispatch. The semantic packet records
  the committed frame and target mapping that produced it; editor document
  commands, saves, close decisions, and nested dialogs run only after routed
  dispatch has completed.

## Application overlays

- The main application renderer draws only the underlying application surface:
  panels, directory shortcuts, clock, command line, and function keys. It does
  not draw command-line completion, panel quick search, or the top menu, and it
  publishes the base command-line cursor through command-line focus metadata
  for the ordinary panels surface.
- Command-line completion is a bubble overlay layer. When hidden it has no input
  policy, draws nothing, publishes no cursor or focus metadata, and does not
  affect lower layers. When visible it handles only completion navigation,
  acceptance, removal, temporary hide, item mouse activation, and completion
  scrollbar interaction that can be performed in the committed completion
  frame. It does not take focus from the command line, so ordinary text input
  and unrelated shortcuts continue to the application surface.
- Panel quick search is a bubble overlay layer that remains registered while
  hidden so it can own quick-search activation. While inactive it handles only
  activation input. While active it owns search text editing, close behavior,
  popup rendering, and one focus target whose cursor metadata places the quick
  search cursor. Unsupported keyboard input and any mouse input close quick
  search and continue routing of the same physical event to lower layers.
- The top menu is a bubble overlay layer while closed and a modal overlay layer
  while open. While closed it handles plain F9 and left-button down on the top
  screen row, including positions outside visible menu item text. Opening the
  menu closes active panel quick search first. While open it owns menu
  rendering, keyboard navigation, mouse navigation, scrollbar interaction,
  focus representation for the active menu item, and lower-layer blocking. Its
  focus metadata hides lower-layer cursors without publishing a visible cursor
  of its own.
- Top menu activation belongs to the ordinary panels surface. In
  `HiddenCommandLine` the top menu is unavailable: it draws nothing, publishes no
  activation hit region or focus target, and does not handle F9 or top-row mouse
  activation. Entering `HiddenCommandLine` closes any open top menu together
  with transient panel overlays.
- Domain commands selected from the top menu are requested by menu UI handling
  and executed only after routed dispatch has completed. Selecting a command
  never opens dialogs, changes the layer stack, starts rendering, or performs
  other domain work inside the menu layer's routed input handler.
- An invalidate-only routed event does not cause a second composition frame when
  post-dispatch menu or application handling has already committed the current
  composition. An explicit post-dispatch render request still produces its
  requested frame.
- Scrollbar dragging in application overlay layers uses composition mouse
  capture owned by the relevant layer target. Captured move and release input
  is routed to the capture owner according to the committed interaction frame
  and released on the matching button release or when the target is no longer
  eligible.
- An application-overlay scrollbar is published as an input target only when
  its committed bounds and scroll state are interactive. A visually rendered
  scrollbar that cannot process mouse input has no interaction target.
- Local overlay scrollbar drag state is cleared when the corresponding target
  is absent from a committed interaction frame. A rejected render attempt does
  not alter drag state from the preceding committed frame.
- An active application-overlay scrollbar drag is bound to its committed
  scrollbar geometry and list dimensions. When a later frame commits with the
  same interactive target but changed geometry or dimensions, the drag remains
  captured and is rebased to that frame with its pointer offset constrained to
  the new thumb. When the target is absent or no longer interactive, the drag
  and its composition capture are cancelled. Rejected attempts leave the active
  drag unchanged.
- The same committed-geometry rule applies to row-owned dropdown scrollbars in
  interactive form layers. While a dropdown scrollbar drag is active, a
  successful committed resize rebases the drag through the shared scrollbar
  interaction model when the committed dropdown scrollbar target remains
  interactive. If the committed frame no longer publishes that target, the
  dropdown clears its local drag state and composition capture eligibility.
  Rejected render attempts do not rebase or clear the dropdown drag.

## Semantic mouse input

- Console backends publish semantic mouse input using only `Down`, `Up`,
  `DoubleClick`, `Move`, and `Wheel`.
- Backends normalize equivalent physical input to equivalent semantic streams.
  A single button press and release is `Down`, `Up`; a compatible double-click
  sequence is `Down`, `Up`, `DoubleClick`, `Up`; drag is `Down`, one or more
  `Move` events, then `Up`; wheel input uses `Wheel` with wheel-up or
  wheel-down buttons.
- Mouse release events include the released button. Mouse capture and drag
  handling may rely on receiving a matching `Up` for left, right, and middle
  buttons when the backend reports the release.
- Double-click recognition is backend-independent and depends on matching
  button, viewport-relative coordinates, modifiers, and interval. Backend-native
  double-click records are treated as the physical second press before semantic
  normalization.
- Hover motion without a supported pressed button is not part of the routed UI
  contract. Mouse target selection uses the semantic viewport-relative
  coordinates and the committed interaction frame of each layer that is reached
  by composition routing.

## Mouse capture

- Mouse capture belongs to a specific layer entry and target, not only to the
  layer object. Captured mouse events are routed only to the capture owner while
  that owner remains eligible in the current active input route.
- Capture requests are produced by layer input results and applied by the
  composition host. Matching mouse button release or explicit release clears
  capture; disposing the owner layer entry also clears capture.
- A target is eligible for mouse capture only when it is present in at least one
  committed mouse hit region for the owner layer. A target that exists only as
  keyboard ownership or focus metadata is part of the interaction frame but is
  not mouse-capturable. Revalidation after a committed render, layer
  replacement, or surface replacement uses the same committed hit-region
  source of truth.
- Capture does not change render order, focus ownership, or active-composition
  selection. If the capture owner is no longer active, has no input policy, or
  is below a modal layer, capture is cleared before normal routing continues.
  Capture is also cleared when its target disappears from the owner's committed
  interaction frame; the event that discovers this condition continues through
  normal layer and hit-target routing.
  Opening a modal layer above a capture owner preempts the lower capture, and a
  capture cleared by temporary deactivation is not restored when the owner later
  becomes active again.
- Editor text selection and scrollbar dragging use composition mouse capture
  owned by the committed target. Captured movement and matching release route
  back to that target even outside the original bounds. A successful committed
  geometry change rebases an active editor scrollbar drag when the same
  interactive target remains published, and clears drag and capture eligibility
  when the target disappears or becomes non-interactive. A rejected render
  attempt leaves the previous committed drag, content geometry, function-key
  mapping, cursor state, syntax diagnostics, and redraw range unchanged.

## Modal behavior

- Ordinary modal dialogs, progress dialogs, popup restoration, and nested
  dialogs must use the shared `ModalDialogHost`.
- `ModalDialogHost` registers modal dialogs as interactive modal layers, not as
  render-only overlays. Those layers participate in the shared rendering stack,
  receive input through composition routed dispatch, and block lower layers.
- A modal layer that accepts an input event stores at most one pending routed
  packet and returns a handled result for that event. Dispatching another input
  to the same modal layer before the packet is consumed is an application error.
- A modal session renders the full composition. Its input methods consume
  `ConsoleResizeInputEvent`, perform a resize-recovery composition render, and
  return only dialog-relevant input. A viewport change detected without an
  explicit resize event is recovered before returning user input. A semantic
  input event already read when a viewport change is detected is returned after
  recovery; it is never replaced or lost.
- A modal session returns only input routed to its own modal layer. The input
  handler receives the committed frame from the same dispatch cycle that
  delivered the input. If routed dispatch already produced a pending packet, the
  session returns that packet before reading another console input event. Dialog
  domain handlers execute after routed dispatch has completed, so nested dialogs
  mutate the stack outside active dispatch. Polling dialogs use the same modal
  layer lifecycle.
- Interactive form modal layers may perform local UI handling during routed
  dispatch and return focus, invalidation, and mouse-capture requests as their
  `UiInputResult`. The modal layer stores the semantic form result together
  with the original routed packet and committed frame, then the owning dialog
  handles validation, submit/cancel, and nested domain actions only after
  dispatch has completed. A form modal layer publishes row targets, body
  scrollbar targets, history dropdown targets, and history scrollbar targets
  through the committed interaction frame. The upper interactive layer's cursor
  is derived from attempt-local focus metadata and applied by the layer, not by
  the form component directly.
- For an interactive modal event, requested focus and capture changes apply
  before the dialog consumes its semantic packet. The dialog domain handler
  then runs outside dispatch and before the next composition render. A
  continued event creates exactly one subsequent composition frame; a completed
  event does not redraw its parent dialog before closing it.
  Dispatch-triggered invalidation from an interactive modal event is observed by
  the owning modal runner after the semantic domain handler, not rendered inside
  the modal input read. If the handler continues, the next prepared render is
  the only new composition frame for that event.
- For a timed interactive modal wake, the dialog wake handler runs after input
  pumping has determined that no owner packet is pending for that cycle. The
  wake handler may synchronize a background-operation snapshot with logical
  dialog state, but it must not run inside routed dispatch or rendering. If it
  completes the modal, the runner closes the overlay without requiring console
  input or busy polling. If it continues and invalidates, the next prepared
  render is the only new composition frame for that wake.
- A timed interactive modal distinguishes runner cancellation, its periodic
  deadline, an external wake signal, an already formed routed owner packet, and
  semantic input that remains unread in the console queue. Runner cancellation
  cancels the runner; an external wake signal only invokes the ordinary wake
  handler. An already formed owner packet has priority and is handled with the
  committed frame that produced it. Once an external wake signal has fired, it
  has priority over reading new semantic input. If it interrupts a blocking
  read, unread console input remains for the composition that becomes active
  after the modal closes and is not dispatched to the closing modal.
- Dialogs own only dialog state, current-frame layout, rendering of their own
  layer, input handling, and their result. They neither render underlying
  surfaces nor accept underlay-rendering callbacks; production dialog APIs use
  the shared modal lifecycle.

## Surface behavior

- The main application surface encapsulates the `Panels` and
  `HiddenCommandLine` workspace modes. It preserves the shell-underlay, terminal-screen,
  bottom-pin, and scroll-position rules of 0004.
- Editor, viewer, and full-screen help are temporary surfaces. Their dialogs
  use the shared modal stack. Interactive temporary surfaces use the shared
  routed layer model: they publish a committed frame and interaction frame,
  retain at most one semantic packet paired with that committed frame, and
  perform domain actions only after routed dispatch. The shared surface runner
  is the sole owner of their synchronous surface lifecycle, input reading,
  timed wake, and resize recovery. The editor does not open its own composition
  surface, run a private input pump, poll for input, or read console input
  outside the shared host.
  Disposing a temporary surface redraws the prior active surface.
- Full-screen help publishes a committed immutable frame containing the viewport,
  header and content bounds, rendered footer action hit regions, current vertical
  and horizontal scroll positions, `MaxScrollTop`, `MaxScrollLeft`, visible row
  count, and vertical scrollbar bounds and scroll state when present. Help
  keyboard and mouse input uses only that committed frame. Horizontal scrolling
  is clamped to the maximum display width of the rendered help document,
  including key-binding lines with their key column and description. Help
  key-binding rendering preserves separate styles: the key column uses
  `HelpKey`, while the description uses `HelpBody`, including when horizontal
  clipping crosses the boundary between them.
- Full-screen help synchronizes its scroll position and vertical scrollbar drag
  state only from accepted `HelpViewerFrame` commits. A committed geometry change
  rebases an active scrollbar drag when the same interactive scrollbar target
  remains published; an accepted frame without the scrollbar cancels the drag
  and composition capture eligibility. A rejected render attempt without the
  scrollbar does not cancel or rebase the active drag. Help cursor visibility is
  determined by the committed interaction frame and focus metadata, not by
  direct cursor calls from the help renderer.
- Search and file-operation progress are long-lived overlays. Confirmation
  dialogs may nest above the relevant progress overlay. Built-in remote-provider
  UI follows the same modal lifecycle.

## Snapshot policy

Screen snapshots remain low-level operations for shell underlay,
external-console or child-process interoperability, terminal-screen mode,
renderer tests, and documented public compatibility APIs. They must not be used
to restore application-owned modal dialogs, progress dialogs, editor, viewer,
help, popups, or nested dialog UI.

## Acceptance criteria

- Resizing while a dialog is over panels, hidden command line, editor, viewer,
  progress UI, or another dialog redraws every visible layer using the final
  viewport.
- Resize events never reach normal dialog-specific input handlers.
- Removing an overlay or temporary surface leaves no stale cells and redraws
  the correct lower composition.
- Text, focus, cursor, valid clamped scrolling, selected values, viewer/editor
  position, search state, and progress state survive a viewport change.
- Progress remains below its nested confirmation overlays and redraws after
  those overlays close.
- Help or editor opened from the viewer returns to the viewer when closed.
- Main application, editor, viewer, and help do not run independent stable
  render loops.
- Application-owned layers do not use snapshots, private frame lifecycles, raw
  screen or console input, or dialog-specific underlay rendering callbacks.
- Reusable, application, progress, editor, viewer, help, and built-in module
  dialogs all use the shared modal or temporary-surface lifecycle.
- Child controls never commit focus state, and post-dispatch focus transitions
  become observable only with an accepted interaction frame.
- Timed modal progress overlays that combine routed input with background
  completion use the shared timed interactive modal lifecycle and its external
  wake signal; completion of the background operation does not require fake
  console input, `TryReadInput` polling, or busy waiting. A completion wake
  that occurs before a new console event is read closes the modal without
  consuming that event, so it remains available to the next active composition.
- Help publishes committed keyboard, content, scrollbar, and rendered
  function-key targets. Its cursor and mouse capture are layer-owned metadata;
  its local scroll and drag state is derived from the committed Help frame.
- `HiddenCommandLine` shell output and bottom-pin behavior remain governed by 0004.
- Active surfaces are rendered through their layer contract. The surface
  lifecycle owns only begin-frame and complete-frame behavior; layer rendering
  owns drawing.
- Interactive layer instances cannot have more than one active registration in
  the composition stack.
- Interactive layers expose committed interaction frames, and focus traversal
  uses the focus frame from the committed interaction frame for the same stable
  render attempt.
- An active panel quick search always publishes its input focus target. When
  its popup does not fit the viewport, the target has no hit region and no
  cursor placement, so it retains keyboard ownership while hiding any cursor
  supplied by a lower application layer. The search remains active and its
  state is retained until the popup can be rendered again.
- Command completion input participation is determined by visibility in its
  most recently committed frame. Model-level intent to show completion and its
  matches may remain while the popup is temporarily unable to fit the viewport.
- A rejected render attempt does not change layer input policy, focus ownership,
  cursor ownership, or committed interaction targets.
- Physical cursor position and visibility are applied as part of the same
  viewport-protected render frame as the screen buffer. A rejected attempt does
  not apply its pending physical cursor state; the committed focus and cursor
  metadata are published only after that frame is accepted.
- Routed capture cannot bypass a modal layer, cannot survive temporary
  deactivation of its owner or removal of its committed target, and does not
  cause the mouse event that invalidated it to be lost.
- Routed input records the selected local target and route kind with its
  committed frame when application and modal layers create their pending packet.
  Existing modal input APIs retain their input-only view of those packets.
- Modal dialog overlays have modal input policy, lower layers do not receive
  modal input, nested dialog handlers run outside active dispatch, and polling
  progress dialogs preserve their existing modal behavior.
- Application routed input is processed in two phases: routed dispatch assigns
  ownership to the application surface and records the selected local target,
  then the runtime invokes the corresponding application target handler after
  dispatch by consuming that packet. Overlay and modal input do not reach
  application compatibility routing.
- Application mouse input for command line, panels, panel scrollbars,
  function-key bar, and directory-shortcut bar is routed by committed
  application targets. Production application mouse handling has no legacy
  fallback router.
- Application mouse handlers use the committed application frame for hit maps,
  item identity, bar action mappings, shortcut mappings, visible rows, and
  scrollbar geometry. They do not recompute input layout from current console
  size, renderer bounds, or mutable settings after routed dispatch.
- Invisible application controls have no mouse hit target: `HiddenCommandLine`
  publishes only the command-line target, passive quick-view preview sides
  publish no file-panel target, non-interactive scrollbars publish no scrollbar
  target, and unrendered bar slots publish no bar region.
- Panel scrollbar thumb drag uses composition mouse capture, captured movement
  routes back to the scrollbar target outside its bounds, and matching release
  releases capture. Committed resize rebases drag when the same interactive
  target remains available and clears drag when the target disappears or becomes
  non-interactive.
- Keyboard behavior for the main application surface uses the same two-phase
  routed application bridge as mouse input. Global application shortcuts are
  evaluated in documented order, then the logical keyboard owner receives the
  key, and unhandled input remains unhandled.
- `HiddenCommandLine` leaves top-menu activation input available to
  the active application surface instead of being intercepted by the top menu
  overlay.
- Application-overlay scrollbar dragging remains valid across a committed
  geometry change by using the new committed scrollbar bounds and list
  dimensions; a disappearing or non-interactive scrollbar cancels drag and
  capture, while a rejected frame leaves them unchanged.
- Editor keyboard, modifier, mouse, scrollbar, and function-key input is routed
  through the editor's committed interaction frame. Editor domain events retain
  their existing public order: input is raised after routed keyboard dispatch
  reaches the editor domain handler, while redraw and syntax diagnostics describe
  only the successfully committed frame.
- Editor cursor placement, text selection, scrollbar drag, function-key
  mappings, syntax diagnostics, redraw ranges, and viewport publication are
  transactional: accepted frames publish them through the editor layer commit
  path, and rejected frames publish none of them.
- The editor wide-character cursor blinks through shared timed wake without
  busy polling or a background timer that outlives the editor surface.

## Verification

Automated verification covers these durable evidence groups:

1. Composition and resize: the final committed viewport is the only
   input-visible viewport; visible layers render in stack order; resize recovery
   preserves already-read semantic input; closing an overlay or temporary
   surface redraws the lower composition from current state.
2. Transactional frames: accepted frames publish frame, focus, cursor, target,
   callback, and surface state atomically; rejected frames publish none of that
   state; later input uses the last committed frame.
3. Routing and modal isolation: the topmost eligible layer receives input
   first; handled bubble routing stops lower layers; modal layers block lower
   layers even when unhandled; domain work runs only after routed dispatch.
4. Focus, cursor, and capture: ownership follows the committed interaction
   frame; physical cursor state is applied only by an accepted
   viewport-protected frame; capture cannot bypass modal layers and is released
   when its owner or target disappears or becomes inactive.
5. Application targets: keyboard and mouse handlers consume routed packets
   paired with committed application frames; invisible controls publish no
   targets; stale panel item, scrollbar, function-key, or directory-shortcut
   packets cannot act on a different live item or command.
6. Interactive surfaces and dialogs: temporary surfaces and modal sessions pair
   each semantic packet with the committed frame that produced it; resize
   recovery happens before semantic dispatch; completion, cancellation,
   disposal, and exceptions clean up the active composition and restore the
   parent surface.
7. Editor routed lifecycle: editor keyboard, modifier, mouse selection,
   function-key, mouse-wheel, and vertical-scrollbar scenarios use committed
   editor geometry; captured selection or scrollbar drags continue outside the
   original bounds and release through composition capture; resize preserves
   text, cursor, selection, viewport, scrollbar geometry, function-key mapping,
   and already-read semantic input.
8. Editor cursor ownership: ordinary editor cursors are applied through
   committed focus metadata, composed wide-character cursors blink through
   shared timed wake, and modal dialogs above the editor temporarily own cursor
   state without the editor overwriting it directly.
9. Platform mouse normalization: supported console backends produce equivalent
   semantic mouse streams for clicks, release preservation, double-clicks,
   drags, wheel input, modifiers, and viewport-relative coordinates.
10. Justified manual verification: real terminal checks are reserved for
   backend-specific modifier, composed-cursor blink, nested-dialog cursor,
   resize, and host-rendering behavior that is too costly or unstable to prove
   solely with deterministic fake-console tests.
