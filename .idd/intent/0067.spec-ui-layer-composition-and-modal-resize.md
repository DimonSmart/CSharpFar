# 0067 - UI layer composition and modal resize

## Goal

Application-owned console UI is rendered through one shared composition host so
that a viewport change redraws the active full-screen surface and every modal
overlay as one current frame. Closing a layer rebuilds the UI from state rather
than restoring pixels captured from an earlier viewport.

## Layer model

- A **surface** is an opaque full-screen UI: the main application screen,
  hidden-panels command-line surface, viewer, editor, or full-screen help.
- Surfaces and overlays share one ordered layer stack. The last surface in that
  stack is active; earlier surfaces and every overlay below that active surface
  are not drawn.
- An **overlay** is drawn only when it is above the active surface in the same
  stack. Eligible overlays are rendered in stack order and support nesting.
- Surface and overlay lifetimes are explicit disposable LIFO scopes. Repeated
  disposal is safe; invalid out-of-order disposal is rejected deterministically.

## Composition and frames

- The application owns one shared composition lifecycle and one associated modal
  lifecycle. They are created during bootstrap and passed through normal
  dependencies; UI components do not use static service lookup or local hosts.
- The shared composition lifecycle exclusively owns stable-frame retry, viewport
  change detection, resize-event consumption, and rendering the active surface
  followed by the visible overlays. It rebuilds the complete composition when a
  frame is interrupted and records a viewport only after a stable frame.
- Layers derive layout from the viewport supplied by the current frame. They do
  not open nested frames or reread console size for the current layout. Mouse
  hit testing uses layout from the most recent stable render.
- The topmost rendered layer determines cursor state. On overlay close the
  composition redraw restores the cursor from the remaining top layer.

## Modal behavior

- Ordinary modal dialogs, progress dialogs, popup restoration, and nested
  dialogs must use the shared `ModalDialogHost`.
- A modal session renders the full composition. Its input methods consume
  `ConsoleResizeInputEvent`, perform a resize-recovery composition render, and
  return only dialog-relevant input. A viewport change detected without an
  explicit resize event is recovered before returning user input. A semantic
  input event already read when a viewport change is detected is returned after
  recovery; it is never replaced or lost.
- Dialogs own only dialog state, current-frame layout, rendering of their own
  layer, input handling, and their result. They neither render underlying
  surfaces nor accept underlay-rendering callbacks; production dialog APIs use
  the shared modal lifecycle.

## Surface behavior

- The main application surface encapsulates normal panels and hidden-panels
  command-line mode. It preserves the shell-underlay, terminal-screen,
  bottom-pin, and scroll-position rules of 0004.
- Editor, viewer, and full-screen help are temporary surfaces. Their dialogs
  use the shared modal stack. Disposing a temporary surface redraws the prior
  active surface.
- Search and file-operation progress are long-lived overlays. Confirmation
  dialogs may nest above the relevant progress overlay. Built-in remote-provider
  UI follows the same modal lifecycle.

## Snapshot policy

Screen snapshots remain low-level operations for shell underlay,
external-console or child-process interoperability, terminal-screen mode,
renderer tests, and documented public compatibility APIs. They must not be used
to restore application-owned modal dialogs, progress dialogs, editor, viewer,
help, popups, or nested dialog UI.

## Acceptance criteria

- Resizing while a dialog is over panels, hidden command line, editor, viewer,
  progress UI, or another dialog redraws every visible layer using the final
  viewport.
- Resize events never reach normal dialog-specific input handlers.
- Removing an overlay or temporary surface leaves no stale cells and redraws
  the correct lower composition.
- Text, focus, cursor, valid clamped scrolling, selected values, viewer/editor
  position, search state, and progress state survive a viewport change.
- Progress remains below its nested confirmation overlays and redraws after
  those overlays close.
- Help or editor opened from the viewer returns to the viewer when closed.
- Main application, editor, viewer, and help do not run independent stable
  render loops.
- Application-owned layers do not use snapshots, private frame lifecycles, raw
  screen or console input, or dialog-specific underlay rendering callbacks.
- Reusable, application, progress, editor, viewer, help, and built-in module
  dialogs all use the shared modal or temporary-surface lifecycle.
- Hidden-panel shell output and bottom-pin behavior remain governed by 0004.

## Verification

Automated coverage verifies composition ordering and LIFO lifetimes,
interrupted-frame retry, resize-event draining without input loss, modal resize
recovery, nested-modal restoration, semantic input preservation, and final
rendered cells for larger and smaller viewports across panels, hidden panels,
temporary surfaces, remote-provider dialogs, and nested progress UI.
