# 0071 - Transactional UI frame commit and resize regression coverage

## Goal

Console UI composition publishes viewport-dependent state only after a complete
frame has been flushed and confirmed to match the current console viewport.
A resize during composition discards the attempted frame and redraws the active
surface and visible overlays for the new viewport.

## Frame lifecycle

- A render attempt captures one viewport and composes the active surface followed
  by eligible overlays into one buffered frame.
- Layout, cursor placement, panel bounds, dialog bounds, scroll position chosen
  for layout, and mouse hit-test geometry prepared by an attempt are pending
  state. They are not observable by input handling before commit.
- After flush, the composition lifecycle detects interrupted writes, drains only
  resize events during resize recovery, and verifies the viewport again.
- A rejected attempt discards every pending publication. A committed attempt
  publishes callbacks once, in registration order, then records the stable
  viewport and reports a committed frame completion.
- Rendering is pure with respect to observable UI state. It may read logical
  state, calculate attempt-local effective state, create immutable frame
  snapshots, write the buffered renderer, and register stable publications,
  but it does not immediately change input state, layout, selection, scroll,
  focus, document/application state, diagnostics, render caches, callbacks, or
  events.
- Attempt-local frame state, including layout and hit-test geometry, becomes
  available to input handling only after commit. A rejected attempt has no
  logical effect; partial physical writes are repaired by the next complete
  composition render.
- Semantic key, mouse, and modifier input accumulated with resize events remains
  available after recovery and is neither lost nor duplicated.

## Layout and input

- Interactive reusable forms retain logical input state across resize, but use
  immutable layout from the last committed frame for cursor placement, scrolling,
  and mouse hit testing.
- Frame-bound form layout uses the viewport supplied by composition; it does not
  reread console size during rendering.
- The main application publishes its rendered viewport and panel bounds only at
  stable-frame commit. The same pending bounds are used for rendering dependent
  overlays in that attempt.
- Every layer uses the viewport and size from its current render context. A
  layer does not reread console size or viewport, start a nested frame, or
  consume input while rendering.
- Reusable frame-dependent state is represented by immutable attempt snapshots
  and committed state. Input handlers read only the most recently committed
  snapshot; pending values are invisible until stable-frame commit.
- The ordinary modal dialog runner passes the input handler the frame returned
  by the same committed input cycle that produced the semantic input. Resize
  recovery frames that are rejected are not passed to handlers and do not run
  committed-frame callbacks.
- Logical preparation for an explicit modal render runs before opening or
  rendering a frame and outside the composition render transaction. It may
  synchronize derived rows or immutable collections, but it does not publish
  viewport geometry or mutate state from inside a render callback.
- Asynchronous or polling UI that renders changing external data uses an
  immutable data snapshot as part of the committed frame. Input handling does
  not mix geometry, scrollbar state, or selected indexes from one frame with a
  newer collection snapshot from another frame.

## Verification

Automated tests use the real buffered `ScreenRenderer` and deterministic fake
viewport changes during physical frame writes. They verify that an interrupted
attempt cannot publish old geometry, that the following attempt publishes only
the new viewport, and that the final buffer is fully redrawn for that viewport.
They also verify that rejected attempts do not publish layout, selection, scroll,
focus, callbacks, diagnostics, or render-cache changes, and that input after a
resize uses the final committed geometry exactly once.

Modal runner verification covers initial render before input, exactly one
explicit render after a continuing transition, no extra render after completion,
semantic input paired with the committed frame from the same cycle, rejected
frames hidden from handlers and apply callbacks, disposal on normal return,
exception and cancellation, and nested modal restoration before the parent
continues.
