# 0002 - Render buffering strategy

Type: ADR
Status: Accepted

## Context

Console UI redraw must avoid flicker and unnecessary physical writes without
making each UI component track dirty regions. Rendering also has to protect the
bottom function-key bar and avoid wrapping or scrolling when drawing near the
right and bottom edges of the console.

## Decision

`ScreenRenderer` owns the application UI screen state through display buffers:

- `front buffer`: the last known frame successfully sent to the console driver.
- `back buffer`: the new logical frame built by the current redraw.

Application code draws a full logical frame through `BeginFrame()`. When the
frame is flushed, the renderer compares the back buffer with the front buffer,
groups changed cells into horizontal runs with the same style, and calls the
console backend only for those runs.

The application-owned UI buffer is the source of truth for rendered UI cells.
Native console screen buffers, when available, are physical output details and
must not be read as the authoritative UI state. A rendered cell consists of the
character plus foreground color, background color, and text attributes such as
bold, underline, and reverse. Platform backends may map that cell state to
Win32 console APIs or ANSI/VT escape sequences, but UI components render only
to the shared logical surface.

`Application` decides whether UI needs repainting after an event, but it does
not compute dirty regions or pass dirty-region data into the renderer.

`Rect` remains a geometry type for component layout, clearing, capture, and
restore operations; it is not a dirty-region API.

`Capture` and `Restore` are compatibility operations for existing overlay and
underlay flows. They must not make the native console buffer the application
source of truth. Cross-platform UI rendering remains based on the
application-owned logical cell buffers.

On console resize, renderer buffers are recreated and the next visible frame is
treated as a full frame. On Windows, resize input wakes the main loop and
triggers a full logical redraw.

During UI rendering on Windows, wrap-at-EOL is disabled so writes to the bottom
right cell do not wrap or scroll the screen. The original console mode is
restored before shell output.

The bottom function-key bar is always rendered as one row. If its content does
not fit, it is clipped to the available width and the rightmost available cells
show a red ellipsis when there is room.

## Consequences

- UI components can render full logical state without knowing physical dirty
  regions.
- Repeated identical frames produce no physical writes.
- Cursor movement and highlight changes write only cells that actually changed.
- Resize invalidates cached physical state explicitly.
- Console-driver behavior is responsible for output mode transitions that would
  otherwise cause wrapping or scrolling.
- Unix-like terminals render the same logical buffer through ANSI/VT output
  rather than attempting to emulate a readable Windows console screen buffer.
