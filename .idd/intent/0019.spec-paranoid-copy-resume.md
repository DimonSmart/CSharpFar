# 0019 - Reliable copy resume

Related:
- 0010
- 0012

## Goal

Add a copy mode named `Reliable` in the copy start dialog. The mode must be
available before copying begins, including recursive folder copy, and must
resume partial destination files only after validating already copied bytes near
the destination tail.

The implementation must keep the actual safety logic outside UI code. UI may let the
user choose the mode and show progress/results, but the decision about the safe resume
offset belongs in a separate tested analyzer.

## Context

The requested behavior is compatible with the current codebase with a few important
shape changes:

1. The current copy UI already has one start dialog for both file and folder
   copy: `FileOperationDialog.ShowCopy`. `Reliable` belongs in a dedicated copy
   mode row, not in the existing-file conflict row.
2. The current `FileOperationService` builds a path-based copy plan, expands folders
   into file items, and handles per-file conflicts. This means one operation-level
   mode can apply to every existing file encountered while copying a folder.
3. The current file operation layer is path-based, not provider-stream-based. That
   matches a path-based `CopyResumeAnalyzer` and the current search-results rule that
   copy may map real paths into the existing operation service.
4. The current conflict dialog can stay as the fallback UI for files that cannot be
   resumed safely. A `Resume` button only in the conflict dialog is not enough for
   this spec, because the user explicitly needs the mode at copy start.
5. The current copy stream path has overwrite and append only. It will need a third
   destination action: resume after tail validation.
6. The current progress model has only `Scanning` and `Copying`. It must grow enough
   to report validation and resume status.
7. The current application does not have resilient retry after an `IOException`.
   Retry integration is possible, but it is new behavior and should be treated as a
   separate stage inside this spec or split before implementation if it becomes too
   large.

Use `Reliable` as the UI label. Prefer explicit code names such as
`TailValidatedResume` and `CopyResumeAnalyzer` for the resume mechanics.
`ResumeWithTailValidation` is not a conflict decision.

## Scope

Expose the mode at copy start:

1. Add a copy mode displayed as `Reliable` in the start copy dialog.
2. The mode must be selectable before the scan/copy begins for a single file, multiple
   files, and folder copy.
3. Persist the mode separately from `AppSettings.FileOperations.ConflictDecision`.
4. `Reliable` does not apply to move/rename or provider copy.

Implement tail-validated resume:

1. Add a separate `CopyResumeAnalyzer` in the filesystem layer.
2. Add explicit result models for the analyzer, for example
   `CopyResumeOptions`, `CopyResumePlan`, and `CopyResumePlanKind`.
3. The analyzer works with seekable source and destination files by path.
4. The analyzer compares byte ranges directly with buffers. Do not add whole-file
   hashes.
5. Use fixed defaults for the first version:
   - initial tail validation: 1 MB;
   - minimum validation range: 64 KB;
   - maximum rollback search: 64 MB;
   - rollback multiplier: 4;
   - comparison buffer: 128 KB.
6. If `0 <= destination.Length < source.Length`, validation may produce `CanResume`.
7. If the destination length is 0, `CanResume` with offset 0 is allowed. This is safe
   and lets the copy path handle empty existing files without a special UI path.
8. If the destination length equals or exceeds the source length, do not resume.
9. If source or destination length cannot be read, do not resume.
10. If either stream cannot seek, do not resume.
11. If a source snapshot is available from the current operation, compare at least
    source length and source `LastWriteTimeUtc` before resuming after an interruption.

Use this resume algorithm:

1. First validate the last `min(1 MB, destination.Length)` bytes of the existing
   destination against the same source range.
2. If the range matches, select `SafeResumeOffset = destination.Length`.
3. If the range does not match, search earlier candidate offsets with rollback windows
   `1 MB`, `4 MB`, `16 MB`, and `64 MB`.
4. For each candidate, validate a range starting at the candidate offset with length
   `min(1 MB, destination.Length - candidateOffset)`.
5. Skip non-zero candidates whose validation range would be smaller than 64 KB.
6. If a candidate range matches, select that candidate as `SafeResumeOffset`.
7. If no candidate matches within the rollback limit, return `CannotResume` with a
   clear reason.

Integrate with ordinary copy:

1. Use a destination action that can express create/overwrite, skip,
   rename, and tail-validated resume.
2. When the operation mode is `Reliable` and an existing destination file is smaller
   than the source file, run `CopyResumeAnalyzer`.
3. If the plan is `CanResume`, open the destination for read/write, truncate it to
   `SafeResumeOffset`, seek source and destination to that offset, and copy the
   remaining bytes through the existing buffered loop.
4. Count accepted already-present bytes in progress so current-file and total progress
   start at the safe resume offset.
5. If bytes were discarded, show the rollback amount in progress.
6. Preserve existing timestamp, attribute, and ACL behavior after the final file is
   complete. Do not copy metadata before the content is complete.
7. If `Reliable` cannot safely resume a file, fall back to an explicit user decision.
   The first implementation may use the existing conflict dialog, where `Overwrite`
   means restart from byte 0, `Skip` skips the file, and `Cancel` cancels the
   operation.
8. Directory creation behavior is unchanged. The mode affects files inside the copied
   directory tree.

Report progress:

1. Add enough progress state to show validation before copy continues. This may be a
   new `FileOperationPhase.Validating` or an explicit status field on
   `FileOperationProgress`.
2. Show `Validating partial file...` while the analyzer runs.
3. After validation, show either `Tail validation passed` or
   `Tail mismatch detected`.
4. Show resume offset and rollback bytes when available.

Retry integration:

1. `Reliable` copy retries source-open, source-read, destination-open, and
   destination-write failures with fixed internal retry defaults.
2. Retry must close source and destination streams,
   wait for user retry or configured retry delay, reopen both paths, read the actual
   destination length from disk, rerun `CopyResumeAnalyzer`, and continue only from the
   analyzer's safe offset.
3. Retry must not continue from the last in-memory byte position after an error.
4. If source length or `LastWriteTimeUtc` changed from the operation snapshot, automatic
   resume is unsafe and the user must choose restart, skip, or cancel.

## Non-goals

Do not implement in this spec:

1. whole-file verification after copy;
2. whole-file hash verification;
3. rsync-like block matching;
4. custom network protocols;
5. non-seekable stream support;
6. parallel copy for one file;
7. UI settings for validation sizes or rollback limits;
8. provider-to-provider stream copy for virtual filesystems;
9. a new logging infrastructure only for this feature;
10. moving or rewriting the file panel architecture;
11. append.

## Acceptance criteria

- The copy start dialog offers `Reliable` before copying starts.
- `Reliable` is available when copying a folder, and it applies to existing files found
  inside the copied tree.
- `Reliable` is not offered for move/rename or provider copy.
- With `source = 100 MB` and `destination = first 40 MB of source`, reliable copy
  validates the tail, keeps the destination length, resumes at 40 MB, and produces a
  destination byte-for-byte equal to source.
- With `source = 100 MB` and `destination = first 40 MB of source` with a corrupted
  tail, reliable copy detects the mismatch, rolls back to an earlier safe offset,
  truncates the destination, resumes from that offset, and produces a destination
  byte-for-byte equal to source.
- If `destination.Length > source.Length`, paranoid resume is not performed and the user
  must choose an ordinary safe action.
- If `destination.Length == source.Length`, paranoid resume is not performed and the
  user must choose an ordinary safe action.
- If no matching range is found within the rollback limit, the file is not resumed
  silently.
- Progress shows validation before resumed copy continues.
- Progress shows resume offset and rollback bytes when available.
- Existing overwrite, skip, rename, only-newer, mask filtering, timestamp preservation,
  attribute preservation, symlink mode, and directory recursion behavior still work.
- Retry reopens both files, reads actual destination length from disk, reruns tail
  validation, and refuses automatic resume when the source snapshot changed.
- Reliable retries destination write failures through the same reopen, reread, and
  tail-validation flow used for source read failures.

## Verification

Automated coverage verifies safe resume for valid, empty, oversized, corrupted,
and changed-source destinations; unsafe cases require the normal conflict
decision. It also verifies reliable copy through directory copying and its
progress state.

Manual checks:

1. Copy one large file over a partial valid destination with `Reliable` selected.
2. Copy one large file over a partial destination with a corrupted tail.
3. Copy a folder containing partial destination files and verify each file follows the
   selected mode.
4. Try equal-size and larger destination files and verify no silent resume happens.
