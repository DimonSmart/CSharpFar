# 0018 - Command history completion

Related:
- 0013
- 0016
- 0017

## Goal

Improve command-line history so executed commands are stored once, repeated commands
move to the top of history, and command entry can use Far-like history completion.

## Context

CSharpFar already has a command line and command execution flow. The current request
changes how command history is stored and how the command line behaves depending on
workspace mode:

1. history must not accumulate duplicate commands;
2. repeating an existing command must make it the most recent history item;
3. in `HiddenCommandLine`, Up and Down must browse command history in the command
   line;
4. in `Panels`, typing a command prefix must show a non-blocking
   suggestion window with commands that start with the typed text.

## Scope

Implement command history as a unique, recency-ordered list.

Rules:

1. Before executing a command, remove its leading space characters. Preserve all other characters, including trailing spaces.
2. Store the normalized command in history; do not store empty or whitespace-only commands.
3. If an executed command already exists in history, remove the old entry and add the
   command as the most recent item.
4. Keep history ordering predictable for `Alt+F8`, Up/Down browsing, and completion.
5. Preserve existing command execution behavior except for the removal of leading spaces.

Implement `HiddenCommandLine` history browsing:

1. In `HiddenCommandLine`, Up and Down browse command history in the command line
   instead of panel selection.
2. Browsing should replace the current command-line text with the selected history
   entry.
3. Browsing is not circular. When the user reaches the oldest or newest history item,
   additional Up or Down presses stay on that item.
4. Editing or typing after browsing should continue from the text currently displayed.

Implement `Panels` command completion:

1. In `Panels`, when the user starts typing command text, show a suggestion window
   above the command line.
2. Suggestions contain history commands that start with the current command-line text.
3. Suggestions are unique and ordered by history recency.
4. The suggestion window always starts with one empty neutral item selected by
   default. The neutral item means "do not use completion".
5. Up and Down move the selection inside the suggestion window while it is visible.
6. Enter with the neutral item selected executes the command line exactly as typed
   and does not insert a suggestion.
7. Enter accepts a selected non-empty suggestion and places that command text into
   the command line without executing it.
8. A mouse click on a non-empty suggestion accepts that suggestion and places it
   into the command line without executing it.
9. A mouse click on the neutral item closes the suggestion window without changing
   the command line.
10. Escape temporarily hides the suggestion window.
11. After Escape hides the window, the next command-line typing/editing key may show it
   again if matches exist.
12. The suggestion window must not block normal command-line typing or editing, except
   that Up and Down select suggestions while the window is visible.
13. The window is displayed above the command line with a single-line border and no
   shadow.
14. The window content uses up to 8 rows including the neutral item, plus the border
   rows.
15. If the console is too short, reduce the suggestion window height or hide it.
16. When a non-empty suggestion is selected, the command-line cursor is at the end of
   the typed text, and the user presses Delete, remove that selected command from
   command history and from the visible suggestion list without changing the typed
   command line.
17. Delete does not remove the empty neutral item. When the cursor is not at the end
   of the command line, Delete keeps its normal command-line editing behavior.

## Non-goals

Do not implement in this spec:

1. persistent command history across application restarts unless it already exists;
2. fuzzy, substring, or case-sensitive mode switches for completion;
3. a one-panel UI mode;
4. configurable history size or key bindings;
5. changing `Alt+F8` semantics beyond duplicate-free recency ordering;
6. completion for file names, paths, environment variables, or aliases;
7. an external or general-purpose UI framework for the suggestion window.

## Acceptance criteria

- Executing the same command several times leaves one history entry.
- Leading spaces are removed before a command is launched and before it is saved in history.
- Repeating an old command moves it to the most recent history position.
- `Alt+F8` shows duplicate-free command history in recency order.
- In `HiddenCommandLine`, Up and Down browse command history in the command line.
- `HiddenCommandLine` history browsing is not circular; it stops at the oldest and newest
  commands.
- In `Panels`, typing a command prefix shows a suggestion window above the
  command line when matching history commands exist.
- The suggestion window lists only commands that start with the current command-line
  text.
- The suggestion window has a single-line border and no shadow.
- The suggestion window starts with a selected empty neutral item.
- Up and Down select suggestions while the suggestion window is visible.
- Enter with the default neutral item executes the typed command unchanged.
- Enter accepts a user-selected non-empty suggestion into the command line without
  executing it.
- A mouse click on a non-empty suggestion accepts it into the command line without
  executing it.
- A mouse click on the neutral item closes the suggestion window without changing the
  command line.
- Escape hides the suggestion window until the next command-line typing or editing key.
- Command-line typing and editing continue while suggestions are shown.
- Pressing Delete on a selected non-empty suggestion while the command-line cursor is
  at the end removes that command from history and from the suggestion window without
  changing the typed command.
- Pressing Delete on the empty neutral item does not remove history, and pressing
  Delete before the end of the command line keeps normal command-line editing behavior.
- The suggestion window uses at most 8 content rows including the neutral item, plus
  its border, and hides or shrinks cleanly on short consoles.
- Existing panel navigation, `Ctrl+O`, `Alt+F8`, and command execution behavior still
  work outside the new command-line cases.

## Verification

Manual checks:

1. Execute `dir`, another command, then `dir`; verify history contains one `dir` and it
   is most recent.
2. Execute a command prefixed with spaces; verify that it runs and appears in history without those spaces.
3. Press `Alt+F8`; verify duplicate-free recency order.
4. Hide panels with `Ctrl+O`, press Up/Down, and verify command-line history browsing.
5. Show panels, type a prefix with known matches, and verify suggestions appear above
   the command line.
6. While suggestions are visible, press Enter on the default empty item and verify the
   typed command executes unchanged.
7. While suggestions are visible, press Down then Enter and verify the selected
   suggestion is inserted without executing it.
8. While suggestions are visible, click the empty item and a non-empty suggestion and
   verify the requested behavior.
9. Press Escape while suggestions are visible and verify the window hides until the
   next command-line typing or editing key.
10. While suggestions are visible, select a non-empty suggestion with the command-line
   cursor at the end, press Delete, and verify the command disappears from the
   suggestion window and `Alt+F8` history while the typed prefix remains unchanged.
11. With the neutral item selected, or with the cursor before the end of the command
   line, press Delete and verify history is not removed and normal command-line
   editing behavior is preserved.
12. Try the same flow in a short console window and verify suggestions shrink or hide
   without corrupting the command line.
