## Goal

Add a provider-based panel source model and a built-in SFTP source so a user can
connect to an SFTP server, browse remote directories in a normal panel, and use
the existing Far-style panel workflows for viewing, copying, creating folders,
renaming, and deleting remote items where the source supports those operations.

## Context

This specification is based on the shared implementation brief:

`https://chatgpt.com/s/6a04444543e48191be46d5324d7a18ac`

The current application is still mostly path-based:

- `FilePanelState.CurrentDirectory` is a local path.
- `PanelViewRequest.DirectoryPath` is a local path.
- `PanelViewBuilder` reads through `IFileSystemService`.
- `FileOperationService` plans and executes operations with `System.IO` paths.
- `PanelProviderCapabilities` already gates operations for local panels and
  search-result panels from spec `0012`.

Spec `0012` deliberately stopped before general provider-to-provider stream
copy. Spike `0023` recommends replacing search-specific fields on
`FilePanelState` with a small source/location model before adding another
virtual panel source. SFTP is the first source that needs that model.

## Scope

### Panel source model

Introduce explicit source and location models before implementing SFTP:

1. Add `PanelSourceId` and `PanelLocation`. `PanelLocation` pairs a source id
   with a provider-owned absolute path.
2. Replace path-only panel state with source-aware state. `FilePanelState`
   must know its current source and source path. Search-specific state should
   move behind the same source/location model or become source metadata that
   does not make local panels know about search internals.
3. Selection identity must include source id and source path. A path string by
   itself is not enough once two panels can point at different sources.
4. Do not store `sftp://...` values in fields whose names imply a local
   filesystem path. If a model still has a property named `FullPath`, it must be
   audited so direct `System.IO` callers only consume it for the local source.
   Prefer names such as `SourcePath`, `DisplayPath`, and `PanelLocation` for
   provider paths.
5. Keep `PanelProviderCapabilities` as the command-gating mechanism. Add new
   flags only if a required SFTP behavior cannot be expressed by the existing
   flags.

### Provider contracts

Add `IFilePanelSource` in Core. The name is intentional: the abstraction feeds a
panel with file-like items, but a source can be local files, search results, or
SFTP. The contract must cover:

1. Source id, display name, root/current path normalization, and capabilities.
2. Directory enumeration returning panel items for a provider path.
3. Parent path resolution and root detection without assuming Windows drive
   paths.
4. `OpenReadAsync` and `OpenWriteAsync` with `CancellationToken` for file
   streams.
5. `CreateDirectoryAsync`, `DeleteAsync`, and `RenameAsync`.
6. Existence or item-info lookup needed by conflict handling and refresh.
7. Clear exception boundaries so provider errors become user-readable file
   operation errors.

The local filesystem provider is the real local provider, not a compatibility
shim. Existing local behavior should move behind it, and callers/tests should be
updated rather than keeping duplicate path-only APIs for compatibility.

### Local and search sources

1. Implement `LocalFilePanelSource` over current local filesystem behavior.
2. Convert normal panels to use the provider model while preserving existing
   local navigation, sorting, selection, status, volume information, refresh,
   watching, create/copy/move/delete, and shell launch behavior.
3. Convert the search results panel from spec `0012` to the same source model.
   It remains read-only and keeps its existing behavior: copy out to local is
   allowed, copy into search results is blocked, and `Enter` navigates to the
   real local item.

### SFTP source

Add a built-in SFTP source provider using `SSH.NET` (`Renci.SshNet`).

SFTP provider behavior:

1. Password authentication is required for the MVP.
2. Private-key authentication is not part of the MVP.
3. Remote paths use POSIX-style absolute paths owned by the SFTP provider.
4. `Capabilities` include enumerate, open read, open write, create directory,
   delete, rename, copy from, copy to, and refresh.
5. `Watch` is not exposed for SFTP.
6. Local-to-SFTP and SFTP-to-local copy stream through provider `OpenReadAsync`
   and `OpenWriteAsync` and report progress through the existing progress UI.
7. SFTP-to-SFTP copy between different connections is not supported in the MVP.
8. Cross-provider move is not supported in the MVP. The command must be
   capability-blocked with a clear message.
9. Remote delete is permanent. Do not route it through the Windows recycle bin.
10. Remote view (`F3`) must work. If the current viewer requires a local file
    path, the provider may materialize the file into a temporary file and clean
    it after the viewer closes.
11. Remote edit (`F4`) is out of scope for this MVP.

### Connections, credentials, and host keys

1. Add an `SftpConnectionInfo` model with at least:
   - id;
   - display name;
   - host;
   - port;
   - username;
   - remote root path;
   - credential id when a password is saved;
   - expected host key fingerprint when the connection is saved.
2. Store connection metadata in `connections.json` under the existing user
   configuration directory.
3. Do not store plaintext passwords in the metadata file.
4. For the MVP, store saved passwords encrypted for the current Windows user
   with DPAPI, referenced by credential id.
5. If the user does not save the password, prompt for it on each connection.
6. On first connection to a host, show the host key fingerprint and require
   explicit acceptance before opening the panel. Saved connections persist the
   accepted fingerprint. A changed fingerprint blocks the connection with a
   clear warning.

### UI integration

1. Extend the volume/source selection flow from `Alt+F1` and `Alt+F2` so it can
   show local volumes, saved SFTP connections, and an `SFTP...` action.
2. Add a Plugins menu entry for SFTP connection management. A generic external
   plugin loader is not required.
3. Add console dialogs in the style of existing dialogs for:
   - creating a temporary SFTP connection;
   - saving a connection;
   - connecting to a saved connection;
   - deleting a saved connection;
   - entering or re-entering a password.
4. After a successful connection, the chosen panel opens the configured remote
   root path. The panel title/status must make the remote source obvious.
5. Unsupported operations must be disabled through capabilities or fail with a
   clear message. They must not fall through to local `System.IO` calls.

### File operations

1. Update file-operation requests so source items and destinations carry source
   identity and source paths, not only local strings.
2. Keep local-to-local behavior from specs `0010` and `0019`, including conflict
   handling, progress, cancellation, masks, paranoid copy/resume behavior, and
   metadata preservation where already supported.
3. Add provider-aware stream copy for local-to-SFTP and SFTP-to-local.
4. Provider operations must report item-level errors without crashing the app.
5. Cancellation must be passed through public async methods.

## Non-goals

1. Loading third-party plugin DLLs from a `Plugins` folder.
2. FTP, S3, archive, Git, favorites, or other provider implementations.
3. Private-key SFTP authentication.
4. Remote file watching.
5. Remote shell execution or launching remote executables.
6. Remote edit with temp-file synchronization.
7. SFTP-to-SFTP copy between different connections.
8. Cross-provider move between local and SFTP.
9. Compatibility wrappers that keep old path-only application APIs alive only to
   avoid updating callers.
10. Cross-platform secret storage. This project currently targets Windows console
   behavior.

## Acceptance criteria

1. Local filesystem panels still load, refresh, watch, sort, select, open,
   view, copy, move, delete, create folders, and show volume information as
   before.
2. Search result panels still behave as a read-only source and keep the behavior
   accepted in spec `0012`.
3. Generic panel state no longer needs search-specific fields to represent a
   non-local panel source.
4. No SFTP operation reaches `System.IO` with a remote provider path.
5. `Alt+F1` and `Alt+F2` show local volumes, saved SFTP connections, and an
   `SFTP...` action.
6. A temporary password-based SFTP connection can open a remote root in the
   active panel.
7. A saved SFTP connection stores metadata without plaintext secrets and can
   reconnect after app restart.
8. First connection requires host key fingerprint acceptance. A changed saved
   fingerprint blocks reconnect.
9. Remote navigation supports entering directories and going to parent
   directories without assuming Windows drive roots.
10. `F3` views a remote text file.
11. `F5` copies files from local to SFTP and from SFTP to local with progress and
    cancellation.
12. `F7` creates a remote directory when the SFTP panel is active.
13. `F8` deletes remote files/directories with the existing confirmation style
    and without using the recycle bin.
14. Single-source SFTP rename works when the server allows it.
15. Unsupported cross-provider operations are disabled or show a clear
    unsupported-operation message.
16. SFTP connection, authentication, permission, missing-file, and network
    errors are shown as user-readable messages and do not crash the app.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Automated checks:

1. Provider model tests for local, search, and fake remote sources.
2. Panel controller/view builder tests proving source id participates in
   selection identity and refresh.
3. File operation tests for local-to-local regression and fake
   provider-to-provider stream copy.
4. SFTP connection storage tests proving metadata contains no plaintext
   password.
5. Host key fingerprint tests for first accept, matching reconnect, and changed
   fingerprint rejection.

Manual checks:

1. Connect to a known test SFTP server with a temporary connection.
2. Save a connection, restart the app, and reconnect.
3. Browse remote directories, including root and parent navigation.
4. View a remote file with `F3`.
5. Copy local to remote and remote to local with visible progress.
6. Cancel a larger copy and verify partial operation reporting.
7. Create, rename, and delete a remote test directory/file.
8. Verify unsupported operations show clear messages.

## Readiness check

Ready to start implementation.

The implementation scope is intentionally limited to the source model and a
built-in SFTP provider. External plugin loading, FTP/S3, private keys,
SFTP-to-SFTP copy, remote watching, and remote edit are explicitly out of scope.

The important choices are fixed enough for coding:

- provider methods are async where they perform I/O and accept
  `CancellationToken`;
- local and search panels must move to the same source model before SFTP is
  exposed;
- saved passwords use DPAPI, with no plaintext secrets in config;
- host key verification is required before connecting;
- `SSH.NET` is the default SFTP library.
