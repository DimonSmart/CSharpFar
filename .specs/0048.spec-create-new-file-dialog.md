## Goal

Add a Far-like `Shift+F4` command that creates a new text file from the file panels and
opens it in the built-in editor.

The command must start from an ordinary modal text dialog, visually consistent with the
copy/move file operation dialogs, and must let the user choose the initial file encoding,
including Unicode encodings with an explicit BOM.

## Context

Far Manager uses `Shift+F4` for "Open/create file": it asks for a file name, asks for a
code page, then opens the editor for a new or existing file. CSharpFar already has a
built-in editor spec (`0046`) with document format handling, encoding/BOM preservation,
and an editor format dialog for saving. That does not replace the panel-level workflow for
creating a new file directly from the file manager.

This feature should reuse existing UI responsibilities:

1. `FileOperationDialog` style and the shared form-dialog controls for the modal surface.
2. `SingleLineTextInput` and history-enabled text fields for the file path field.
3. `DialogButtonBar` for `{ OK } [ Cancel ]`.
4. `TextEncodingCatalog` and the encoding selection behavior from `0033`/`0046` for code
   page choices.
5. The built-in editor session and save pipeline from `0046` for editing and writing the
   selected document format.

## Scope

Keyboard command:

1. `Shift+F4` from the panel area opens a modal `Open/create file` dialog.
2. The command is available through the same application command binding path as other
   function-key commands.
3. The function-key bar shows the command in a Far-like way when panels are active.
4. `Esc` or `Cancel` closes the dialog without creating or opening anything.

Dialog:

1. Title: `Editor`.
2. First field label: `Open/create file:`.
3. First field value is a path text input.
4. Second field label: `Code page:`.
5. The code page field is a selectable option row, not free text.
6. The default code page option is `Default`, meaning the editor's configured default for
   new files.
7. The dialog uses the same frame, palette, spacing, focus highlight, dropdown arrow style,
   and button bar conventions as existing file-operation dialogs.
8. The path field starts with a useful default based on the active panel directory.
9. Relative paths are resolved against the active panel directory.
10. The dialog rejects an empty path with validation feedback and remains open.

Encoding choices:

1. The code page selector must include at least:
   - `Default`;
   - `UTF-8`;
   - `UTF-8 with BOM`;
   - `UTF-16 LE with BOM`;
   - `UTF-16 BE with BOM`;
   - `Windows-1251`;
   - `OEM 866` when available from the registered code pages provider.
2. Unicode BOM behavior is explicit in the selected option. For example, `UTF-8` writes no
   BOM, while `UTF-8 with BOM` writes a UTF-8 preamble.
3. The selected code page becomes the initial `EditorDocumentFormat` for a newly created
   editor session.
4. If the target file already exists, the editor opens the existing file using normal
   detection/preservation rules from `0046`; the selected code page must not silently
   rewrite the existing file's format before the user saves.
5. If the target file does not exist, no bytes are written until the user saves from the
   editor, unless implementation discovers that an empty placeholder is required for a
   provider path. Stop and review before adding placeholder writes.

Editor handoff:

1. `OK` opens the target path in the built-in editor.
2. New editor sessions for missing files start with an empty document, clean state, and the
   selected initial encoding/BOM.
3. The editor header/status shows the selected encoding/BOM before first save.
4. `F2` saves the new file using the selected encoding/BOM and the editor's default line
   ending for new files.
5. If saving fails, the editor stays open and dirty handling follows `0046`.
6. After a successful first save, the active panel refreshes and can select the new file
   when it belongs to the current panel source.

History:

1. The path field participates in file-path input history using a logical key specific to
   this command.
2. The last explicitly selected code page may be remembered for the process or settings
   only if that matches existing encoding-dialog conventions at implementation time.
3. Do not invent a new persistence subsystem for this dialog.

Provider behavior:

1. Local filesystem paths are in scope.
2. Provider-backed panel paths may be supported only through existing panel/provider edit
   and save semantics.
3. Stop and create a follow-up spec if provider-backed creation needs a new upload,
   temporary-file, or transaction model.

## Non-goals

Do not implement in this spec:

1. a new general-purpose dialog framework;
2. a separate editor model or save path outside `0046`;
3. binary, hex, or non-text file creation modes;
4. templates, snippets, or file-type-specific initial content;
5. multi-file creation;
6. directory creation;
7. overwrite confirmation beyond the normal "open existing file in editor" behavior;
8. exact Far Manager code page list parity;
9. persistent per-directory or per-extension encoding defaults;
10. new remote provider creation semantics unless an existing provider path already
    supports the editor workflow cleanly.

## Acceptance criteria

- Pressing `Shift+F4` from panels opens an `Editor` dialog with `Open/create file:` and
  `Code page:` rows plus `{ OK } [ Cancel ]`.
- The dialog visually follows the existing Far-like file operation dialog style.
- Empty path input is rejected without closing the dialog.
- Relative paths resolve against the active panel directory.
- `Default` opens a new empty editor session with the configured new-file default format.
- `UTF-8` saves a new file as UTF-8 without BOM.
- `UTF-8 with BOM` saves a new file with a UTF-8 BOM.
- `UTF-16 LE with BOM` and `UTF-16 BE with BOM` save new files with the corresponding
  BOM and byte order.
- Opening an existing file through this dialog preserves the existing file's detected
  encoding/BOM until the user explicitly changes it.
- Cancelling the dialog does not create a file and does not open the editor.
- After saving a new local file, the active panel refreshes and shows the file.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused automated checks:

- `Shift+F4` command dispatch opens the create-file editor dialog from panel focus.
- Dialog `OK` maps path and code page selection into an editor open request.
- Empty path validation leaves the dialog open.
- New-file save byte checks for UTF-8 without BOM, UTF-8 BOM, UTF-16 LE BOM, and
  UTF-16 BE BOM.
- Existing-file open ignores the selected create-code-page for initial detection.
- Panel refresh/select behavior after first save of a new local file.

Manual checks:

1. Press `Shift+F4`, enter a relative filename, choose `UTF-8 with BOM`, type text, save,
   and verify the BOM bytes.
2. Repeat with `UTF-16 LE with BOM` and Cyrillic text.
3. Enter the name of an existing Windows-1251 file, choose a different code page in the
   dialog, save without format changes, and verify the original encoding is preserved.
4. Cancel the dialog and verify no file appears in the panel.
5. Use the path field history after creating several files.
