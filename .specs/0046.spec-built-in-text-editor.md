## Goal

Turn the current F4 editor into a predictable Far-like text editor with a real internal
editor model, explicit file format handling, undo/redo, search, selections, bookmarks,
and editor-domain events that can later support native CSharpFar services.

The first implementation must remove the current 10 MB hard stop as a product behavior,
preserve the source file encoding and line endings by default, and make editor operations
testable outside the console loop.

## Context

The current editor is intentionally small:

1. `FileEditor` rejects files larger than `TextFileReader.MaxFileSizeBytes` (`10 MB`).
2. `TextFileReader` reads the whole file into memory and returns `string[]` plus an
   `Encoding`.
3. `EditorModel` stores a mutable `List<string>`, cursor row/column, and one dirty flag.
4. Saving uses `File.WriteAllText(filePath, model.GetText(), encoding)`, so it writes
   `Environment.NewLine` instead of the source file line ending.
5. There is no undo/redo, selection, rectangular block, search, replace, bookmarks,
   syntax coloring, editor status model, read-only mode, or editor-domain event
   surface.

Recent active work already gives the editor useful building blocks:

1. `0032` made the viewer streaming and large-file capable.
2. `0033` added shared encoding detection and an encoding catalog.
3. `0012` added search contracts, progress UI patterns, and panel provider capabilities.
4. `0013` made the function-key bar data-driven.
5. `0020` standardized scrollbar rendering for window-like UI.
6. `0038` documented the reusable control ecosystem.
7. CSharpFar uses native product models rather than a compatibility proxy
   plugin architecture.

The comparison report is useful and mostly directionally correct, but its roadmap needs
these adjustments before implementation:

1. Block reading alone is not a large-file editor design. Editing needs an explicit text
   buffer, line index, dirty revision, save pipeline, and undo transactions.
2. CSharpFar should not add a parallel `IEditorHighlighter`, plugin, or adapter layer.
   Editor behavior must be built into CSharpFar's internal editor model first.
3. Encoding and BOM work should reuse `TextEncodingDetector` and `TextEncodingCatalog`
   from `0033`, not add another detector.
4. Search/replace keys inside the editor must not steal `F2` from save. `F2` remains save;
   editor search starts from `F7`, with replace assigned separately.
5. Macro language, Lua scripting, LSP, formatters, and full Far Manager editor API
   compatibility are later capabilities. They should not be started before the editor core
   is stable.

## Scope

Implement the editor in staged slices. Each stage should leave the product coherent and
should not keep compatibility wrappers for replaced editor internals unless another spec
explicitly asks for them.

### Stage 1. Editor session, file format, and save correctness

Introduce a model that separates the editor core from the console loop.

Preferred names:

```csharp
public sealed class EditorSession
public sealed class EditorDocument
public sealed class EditorDocumentFormat
public enum EditorLineEnding
public sealed class EditorPosition
public sealed class EditorViewport
```

Naming rules:

1. Use `EditorDocumentFormat`, not a generic `FileFormat`, because the model is editor
   specific.
2. Use `EditorLineEnding`, not `NewLineMode`, because the stored value describes file
   line separators.
3. Use `EditorSession` for per-open-file UI/editor state: document, path, cursor,
   viewport, selection, read-only mode, and editor-domain event dispatch.
4. Keep side effects out of `EditorDocument`; loading and saving belong in an explicit
   service such as `EditorFileService`.

File format requirements:

1. Detect and store the original encoding, BOM state, and line ending style when the file
   is opened.
2. Use the shared `TextEncodingDetector` for encoding detection.
3. Move reusable line-ending detection into a shared text component instead of keeping it
   private in `TextFileInfo`.
4. Supported line endings:
   - DOS/Windows: `CRLF`;
   - Unix: `LF`;
   - classic Mac: `CR`;
   - mixed.
5. Mixed line endings should be displayed as `Mixed` and saved exactly as loaded until
   the user explicitly selects a single output line ending.
6. New files default to UTF-8 without BOM and the platform-independent editor default line
   ending from settings, not blindly to `Environment.NewLine`.
7. Existing files preserve encoding, BOM behavior, and line endings by default.
8. `F2` saves with the session's selected format.
9. `Shift+F2` opens an editor format dialog that can change encoding, BOM, and line
   endings before saving.
10. The format dialog should reuse the existing encoding catalog style from the viewer and
    add line-ending/BOM controls.

Save requirements:

1. Saving must not use `EditorModel.GetText()` with `Environment.NewLine`.
2. Saving must write the selected line ending and encoding explicitly.
3. Successful save updates the clean revision; dirty state must be based on document
   revision, not only on a boolean that can drift after undo.
4. Failed save leaves the editor open and preserves dirty content.
5. Read-only file attributes must produce a clear save error or read-only open mode based
   on settings.
6. Atomic save via a temporary file and replace is preferred for local files, but stop and
   review if this conflicts with cross-platform metadata preservation.

File size behavior:

1. Replace the hard 10 MB product limit with an editor file-size warning setting:
   `Editor.FileSizeLimitBytes`.
2. `0` means no warning, matching the Far-style configuration meaning.
3. A warning threshold is not a hard block. If the user confirms, the editor attempts to
   open the file.
4. If the file cannot be represented safely because of memory pressure or an unsupported
   buffer mode, fail with a clear message and leave the panels usable.

### Stage 2. Text buffer and undo/redo

Replace the current per-line string mutation model with an editor buffer that can support
large text and transaction-based undo.

Preferred names:

```csharp
public interface IEditorTextBuffer
public sealed class EditorUndoHistory
public sealed class EditorTextChange
public sealed class EditorTransaction
```

Buffer requirements:

1. The buffer exposes line count, line retrieval, insert, delete, replace, and line-ending
   serialization operations.
2. Cursor positions are logical character positions. Rendering maps them to visual columns
   using tab settings.
3. Do not replace tabs with single spaces in the data or render path.
4. Add editor settings for `TabSize` and `ExpandTabs`.
5. Preserve exact tab characters when `ExpandTabs` is false.
6. Keep line indexing incremental enough that inserting a character does not rebuild the
   whole document.
7. A piece-table, rope, or gap-buffer design is acceptable. Choose the smallest design that
   supports the acceptance tests without surprising class responsibilities.
8. If true large-file editing requires a broader storage decision, stop and create an ADR
   or spike before adding a partial hidden design.

Undo/redo requirements:

1. Add `Ctrl+Z` undo and `Ctrl+Y` redo.
2. Store undo as editor transactions, not raw key presses.
3. Coalesce ordinary typed character runs into useful undo units.
4. Keep destructive commands such as paste, block delete, replace, and line join as clear
   separate transactions.
5. Make undo depth configurable as `Editor.UndoSize`, defaulting to `2048`.
6. `0` disables undo history.
7. Undo/redo must restore text, cursor, selection, and dirty revision state.
8. New edits after undo clear redo history.

Minimum performance target:

1. Opening, editing one line, saving, and closing a 50 MB UTF-8 text file must work in
   automated tests.
2. A 100 MB text file should be covered by a manual or long-running test path.
3. The editor should not keep both a `string[]` line copy and another full serialized text
   copy as steady-state data.

### Stage 3. Navigation, search, replace, and word rules

Add editor-local navigation and text operations.

Word/navigation requirements:

1. Add `Editor.WordDiv` setting for word separators.
2. `Ctrl+Left` and `Ctrl+Right` use `Editor.WordDiv`.
3. Whole-word search uses the same word boundary rules.
4. `Home`, `End`, `Ctrl+Home`, `Ctrl+End`, `PageUp`, and `PageDown` keep their current
   meanings.
5. Add `AllowEmptySpaceAfterEof` setting for editor scrolling below the last line.

Search/replace requirements:

1. `F7` opens editor find.
2. `Shift+F7` repeats the last find in the previous direction.
3. `Ctrl+F7` opens replace.
4. `F2` remains save.
5. Support forward/backward direction, case sensitivity, whole words, and regular
   expressions.
6. Add `Editor.F7StartsAtNextCharacter` or similarly explicit setting to model whether a
   forward find starts at the current position or after it.
7. Searching inside a selection is supported for normal linear selection.
8. Regex failures show validation feedback and do not mutate text.
9. Replace all is one undo transaction.

### Stage 4. Selection, rectangular blocks, bookmarks, and status

Add expected editor interaction features.

Selection requirements:

1. Support linear selection with shift+arrows and keyboard commands.
2. Support rectangular selection with an explicit selection mode.
3. Rectangular selection operations include copy, cut, delete, paste, and type-over insert.
4. Add `Editor.BSLikeDel` for Backspace behavior on rectangular selection.
5. Deleting a selection is one undo transaction.
6. The selection model must be independent from rendering.

Clipboard requirements:

1. Use an existing clipboard abstraction if one exists at implementation time.
2. If no cross-platform clipboard service exists, implement editor-internal copy/paste
   first and record OS clipboard integration as follow-up.

Bookmark requirements:

1. Support line bookmarks.
2. Support navigation to next/previous bookmark.
3. Bookmarks move predictably when lines are inserted or deleted before them.
4. Bookmark storage is per editor session in this spec; persistence is out of scope.

Status/rendering requirements:

1. The editor header/status shows file path, dirty marker, read-only marker, encoding,
   BOM state, line ending, cursor row/column, and current character code.
2. Character code display supports decimal and hexadecimal at minimum.
3. The editor uses the same `ScreenRenderer` frame style and palette conventions as the
   rest of the app.
4. The editor function-key bar is driven from editor command bindings, not hard-coded
   writes in `FileEditor.Draw`.
5. Add an editor scrollbar using the shared `ScrollBarRenderer` where it fits the
   full-screen editor layout.
6. Support vertical text scrolling with the mouse wheel.

### Stage 5. Editor-domain events

Expose editor-domain events only after the core editor model is stable.

Editor-domain surface:

1. Model editor events as part of CSharpFar's internal editor domain, not as a separate
   plugin system or adapter layer.
2. Internal subscribers can observe editor open, input, text changed, save, close, and redraw
   events.
3. Internal services can request color spans for visible ranges.
4. The exposed editor context includes cursor, selection, file path, document format, and
   visible text.
5. Mutating operations must go through editor commands or buffer transactions so undo and
   dirty tracking remain correct.
6. A built-in C# syntax-color provider may be added as the first internal service, but the core
   editor must not depend on C#-specific logic.

## Non-goals

Do not implement in this spec:

1. a separate CSharpFar plugin, highlighter, or adapter abstraction for editor behavior;
2. Far Manager native plugin ABI compatibility;
3. exact Far macro language or Lua macro compatibility;
4. Language Server Protocol, autocomplete, diagnostics, formatters, or linters;
5. multi-tab or multi-window editor management;
6. persistent bookmarks;
7. hex editing or binary editing;
8. editing inside archives;
9. changing FTP/SFTP/provider editing semantics except where current F4 temp-file editing
   needs to keep working;
10. perfect arbitrary-size editing beyond available memory without a separate storage ADR;
11. replacing the streaming F3 viewer;
12. compatibility wrappers for the old `EditorModel` API after tests are migrated.

## Acceptance criteria

Stage 1:

- Opening a file larger than 10 MB is no longer blocked solely by
  `TextFileReader.MaxFileSizeBytes`.
- `Editor.FileSizeLimitBytes = 0` means no size warning.
- Existing files preserve detected encoding, BOM state, and line ending style by default.
- `F2` saves using the selected editor document format, not `Environment.NewLine`.
- `Shift+F2` lets the user change encoding, BOM, and line ending settings.
- Failed save keeps the editor open and leaves the document dirty.
- Read-only files follow the configured read-only behavior and never fail silently.

Stage 2:

- Basic insert, delete, line split, line join, and cursor movement work through
  `IEditorTextBuffer`.
- Tabs render with predictable visual columns and are not converted to single spaces.
- `Ctrl+Z` and `Ctrl+Y` undo/redo text, cursor, selection, and dirty state.
- Undo depth follows `Editor.UndoSize`.
- A 50 MB text fixture can be opened, edited, saved, and reopened by automated tests.

Stage 3:

- `Ctrl+Left` and `Ctrl+Right` use `Editor.WordDiv`.
- `F7` find supports direction, case sensitivity, whole words, and regex.
- `Ctrl+F7` replace supports single replace and replace all.
- Replace all is undoable as one transaction.
- `F2` still saves after search/replace is added.

Stage 4:

- Linear selection supports keyboard selection and delete/copy/cut/paste.
- Rectangular selection supports delete and paste at minimum.
- `Editor.BSLikeDel` controls Backspace behavior for rectangular selection.
- Bookmarks can be set, cleared, and navigated within one editor session.
- Header/status shows dirty, read-only, encoding, BOM, line ending, cursor, and character
  code data.
- The editor key bar is rendered from editor command bindings.
- The mouse wheel scrolls the visible text vertically.

Stage 5:

- Internal editor-domain subscribers can observe open/input/change/save/close/redraw events.
- Color spans can be supplied for visible ranges without mutating editor text.
- A built-in or test syntax color provider can color visible text.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused automated checks:

- Format detection and save round-trip for UTF-8 without BOM, UTF-8 BOM, UTF-16 LE,
  Windows-1251, CRLF, LF, CR, and mixed line endings.
- `Shift+F2` format selection changes saved bytes as expected.
- Save failure on read-only files leaves the editor open and dirty.
- Undo/redo transaction coalescing for typing, paste, line split, line join, replace, and
  block delete.
- Word navigation and whole-word search with custom `Editor.WordDiv`.
- Regex find/replace validation and replace-all undo.
- Linear and rectangular selection mutation tests.
- Bookmark movement after inserting and deleting lines.
- Large fixture open/edit/save tests for at least 50 MB.
- Editor command binding and key-bar rendering tests.
- Color span and editor-domain event ordering tests.

Manual checks:

1. Open a large log file, accept the size warning, edit one line, save, and reopen.
2. Open a UTF-8 BOM file, save without format changes, and verify the BOM remains.
3. Change the same file to UTF-8 without BOM and LF through `Shift+F2`.
4. Open a Windows-1251 file, edit Cyrillic text, and save without mojibake.
5. Use `F7`, `Shift+F7`, and `Ctrl+F7` in the editor.
6. Use linear and rectangular selections on tab-indented text.
7. Open a read-only file and verify the configured warning/lock behavior.
8. Verify editor status and key bar updates after save, undo, selection, and format changes.
