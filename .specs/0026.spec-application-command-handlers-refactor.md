# 0026 SPEC Application command handlers refactor

Related:

## Goal

Reduce `Application` command responsibility by moving concrete user command
handlers into dedicated command files without changing behavior.

## Context

Worklog 0025 moved function-key metadata out of `Application`, but command
execution still lives in large `Application` switch blocks and `Handle*`
methods. Architecture review 0023 identifies command dispatch and application
orchestration as part of the main `Application` hotspot.

## Scope

- Add command abstractions, command context, and a command registry.
- Route function-key and menu command ids through the same registry.
- Move concrete function-key, menu, settings, panel, history, search, file, and
  volume command bodies into separate command files.
- Keep shared helpers and low-level input/rendering orchestration outside command
  files.
- Add focused command registry/dispatcher tests.

## Non-goals

- Do not change runtime behavior.
- Do not change `Program.cs`.
- Do not introduce partial `Application` files.
- Do not refactor command completion, hidden console underlay, mouse input,
  rendering, or file operation internals beyond shared helpers needed by
  commands.
- Do not add configurable command bindings or compatibility shims.

## Acceptance criteria

- `Application` no longer contains concrete command handler bodies for the
  extracted user commands.
- Function-key and menu command ids execute through one command registry.
- Existing behavior tests keep passing.
- Focused tests cover registry coverage and invalid menu arguments.
- Repository checks pass.
