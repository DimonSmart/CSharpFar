## Goal

Понять, что нужно изменить, чтобы CSharpFar мог работать под Unix-like системами,
в первую очередь Linux и macOS.

Нужно зафиксировать:

- места кода, где поведение Windows, Unix и macOS существенно различается;
- варианты разрешения этих различий;
- рекомендуемую структуру solution и файлов для multi-platform реализации;
- проверки, которые должны подтвердить портируемость.

## Hypothesis

Основная доменная модель, панели, сортировка, маски, поиск, viewer, editor, FTP и
SFTP в целом переносимы, потому что используют .NET abstractions и уже отделены от
консольного драйвера.

Основные блокеры находятся не в `Core`, а на границах процесса:

1. console input/output;
2. запуск shell-команд и открытие файлов;
3. список томов/корней и mount point metadata;
4. credential storage;
5. clipboard;
6. FarNet compatibility surface;
7. composition root в `Program.cs`;
8. тесты, которые закрепляют Windows-пути и Windows launch semantics.

## Constraints

- Do not change production behavior during this spike.
- Do not implement Unix/macOS support in this document.
- Do not keep legacy compatibility shims only to preserve current Windows class
  names or fallback behavior.
- Keep platform-specific behavior explicit in names, project placement, and
  composition.
- Prefer predictable runtime selection in the composition root over platform
  checks scattered through domain code.

## Done when

- current portability blockers are documented with file paths;
- major Windows/Unix/macOS behavior differences are listed;
- recommended solution structure is written;
- follow-up implementation slices are clear;

### High-impact differences

#### Console rendering and input

Files:

- `src/CSharpFar.Console/SystemConsoleDriver.cs`
- `src/CSharpFar.Console/Win32/Win32ConsoleApi.cs`
- `src/CSharpFar.Console/Win32/Win32ModifierKeyTracker.cs`
- `src/CSharpFar.Console/IConsoleDriver.cs`
- `src/CSharpFar.Console/ScreenRenderer.cs`
- `src/CSharpFar.App/Application.cs`

Current state:

- Windows uses Win32 console input records, mouse events, resize events,
  viewport snapshots, `ReadConsoleOutput`, `WriteConsoleOutput`, console mode
  flags, palette mutation, and explicit screen buffer sizing.
- Non-Windows code path in `SystemConsoleDriver` is key-only and writes with
  `System.Console`. It does not provide equivalent mouse input, resize events,
  screen capture/restore, scrollback control, or robust alternate-screen
  behavior.
- `Application` still contains direct `System.Console` access through local
  `SysConsole` wrappers and `TryReadConsoleKey`. This matters for shell underlay,
  current-console launch, prompt cleanup, and tests.

Platform difference:

- Windows console has a screen buffer with a movable viewport. ADR `0016`
  correctly models this.
- Unix terminals normally expose a terminal viewport, escape sequences,
  alternate screen, mouse reporting modes, bracketed paste, and SIGWINCH-driven
  resize behavior. There is no Win32-style `srWindow` buffer API.
- macOS Terminal/iTerm2 follow the Unix terminal model but differ in option-key
  handling, clipboard tools, default shell, and terminal capability details.

Resolution:

- Split console implementation into explicit platform backends:
  - Windows backend keeps Win32 viewport-bound rendering.
  - Unix backend uses terminal escape sequences and termios/VT input parsing.
- Keep `IConsoleDriver` as the app-facing contract only if it can describe both
  models predictably. If not, split it into smaller contracts:
  `IConsoleScreen`, `IConsoleInput`, `IConsoleShellUnderlay`, and
  `IConsoleClipboardMode` or similar names with real responsibilities.
- Remove direct `System.Console` access from `Application`; current-console
  prompt/underlay behavior should be owned by the console backend or a
  console-session service.
- Do not implement Unix as a quiet fallback inside `SystemConsoleDriver`. A class
  named `SystemConsoleDriver` hiding Win32 and VT behavior is already surprising.

#### Shell command execution

Files:

- `src/CSharpFar.Shell/ShellService.cs`
- `src/CSharpFar.App/Program.cs`
- `src/CSharpFar.Core/Models/AppSettings.cs`
- `tests/CSharpFar.Tests/JsonSettingsStoreTests.cs`

Current state:

- `ShellService` defaults to `cmd.exe` and `"/c {0}"`.
- `Program.cs` passes settings directly into `ShellService`.
- Settings and tests assume Windows defaults.

Platform difference:

- Linux default interactive shell is usually `/bin/sh`, `/bin/bash`, or the
  user's `$SHELL`; command execution commonly uses `-c`.
- macOS default user shell is commonly `/bin/zsh`; non-interactive command
  execution can still use `/bin/sh -c` for predictable POSIX semantics.
- Quoting rules differ sharply between `cmd.exe`, PowerShell, POSIX shells, and
  zsh.

Resolution:

- Introduce a platform shell profile selected in composition:
  `WindowsShellProfile`, `UnixShellProfile`, and optionally `MacOsShellProfile`
  only where macOS differs.
- Keep command execution semantics explicit:
  - Windows: configured shell executable and arguments format, default
    `cmd.exe /c`.
  - Unix/macOS: configured shell executable and arguments format, default
    `/bin/sh -c`.
- Avoid a shared placeholder formatter that pretends Windows and POSIX quoting
  are equivalent. If quoting is needed, use platform-specific command builders.

#### File opening and executable launch

Files:

- `src/CSharpFar.Shell/WindowsShellFileLauncher.cs`
- `src/CSharpFar.Core/Abstractions/IFileLauncher.cs`
- `src/CSharpFar.Core/Models/FileLaunchMode.cs`
- `src/CSharpFar.App/Commands/OpenCurrentItemCommand.cs`
- `tests/CSharpFar.Tests/Spec009OpenPanelItemTests.cs`

Current state:

- File launcher is named `WindowsShellFileLauncher`.
- `.exe`, `.com`, `.bat`, `.cmd` are current-console launches.
- Other files use Windows shell association with `UseShellExecute = true` and
  `Verb = "open"`.

Platform difference:

- Unix executables are identified by executable permission and shebang, not file
  extension.
- macOS app/document opening usually uses `open`.
- Linux desktop opening usually uses `xdg-open`, but terminal-only environments
  may not have it.
- Current-console execution should run executable files directly when the file
  is executable and the panel action means "execute", not "open association".

Resolution:

- Rename and split the launcher implementations:
  - `WindowsFileLauncher`;
  - `UnixFileLauncher`;
  - `MacOsFileLauncher` if document/app opening differs enough from Linux.
- Keep the current `FileLaunchMode` concept, but define Unix rules separately:
  current-console if regular file has executable bit or starts with a supported
  script shebang; association/open otherwise.
- Document opener should be platform-specific:
  - Windows shell association;
  - macOS `open`;
  - Linux `xdg-open` or configured opener.
- Do not keep a `WindowsShellFileLauncher` class as the common type. The common
  type should be the interface; concrete names should reveal platform behavior.

#### Volumes, roots, mount points, and network paths

Files:

- `src/CSharpFar.FileSystem/WindowsVolumeService.cs`
- `src/CSharpFar.FileSystem/VolumeInfoService.cs`
- `src/CSharpFar.FileSystem/VolumeMountPointService.cs`
- `src/CSharpFar.FileSystem/FileSystemLocationService.cs`
- `src/CSharpFar.App/Commands/DriveSelectionCommand.cs`
- `src/CSharpFar.App/Dialogs/DriveDialog.cs`
- `tests/CSharpFar.Tests/Spec007RootParentTests.cs`
- `tests/CSharpFar.Tests/Spec007MountPointTests.cs`

Current state:

- Volume service is named and documented as Windows-specific but currently wired
  for every run in `Program.cs`.
- `VolumeInfoService` has a Windows P/Invoke path for volume labels and a
  `DriveInfo` fallback.
- `VolumeMountPointService` returns no mount point metadata outside Windows.
- Tests include drive-root and UNC assumptions.

Platform difference:

- Windows navigation roots are drives and UNC shares.
- Unix/macOS roots start at `/`; mounted filesystems usually appear under `/`,
  `/mnt`, `/media`, `/Volumes`, or custom mount locations.
- Network location detection differs. UNC is Windows-specific; Unix network
  mounts are ordinary paths unless inspected through mount tables.

Resolution:

- Rename `WindowsVolumeService` to remain Windows-only and add
  `UnixVolumeService`.
- Model volume selection as "locations" rather than "drives" at the UI boundary:
  Windows locations are drives/UNC/module entries; Unix/macOS locations are `/`,
  home, mounted filesystems, and module entries.
- Implement Unix/macOS mount discovery behind `IVolumeService`, likely from
  .NET `DriveInfo.GetDrives()` first, with optional `/proc/mounts` on Linux and
  `mount`/system APIs only if needed.
- Keep `VolumeMountPointService` Windows-only unless a real Unix mount identity
  feature is required. A silent always-false Unix implementation should not be
  the long-term design if UI claims to display mount points.

#### Credential storage

Files:

- `src/CSharpFar.FileSystem/DpapiCredentialStore.cs`
- `src/CSharpFar.App/Program.cs`
- `src/CSharpFar.Module.Sftp/*ConnectionStore.cs`
- `src/CSharpFar.Module.Ftp/*ConnectionStore.cs`

Current state:

- `DpapiCredentialStore` uses `ProtectedData.Protect/Unprotect`.
- It throws `PlatformNotSupportedException` outside Windows.
- `Program.cs` wires it unconditionally.

Platform difference:

- Windows DPAPI is not available on Unix/macOS.
- macOS has Keychain.
- Linux has Secret Service/libsecret in desktop sessions, but headless systems
  may not have a credential service.

Resolution:

- Introduce platform credential stores:
  - `WindowsDpapiCredentialStore`;
  - `MacOsKeychainCredentialStore`;
  - `LinuxSecretServiceCredentialStore` or an explicitly configured
    file-based store if accepted by product requirements.
- Make unavailable secure storage a visible capability/error at connection-save
  time, not a late crash from a Windows class.
- Keep credential store selection in composition.

#### Clipboard

Files:

- `src/CSharpFar.Ui/TextCopyTextClipboard.cs`
- `src/CSharpFar.FarNetHost/CSharpFarFarNetApi.cs`
- `src/CSharpFar.App/Editor/FileEditor.cs`
- `src/CSharpFar.App/Viewer/LargeFileViewer.cs`

Current state:

- Main app clipboard uses `TextCopy.Clipboard`.
- FarNet compatibility clipboard uses direct Win32 P/Invoke to `user32.dll` and
  returns no data outside Windows.
- Editor normalizes copied line endings differently on Windows vs non-Windows.

Platform difference:

- Terminal clipboard may be GUI clipboard, OSC 52, `pbcopy`/`pbpaste`,
  `wl-copy`/`wl-paste`, `xclip`, `xsel`, or unavailable.
- macOS has `pbcopy`/`pbpaste`.
- Linux differs across Wayland/X11/headless sessions.

Resolution:

- Keep `ITextClipboard` as the native app boundary and make FarNet host consume
  the same service instead of owning Win32 clipboard P/Invoke.
- Add explicit platform clipboard implementations or a provider chain selected
  by platform and environment.
- If clipboard is unavailable, return a clear unsupported result through
  `ITextClipboard`; do not silently pretend copy/paste succeeded.

#### Settings and data directories

Files:

- `src/CSharpFar.App/Settings/JsonSettingsStore.cs`
- `src/CSharpFar.App/History/JsonHistoryStore.cs`
- `src/CSharpFar.App/Application.cs`
- `tests/CSharpFar.Tests/JsonSettingsStoreTests.cs`

Current state:

- Settings and history default to
  `Environment.SpecialFolder.ApplicationData`.
- Portable mode uses `CSharpFar.portable` and `CSharpFar.config`.

Platform difference:

- Windows roaming app data maps naturally to `%APPDATA%`.
- Linux should normally follow XDG base directory rules:
  `$XDG_CONFIG_HOME/csharpfar`, `$XDG_DATA_HOME/csharpfar`.
- macOS should normally use `~/Library/Application Support/CSharpFar`.

Resolution:

- Add an app data directory resolver with platform-specific implementations:
  `WindowsAppDataDirectoryResolver`, `LinuxXdgDirectoryResolver`,
  `MacOsApplicationSupportDirectoryResolver`.
- Keep portable mode behavior if explicitly retained, but name and test it as a
  cross-platform product decision rather than an incidental Windows config path.

#### File system semantics

Files:

- `src/CSharpFar.FileSystem/FileSystemService.cs`
- `src/CSharpFar.FileSystem/FileOperationService.cs`
- `src/CSharpFar.FileSystem/FileSystemSearchService.cs`
- `src/CSharpFar.FileSystem/FileSystemChangeWatcher.cs`
- `src/CSharpFar.App/DirectoryShortcuts/DirectoryShortcutNormalizer.cs`
- `src/CSharpFar.App/Commands/CreateFolderCommand.cs`
- `src/CSharpFar.App/Commands/OpenCreateFileCommand.cs`

Current state:

- Most local file work uses `System.IO` and `Path`.
- Some code explicitly handles both `Path.DirectorySeparatorChar` and
  `Path.AltDirectorySeparatorChar`.
- `FileSystemWatcher` is used for panel auto-refresh.

Platform difference:

- Unix/macOS are case-sensitive or case-preserving depending on filesystem.
- Unix hidden files use leading `.`, not hidden/system attributes.
- Unix permissions, executable bit, symlink ownership, hard links, extended
  attributes, and invalid file names differ from Windows.
- macOS often uses case-insensitive APFS by default and Unicode normalization can
  differ from Windows/Linux.
- `FileSystemWatcher` works through different backends and has different
  overflow/rename behavior.

Resolution:

- Introduce a small file-system platform policy where domain behavior really
  differs:
  - path comparison;
  - hidden/system classification;
  - executable detection;
  - permission/attribute preservation;
  - invalid name validation.
- Keep ordinary file operations in `FileSystemService` and
  `FileOperationService`; split only platform rules that change observable
  behavior.
- Add Unix/macOS tests using temporary files with executable bit, leading-dot
  hidden files, symlinks, and case-sensitive expectations where available.

#### FarNet compatibility

Files:

- `src/CSharpFar.FarNetCompat/*`
- `src/CSharpFar.FarNetHost/*`
- `tests/CSharpFar.Tests.Fixtures.*`
- `tests/CSharpFar.Tests/Spec039FarNetModuleToolLoaderTests.cs`
- `tests/CSharpFar.Tests/Spec040RealFarNetModuleDllTests.cs`

Current state:

- Compatibility is modeled as native CSharpFar extension domain, which matches
  local rules.
- Clipboard implementation inside `CSharpFarFarNetApi` is Windows P/Invoke.
- Some FarNet concepts originate from Windows/Far Manager semantics.

Platform difference:

- Existing third-party FarNet modules may assume Windows paths, Windows
  clipboard, Far Manager key codes, and Windows-only APIs.
- Loading managed assemblies is cross-platform in .NET, but module behavior may
  not be.

Resolution:

- Keep FarNet host as a boundary-facing compatibility layer.
- Mark unsupported platform-specific FarNet APIs explicitly with precise
  exceptions or capability checks.
- Route clipboard and filesystem services through native CSharpFar abstractions
  instead of duplicating platform code in FarNet host.

#### Tests

Files:

- `tests/CSharpFar.Tests/Spec007RootParentTests.cs`
- `tests/CSharpFar.Tests/Spec007MountPointTests.cs`
- `tests/CSharpFar.Tests/Spec009OpenPanelItemTests.cs`
- `tests/CSharpFar.Tests/JsonSettingsStoreTests.cs`
- `tests/CSharpFar.Tests/UnderlaySnapshotTests.cs`
- `tests/CSharpFar.Tests/Win32ConsoleInputLayoutTests.cs`

Current state:

- Some tests intentionally verify Win32 layout and Windows launch behavior.
- Some tests use Windows path literals where the behavior being tested is not
  necessarily Windows-specific.

Resolution:

- Split tests by responsibility:
  - portable domain tests should use platform-neutral temp paths and run
    everywhere;
  - Windows backend tests should be named and guarded as Windows-only;
  - Unix/macOS backend tests should verify terminal parsing, file launch,
    app-data paths, executable-bit detection, and volume/location rules.
- Keep Win32 layout tests, but place them near the Windows console backend or
  name them clearly as backend-specific.

`OperatingSystem.IsWindows()` checks.

Dependency direction should be:

- `Core` has no platform dependencies.
- `Ui` depends on console contracts/models, not concrete backends.
- `App` depends on contracts and composition.
- Platform projects depend inward on `Core`, `Console`, and possibly `Ui`
  contracts, never the other way around.
- `Program.cs` or a dedicated composition factory selects concrete services.

### First implementation slices

Recommended order:

1. Composition cleanup:
   - introduce `PlatformServices`;
   - move concrete service selection out of inline `Program.cs`;
   - keep Windows behavior unchanged.
2. Rename/move Windows-specific concrete services:
   - `WindowsShellFileLauncher` stays Windows-only;
   - `WindowsVolumeService`, `VolumeMountPointService`,
     `DpapiCredentialStore`, Win32 clipboard code and Win32 console code become
     clearly platform-owned.
3. Add Unix startup viability:
   - Unix shell profile;
   - Unix file launcher with executable-bit detection and configured opener;
   - app-data resolver;
   - credential store behavior that is explicit when unavailable.
4. Console Unix backend spike:
   - prove full-screen rendering, key input, resize, mouse, capture/restore or
     alternate-screen underlay behavior in a real terminal.
5. Add CI matrix:
   - Windows;
   - Linux;
   - macOS.

## Recommendation

Do not try to make the current `SystemConsoleDriver` and `Program.cs` portable by
adding more `if (OperatingSystem.IsWindows())` branches. That would hide platform
semantics in surprising places and make Unix/macOS behavior look like a fallback.

Recommended decision:

1. Keep `CSharpFar.Core`, panels, editor, viewer, FTP/SFTP modules and most UI
   rendering code portable.
2. Create explicit platform infrastructure projects or folders for console,
   shell/file launching, volumes, credentials, clipboard, and app-data paths.
3. Move platform selection to composition.
4. Treat Unix terminal support as its own console backend, not as a degraded
   `System.Console` path.
5. Add Linux/macOS CI only after platform-specific tests are separated enough
   that Windows-only tests do not mask portability progress.

The main unresolved technical risk is the Unix/macOS console backend. Shell,
file launch, app-data, credentials, clipboard, and volume discovery are
straightforward platform services. Terminal rendering/input needs a separate
experiment in a real terminal because resize, mouse, alternate screen, underlay,
and modifier keys cannot be validated well by current Windows-oriented tests.

## Follow-up

- Create an ADR for platform project structure and composition ownership.
- Create a spike for Unix/macOS terminal backend behavior.
- Create a spec for platform service extraction that preserves current Windows
  behavior.
- Create a spec for Unix/macOS shell/file launch behavior.
- Create a spec for app-data, credentials, and clipboard providers.
- Add CI jobs for Linux and macOS after portable tests are separated from
  Windows-only backend tests.

Verification during this spike was static review only. Production code was not
changed, and repository build/test checks were not run.
