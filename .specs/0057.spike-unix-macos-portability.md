## Question

What must change for CSharpFar to run predictably on Unix-like systems,
especially Linux and macOS, without hiding platform differences behind
surprising fallback behavior?

## Findings

The core domain model, panels, sorting, masks, search, viewer, editor, FTP, and
SFTP are broadly portable because they are already based on .NET abstractions or
separate services.

The main platform differences are at process boundaries:

- Console rendering and input.
- Shell command execution and platform file opening.
- Volume, root, and mount-point metadata.
- Credential storage.
- Clipboard integration.
- Native plugin composition.
- Application composition.
- Tests that assume Windows paths or Windows process launch semantics.

Windows uses Win32 console input records, mouse events, resize events, viewport
snapshots, screen-buffer APIs, console modes, palette mutation, and explicit
screen-buffer sizing.

Unix terminals normally use terminal escape sequences, termios input handling,
alternate screen behavior, mouse-reporting modes, bracketed paste, and
SIGWINCH-driven resize. macOS follows the Unix terminal model but differs in
option-key behavior, clipboard tools, default shell behavior, and terminal
capabilities.

The current Windows console behavior from `0016` remains the right Windows
model. A Unix implementation should not be a quiet fallback inside a class whose
name implies one generic console driver.

## Result / Recommendation

Use explicit platform backends selected in the composition root:

- A Windows backend keeps Win32 viewport-bound rendering.
- A Unix backend uses terminal escape sequences and termios/VT input parsing.
- The application owns a cross-platform cell-based virtual UI buffer. That
  buffer is the source of truth for UI state; Windows console screen buffers and
  Unix terminal state are physical output details only.
- Unix renders the virtual UI buffer through ANSI/VT SGR styles, cursor control,
  dirty horizontal runs, and alternate-screen mode. It must not try to emulate
  `ReadConsoleOutput` as a native terminal source of truth.
- Unix keyboard input is parsed from VT escape sequences, including arrows,
  navigation keys, F1-F12, CSI modifiers, Alt-key prefixes, and Ctrl-letter
  combinations.
- Unix resize may initially be detected by polling terminal size before input;
  resize produces an input/backend event that triggers logical surface resize
  and full repaint.
- `--check-terminal` is the manual diagnostic entry point for color/style,
  cursor, restore, and input parser checks on terminal backends.
- Unix/Linux command execution uses a platform shell command-line builder that
  passes the command through `ProcessStartInfo.ArgumentList`, not string
  formatting into `Arguments`.
- Platform file-operation details such as recycle-bin delete, ACL copy,
  symbolic-link recreation, and Unix file modes belong to platform services
  rather than the shared file-operation workflow.
- Unix/Linux association opening should try standard desktop tools and WSL
  friendly fallbacks before reporting that no association launcher is
  available.
- Unix/Linux volume and mount-point discovery should use
  `/proc/self/mountinfo` when available and filter technical pseudo mounts from
  the user-facing volume list.
- Windows and Unix hosts expose non-interactive smoke options: `--version`,
  `--self-test`, and `--check-terminal`.

Keep the application-facing console contract only if it can describe both
platform models predictably. Otherwise split it into smaller contracts such as
screen output, input, shell underlay, and clipboard mode.

Remove direct `System.Console` usage from application orchestration before
adding a Unix backend. Prompt, underlay, and current-console behavior should be
owned by the console backend or a console-session service.

Shell launching, file opening, volume discovery, credential storage, clipboard
access, and native plugin composition should each have explicit platform
ownership rather than scattered platform checks in domain code.

## Follow-up

- Continue expanding Unix terminal input beyond the minimum keyboard parser,
  including mouse reporting and richer modifier support.
- Define platform-specific credential and clipboard behavior.
- Define how native plugin composition locates platform-specific services.
