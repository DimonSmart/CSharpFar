# 0064 - Unix terminal input

Type: Spec
Status: Current
Related:
- 0004
- 0017

## Goal

Unix production input uses raw VT as its only supported backend. The
application continues to consume the shared `ConsoleInputEvent` model through
`IConsoleDriver`; Windows input behavior is unchanged.

## Production input behavior

The Unix console driver owns one internal input reader that:

- enters Unix raw terminal mode;
- reads keyboard and mouse input from the terminal byte stream;
- parses ordinary and UTF-8 text, Enter, Tab, shifted Tab, Backspace,
  standalone Escape, arrows, navigation keys, F1-F12, CSI modifiers,
  Alt-prefixed text, and Alt-prefixed VT keys;
- parses SGR mouse button press, release, wheel, and button-motion packets into
  zero-based viewport-relative `MouseConsoleInputEvent` coordinates;
- polls terminal size and emits `ConsoleResizeInputEvent` when it changes;
- on Linux, may additionally use evdev to detect standalone Shift press and
  release events that are not delivered through the terminal byte stream;
- ignores malformed mouse packets and handles unknown keyboard sequences
  without terminating the application.

The legacy `Console.ReadKey` Unix backend is not supported. The
`CSHARPFAR_UNIX_INPUT` environment variable does not select production input
behavior. Enhanced keyboard protocols are not required for production input.

## Linux Shift-only tracking

On Linux, production input may create an optional evdev modifier tracker that
observes only left and right Shift key state from readable `/dev/input/event*`
devices. When available, pressing Shift alone emits
`ModifierKeyConsoleInputEvent(ConsoleModifiers.Shift)` and releasing the final
held Shift emits `ModifierKeyConsoleInputEvent(default)`. These events update
the function-key bar layer without executing commands or inserting command-line
text.

The tracker must remain narrowly scoped:

- it activates only when `OperatingSystem.IsLinux()` is true;
- it reads only readable Shift-capable evdev devices;
- it ignores every event except left and right Shift key down/up;
- it does not store, log, or expose raw keyboard input;
- it tracks left and right Shift independently so releasing one held Shift does
  not clear the layer while the other Shift remains held;
- duplicate effective Shift states do not produce duplicate modifier events.

Evdev access is optional. Missing `/dev/input`, unreadable event devices,
permission denied, unsupported devices, and initialization failures must not
prevent startup or normal stdin/ANSI input. In those cases Shift-only tracking
is unavailable, while ordinary terminal input, modified function-key escape
sequences, resize, and mouse behavior continue to work.

The input reader suspends evdev modifier tracking while child-process console
mode is active and resumes it when application input mode is restored, matching
raw terminal and mouse tracking lifetime.

## Diagnostics

The terminal diagnostics command reports runtime and input environment details:

- OS description, OS architecture, process architecture, framework, and
  platform flags for Windows, Linux, and macOS;
- Unix effective user id, root status, sudo environment variables, and whether
  the process appears to be running via sudo;
- Windows elevated/admin status when available, or an explicit not-applicable
  or not-checked value;
- console driver, input backend, and mouse tracking status;
- modifier tracking backend, platform support, enabled status, whether
  Shift-only can currently be tracked, status, failure reason, and evdev device
  discovery summary.

Diagnostics must distinguish theoretical platform support from the tracker
being enabled and from Shift-only tracking being currently usable. When evdev
tracking is unavailable, diagnostics explain the reason without recommending
sudo as a normal requirement.

`ReadKey` skips mouse and resize events until it receives a key event so callers
that require a key keep their existing contract.

## Terminal mode lifetime

Production input enables xterm cell-motion tracking (`1002`) and SGR mouse
encoding (`1006`). It does not enable all-motion tracking (`1003`).

Disposal and error paths restore the original terminal input mode and disable
mouse modes `1000`, `1002`, `1003`, and `1006`. Entering child-process console
mode suspends raw input and mouse tracking; returning from child-process mode
restores raw input and `1002 + 1006` mouse tracking. Repeated suspend, restore,
and disposal operations are safe.

The combined input laboratory is the interactive Unix input diagnostic. It
does not define production behavior.

## Acceptance Criteria

- `AnsiTerminalConsoleDriver` delegates Unix production input to one raw VT
  input reader through the internal input-reader abstraction.
- Unix production input does not use `Console.ReadKey`, backend-selection
  environment variables, or a legacy fallback.
- Raw VT keyboard and SGR mouse input produce only standard
  `ConsoleInputEvent` values.
- On Linux with readable Shift-capable evdev devices, standalone Shift press
  and release produce `ModifierKeyConsoleInputEvent` values.
- When Linux evdev access is missing or denied, production input continues
  without Shift-only tracking and without startup or input-read exceptions.
- Evdev tracking ignores non-Shift events and does not log raw keyboard input.
- Mouse coordinates are zero-based and viewport-relative on Unix.
- Production mouse tracking uses `1002 + 1006`, never `1003`.
- Raw mode and mouse tracking are disabled for child processes and restored
  after the child-process scope ends.
- Disposal and input error paths restore terminal and mouse modes.
- Child-process console mode suspends optional evdev modifier tracking and
  restores it afterwards.
- Unknown keyboard sequences and malformed mouse packets do not crash the UI
  or write diagnostic text into it.
- Resize is represented by `ConsoleResizeInputEvent`, not a synthetic key.
- `--check-terminal` and `--check-terminal --input-lab` remain available; old
  overlapping input experiment commands are not public diagnostics.
- Terminal diagnostics include OS/runtime, privilege/sudo/elevation,
  input-backend, mouse-tracking, and modifier-tracking status.
- Parser tests retain coverage for text, UTF-8, standalone Escape, navigation,
  F1-F12, modified keys, Alt-prefixed input, mouse buttons, release, wheel,
  motion, modifiers, and coordinate conversion.
- Input-reader lifecycle tests cover raw mode, mouse tracking, suspension,
  restoration, optional modifier-tracker suspension, repeated lifecycle calls,
  and idempotent disposal.

## Verification

Automated verification:

```bash
dotnet build
dotnet test
```

Manual verification in Windows Terminal plus WSL runs the Unix host without
`CSHARPFAR_UNIX_INPUT` and checks keyboard navigation, Enter, Escape, arrows,
F-keys, mouse buttons, wheel, drag, resize, child-process entry/return, and clean
terminal restoration. On Linux with readable evdev devices, hold and release
Shift alone and verify the bottom function-key bar changes to and from the
Shift layer. On Linux without evdev access, verify the diagnostics report why
Shift-only tracking is unavailable and the app otherwise works normally. Run
`--check-terminal` and `--check-terminal --input-lab` for focused terminal
diagnostics.
