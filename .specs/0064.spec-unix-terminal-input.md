# 0064 - Unix terminal input

Type: Spec
Status: Current
Related:

## Goal

Unix production input uses raw VT as its only supported backend. The
application continues to consume the shared `ConsoleInputEvent` model through
`IConsoleDriver`; Windows input behavior is unchanged.

## Production input behavior

The Unix console driver owns one internal input reader that:

- enters Unix raw terminal mode;
- reads keyboard and mouse input from the terminal byte stream;
- parses ordinary and UTF-8 text, Enter, Tab, shifted Tab, Backspace,
  standalone Escape, arrows, navigation keys, F1-F12, CSI modifiers,
  Alt-prefixed text, and Alt-prefixed VT keys;
- parses SGR mouse button press, release, wheel, and button-motion packets into
  zero-based viewport-relative `MouseConsoleInputEvent` coordinates;
- polls terminal size and emits `ConsoleResizeInputEvent` when it changes;
- ignores malformed mouse packets and handles unknown keyboard sequences
  without terminating the application.

The legacy `Console.ReadKey` Unix backend is not supported. The
`CSHARPFAR_UNIX_INPUT` environment variable does not select production input
behavior. Enhanced keyboard protocols and modifier-only events are not required
for production input.

`ReadKey` skips mouse and resize events until it receives a key event so callers
that require a key keep their existing contract.

## Terminal mode lifetime

Production input enables xterm cell-motion tracking (`1002`) and SGR mouse
encoding (`1006`). It does not enable all-motion tracking (`1003`).

Disposal and error paths restore the original terminal input mode and disable
mouse modes `1000`, `1002`, `1003`, and `1006`. Entering child-process console
mode suspends raw input and mouse tracking; returning from child-process mode
restores raw input and `1002 + 1006` mouse tracking. Repeated suspend, restore,
and disposal operations are safe.

The combined input laboratory is the interactive Unix input diagnostic. It
does not define production behavior.

## Acceptance Criteria

- `AnsiTerminalConsoleDriver` delegates Unix production input to one raw VT
  input reader through the internal input-reader abstraction.
- Unix production input does not use `Console.ReadKey`, backend-selection
  environment variables, or a legacy fallback.
- Raw VT keyboard and SGR mouse input produce only standard
  `ConsoleInputEvent` values.
- Mouse coordinates are zero-based and viewport-relative on Unix.
- Production mouse tracking uses `1002 + 1006`, never `1003`.
- Raw mode and mouse tracking are disabled for child processes and restored
  after the child-process scope ends.
- Disposal and input error paths restore terminal and mouse modes.
- Unknown keyboard sequences and malformed mouse packets do not crash the UI
  or write diagnostic text into it.
- Resize is represented by `ConsoleResizeInputEvent`, not a synthetic key.
- `--check-terminal` and `--check-terminal --input-lab` remain available; old
  overlapping input experiment commands are not public diagnostics.
- Parser tests retain coverage for text, UTF-8, standalone Escape, navigation,
  F1-F12, modified keys, Alt-prefixed input, mouse buttons, release, wheel,
  motion, modifiers, and coordinate conversion.
- Input-reader lifecycle tests cover raw mode, mouse tracking, suspension,
  restoration, repeated lifecycle calls, and idempotent disposal.

## Verification

Automated verification:

```bash
dotnet build
dotnet test
```

Manual verification in Windows Terminal plus WSL runs the Unix host without
`CSHARPFAR_UNIX_INPUT` and checks keyboard navigation, Enter, Escape, arrows,
F-keys, mouse buttons, wheel, drag, resize, child-process entry/return, and clean
terminal restoration. Run `--check-terminal` and `--check-terminal --input-lab`
for focused terminal diagnostics.
