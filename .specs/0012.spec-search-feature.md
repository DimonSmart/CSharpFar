# 0012 - Search feature

Type: Spec
Status: Current
Replaces:
- none

Related:

## Goal

Add file, directory, and text search to CSharpFar.

Search starts from the active file panel, collects a normalized request, runs as a cancellable operation, and opens results as a read-only panel-like view instead of only showing a temporary popup list.

The search engine must return data. It must not render panels directly.

## Context

Current code already has a minimal `Alt+F7` path:

1. `Application.HandleSearchFiles()` asks for one mask with `InputDialog`.
2. `CSharpFar.App.Search.FileSearcher` recursively collects matching file paths.
3. `SearchProgressDialog` shows only the found count.
4. `SearchResultsDialog` shows a scrollable popup.
5. Choosing a result loads the real parent directory into the active panel and places the cursor on the file.

Current panels and operations are also filesystem-path based:

1. `FilePanelState.CurrentDirectory` is a real directory path.
2. `FilePanelItem.FullPath` is used by open, view, edit, copy, move, delete, selection, and history paths.
3. `PanelController` builds views through `PanelViewBuilder`, which reads `IFileSystemService`.
4. `StartWatching`, `SafeRefresh`, and pending refresh handling assume the panel location is a real directory.
5. `IFileOperationService` takes source and destination paths, not provider items or streams.

The project already has Far-like mask support in `CSharpFar.Core.FileMasks.FarMaskMatcher`. Search must reuse or extend that matcher instead of creating a second mask language.

## Scope

Implement the feature in three stages:

1. search dialog and `SearchRequest`;
2. search engine and progress;
3. read-only search results panel.

Each stage should leave the product in a coherent state. Do not keep compatibility wrappers for the old search path unless a later task explicitly asks for them.

### Stage 1. Search dialog and request

Replace the current one-line `Alt+F7` prompt with a modal search dialog built from the existing dialog/rendering style.

The command remains `Alt+F7`. Plain `F7` must keep creating folders.

Initial UI:

```text
Find file

A file mask or several file masks:
*

Containing text:

Using code page:
Automatic detection

[ ] Case sensitive
[ ] Whole words
[ ] Not containing
[ ] Search folders
[ ] Search in symbolic links

Select search area:
From the current folder

Parallelism:
4

{ Find } [ Cancel ]
```

Dialog rules:

1. `{ Find }` validates fields and returns a `SearchRequest`.
2. `[ Cancel ]` and `Esc` close the dialog without starting search.
3. Stage 1 may pass the request to a stub/log path. It must not perform the full search before Stage 2.
4. Use existing console dialog helpers, reusable console UI components, and
   palette conventions. Do not introduce an external or general-purpose UI
   framework.

Search area:

```csharp
public enum SearchScope
{
    CurrentDirectoryOnly,
    CurrentDirectoryRecursive,
}
```

Rules:

1. `CurrentDirectoryOnly` searches only the active panel directory.
2. `CurrentDirectoryRecursive` searches the active panel directory and descendants.
3. UI labels may be `In current folder` and `From the current folder`.

Mask field:

1. Default value is `*`.
2. Empty or whitespace-only input normalizes to `*`.
3. The value is a Far mask expression, not a new custom mask list.
4. The expression must be passed through the existing `FarMaskMatcher` semantics: `*`, `?`, ranges, comma/semicolon lists, `|` excludes, regex tokens, groups, `%PATHEXT%`, and `*.*` normalization.
5. Whitespace around the whole field is trimmed.
6. Do not split the request into `IReadOnlyList<string>` in a way that loses existing Far mask expression semantics.

Preferred request model:

```csharp
public sealed record SearchRequest
{
    public required string RootPath { get; init; }
    public required string FileMaskExpression { get; init; }
    public string? ContainingText { get; init; }
    public bool CaseSensitive { get; init; }
    public bool WholeWords { get; init; }
    public bool NotContaining { get; init; }
    public bool IncludeDirectoriesInResults { get; init; }
    public bool SearchInSymbolicLinks { get; init; }
    public SearchScope Scope { get; init; }
    public int MaxDegreeOfParallelism { get; init; }
    public SearchEncodingMode EncodingMode { get; init; } = SearchEncodingMode.Automatic;
    public long MaxContentSearchFileSizeBytes { get; init; } = 100L * 1024 * 1024;
}

public enum SearchEncodingMode
{
    Automatic,
}
```

Naming rules:

1. Use `FileMaskExpression`, not `FileMasks`, because the existing matcher accepts one expression.
2. Use `IncludeDirectoriesInResults`, not `SearchFolders` or `Recursive`.
3. Use `SearchInSymbolicLinks` only for link traversal.
4. Use `SearchScope` for recursive vs non-recursive search.
5. Use `MaxDegreeOfParallelism` for the parallelism setting.

Containing text:

1. Empty text means search by file and optional directory names only.
2. Non-empty text means files must match the name mask and then pass content search.
3. Directories are never searched by content.
4. Directories may appear only when `IncludeDirectoriesInResults` is true and their name matches the mask.

Case sensitivity:

1. When disabled, wildcard mask matching, directory/file name matching, and content search are case-insensitive on Windows.
2. When enabled, wildcard mask matching, directory/file name matching, and content search are case-sensitive.
3. Implement this by extending or optioning the existing mask matcher. Do not create a second mask implementation.
4. Regex mask tokens should have explicit tests before implementation because they already have flags.

Whole words:

1. Applies only when `ContainingText` is non-empty.
2. Word characters are Unicode letters, Unicode digits, and `_`.
3. `Customer` must not match inside `CustomerId` when enabled.

Not containing:

1. Applies only when `ContainingText` is non-empty.
2. If text is empty, the checkbox is disabled or ignored during request construction.
3. A file matches `NotContaining` only when its name matches the mask and searchable text content was checked and did not contain the text.

Encoding:

1. First version supports only automatic detection.
2. Detection order:
   - UTF-8 BOM;
   - UTF-16 LE BOM;
   - UTF-16 BE BOM;
   - UTF-8 without BOM when the byte stream is valid UTF-8;
   - Windows ANSI code page.
3. The ANSI fallback must use the Windows ANSI code page explicitly. Do not assume `Encoding.Default` is ANSI on modern .NET.
4. If code page support requires `System.Text.Encoding.CodePages`, add it deliberately in the project that owns content reading.

Parallelism:

1. Valid range is `1..16`.
2. `1` means sequential file content processing.
3. Default is `Math.Min(Environment.ProcessorCount, 4)`.
4. Invalid input shows validation error and does not start search.

### Stage 2. Search engine and progress

Add a search service that accepts `SearchRequest`, reports progress, supports cancellation, and streams results.

Place request/result/progress contracts in `CSharpFar.Core`. Put the implementation that touches `System.IO` in the filesystem/app layer that best matches the final dependency direction. The current static `CSharpFar.App.Search.FileSearcher` is a temporary implementation and should be replaced or retired by this stage.

Preferred service contract:

```csharp
public interface ISearchService
{
    IAsyncEnumerable<SearchResultItem> SearchAsync(
        SearchRequest request,
        IProgress<SearchProgress>? progress,
        CancellationToken cancellationToken = default);
}
```

Preferred result model:

```csharp
public sealed record SearchResultItem
{
    public required string FullPath { get; init; }
    public required string Name { get; init; }
    public required SearchResultItemKind Kind { get; init; }
    public long? Size { get; init; }
    public DateTime LastWriteTime { get; init; }
    public FileAttributes Attributes { get; init; }
    public string? MatchedTextPreview { get; init; }
    public int? LineNumber { get; init; }
}

public enum SearchResultItemKind
{
    File,
    Directory,
}
```

Do not emit filesystem errors as fake search result rows. Report diagnostics through progress, logging, or a separate diagnostic model.

Preferred progress model:

```csharp
public sealed record SearchProgress
{
    public long ScannedDirectories { get; init; }
    public long ScannedFiles { get; init; }
    public long MatchedItems { get; init; }
    public long ErrorCount { get; init; }
    public string? CurrentPath { get; init; }
    public string? LastErrorPath { get; init; }
    public string? LastErrorMessage { get; init; }
}
```

Traversal:

1. Validate `RootPath` before starting.
2. Enumerate files and directories under `RootPath`.
3. Apply `FileMaskExpression` to item names, not full paths.
4. Include directories as results only when `IncludeDirectoriesInResults` is true.
5. Always traverse directories needed by `SearchScope`, even if directory names do not match the mask.
6. If `SearchScope` is `CurrentDirectoryOnly`, do not enqueue child directories.
7. Access denied, path too long, disappeared files, and invalid reparse points must not fail the whole search.

Symbolic links and reparse points:

1. `SearchInSymbolicLinks` defaults to false.
2. When false, do not follow symlink or junction directories during recursive traversal.
3. When true, follow them with cycle protection.
4. Cycle protection may start with normalized full paths of resolved directories.
5. Windows file identity is allowed later if needed, but it is not required for the MVP if normalized paths already prevent ordinary cycles.

Content search:

1. If `ContainingText` is empty, a matching file name is enough.
2. If `ContainingText` is set, a file must match the mask before content is read.
3. `NotContaining == false` means the content must contain the text.
4. `NotContaining == true` means searchable text content must be checked and must not contain the text.
5. Binary files, too-large files, and decode failures do not match either containing or not-containing searches. They may add diagnostics.
6. `MatchedTextPreview` and `LineNumber` are populated only for positive content matches when practical.

Large and binary files:

1. Default `MaxContentSearchFileSizeBytes` is 100 MB.
2. Files larger than the limit are not read for content search.
3. A simple binary heuristic is enough for the first version: sample the beginning of the file and treat it as binary when it contains NUL bytes or a high NUL density.
4. Hex search is out of scope.

Parallelism:

1. Bound all file content work by `MaxDegreeOfParallelism`.
2. Do not create an unbounded task per file.
3. A practical design is one directory producer plus bounded file workers.
4. Use a bounded `Channel` or equivalent backpressure so very large trees do not accumulate unbounded memory.
5. Directory traversal may stay sequential for the first implementation.

Cancellation:

1. Pass `CancellationToken` through every async boundary.
2. Check cancellation during traversal, before content reads, and between read chunks for large files.
3. `Go to` during an active search may cancel the running search and use the selected result to navigate.
4. Explicit stop/cancel of the active search discards already collected results and returns to the previous panel.

Progress UI:

Use a modal progress dialog or status overlay similar to existing operation progress:

```text
Searching
C:\Project\src

Files:   1520
Folders: 84
Found:   17
Errors:  0

[ Cancel ]
```

Rules:

1. Do not show a percentage because total work is unknown.
2. `Esc` or `[ Stop ]` asks for confirmation using the standard red cancellation dialog.
3. Confirmed stop cancels the search and returns to the previous panel without opening partial results.
4. Keep progress rendering on the UI side. The search service only reports data.

### Stage 3. Search results panel

Open search results as a panel-like read-only source after search completes or is cancelled with partial results.

The current code is path-based, so this stage must explicitly make panel operations source-aware before search results are exposed as a panel. Do not let existing `F4`, `F8`, `F6`, `SafeRefresh`, `StartWatching`, or copy destination logic accidentally treat the search results panel as a normal writable directory.

Preferred capability model:

```csharp
[Flags]
public enum PanelProviderCapabilities
{
    None = 0,
    Enumerate = 1 << 0,
    OpenRead = 1 << 1,
    OpenWrite = 1 << 2,
    CreateFile = 1 << 3,
    CreateDirectory = 1 << 4,
    Delete = 1 << 5,
    Rename = 1 << 6,
    CopyFrom = 1 << 7,
    CopyTo = 1 << 8,
    MoveFrom = 1 << 9,
    MoveTo = 1 << 10,
    Edit = 1 << 11,
    Refresh = 1 << 12,
    Watch = 1 << 13,
}
```

Use `PanelProviderCapabilities`, not `FileSystemCapabilities`, because the flags describe what the panel may do with a source. A source may be the local filesystem, search results, and later something else.

The exact provider interface may follow the current synchronous panel architecture instead of forcing async panel loading. For the MVP, search results are already in memory, and local panel loading is currently synchronous.

Provider requirements:

1. Local filesystem provider keeps current behavior and has normal write/copy/move/delete/create/refresh/watch capabilities.
2. Search results provider has read-only capabilities:
   - `Enumerate`;
   - `OpenRead`;
   - `CopyFrom`;
   - `Refresh`.
3. Search results provider must not expose:
   - `CopyTo`;
   - `MoveTo`;
   - `CreateFile`;
   - `CreateDirectory`;
   - `Delete`;
   - `Rename`;
   - `OpenWrite`;
   - `Edit`.

MVP compatibility rule:

1. The existing `FileOperationService` may remain path-based in this spec.
2. Copying from search results to a local filesystem panel may map selected result items to their real `FullPath` values and call the existing operation service after capability checks.
3. General provider-to-provider copy/move by stream is a future extension, not part of this spec.
4. Do not implement archives, FTP/SFTP, Git views, or plugin providers in this spec.

Panel representation:

1. Reuse existing `FilePanelItem` rendering where practical.
2. Search result items must keep the real filesystem path needed to open and copy the original item.
3. If new provider metadata is added, do not change the meaning of `FilePanelItem.FullPath` to a non-filesystem URI without first updating every operation path that consumes it.
4. The search results panel is a flat list in the first version. Do not add virtual folder grouping.
5. Do not show a `..` item in the flat search results panel.
6. Show duplicate file names safely; selection identity must be based on full real path.
7. The UI must expose the full real path of the current result, at least in a status row or a search-specific path row. A full path column is optional for the MVP.

Title examples:

```text
Search results: *.cs;*.json
Search results: IDisposable
Search results: *.cs, cancelled
```

Operations in search results:

1. `Enter` on a file loads the real parent directory in the current panel and selects the file.
2. `F3` view is the explicit file-view action and is allowed if the real file exists.
3. `F4` edit is disabled in search results for the MVP.
4. `Enter` on a directory opens the real directory in the current panel through the local filesystem panel.
5. `F5` copy is allowed only when the active source supports `CopyFrom` and the target panel supports `CopyTo`.
6. `F6` move is disabled from search results.
7. `F7` create folder is disabled in search results.
8. `F8` delete is disabled in search results.
9. Copying into search results is disabled with a clear message:

```text
Cannot copy to search results panel.
Search results are read-only.
```

Refresh:

1. Refresh on a search results panel re-runs the original `SearchRequest` and replaces the result list.
2. If refresh is cancelled, keep the previous result list unless the user explicitly accepts partial refreshed results.
3. Files that disappeared since the search was run should be skipped or shown with a clear open/copy error when acted on.

Watchers and history:

1. Do not start `IFileSystemChangeWatcher` for search results panels.
2. Directory history should record real directories opened from search results, not the virtual search results title.
3. File history should record real file paths viewed or opened from search results.

## Non-goals

Do not implement in this spec:

1. a new mask syntax separate from `FarMaskMatcher`;
2. manual code page selection beyond automatic detection;
3. hex/binary content search;
4. approximate/fuzzy text search;
5. search inside archives;
6. search in alternate NTFS streams;
7. live updating search results while the filesystem changes;
8. editing files directly from search results;
9. deleting originals from search results;
10. removing items from results without deleting originals;
11. moving from or into search results;
12. copying into search results;
13. provider-to-provider stream copy for arbitrary virtual providers;
14. archive, FTP/SFTP, Git status, favorites, or plugin providers;
15. compatibility wrappers for the old `FileSearcher.Search` API unless another task requests them.

## Acceptance criteria

Stage 1:

1. `Alt+F7` opens the search dialog.
2. `F7` still creates folders.
3. Empty mask normalizes to `*`.
4. Mask input is treated as a Far mask expression.
5. Invalid parallelism blocks `{ Find }` with validation feedback.
6. `Not containing` is disabled or ignored when containing text is empty.
7. `{ Find }` creates a `SearchRequest` with the active panel directory as `RootPath`.
8. `[ Cancel ]` does not create or execute a request.

Stage 2:

1. Search supports recursive and non-recursive scopes.
2. Search applies masks to item names, not full paths.
3. Search can find files by name only.
4. Search can find files by text content.
5. `Case sensitive`, `Whole words`, and `Not containing` work for content search.
6. Directories appear only when `IncludeDirectoriesInResults` is true and their names match the mask.
7. Symbolic-link traversal follows `SearchInSymbolicLinks` and avoids cycles.
8. Large and binary files are skipped for content search with diagnostics.
9. Access errors do not fail the whole search.
10. Results stream before the whole search is complete.
11. Progress reports scanned files, scanned directories, matched items, current path, and errors.
12. `Go to` during search cancels the running search and navigates to the selected result.
13. Parallel file content work is bounded by `MaxDegreeOfParallelism`.

Stage 3:

1. Search results open in the active panel after completion.
2. Confirmed stop during the initial search returns to the previous panel without opening partial results.
3. Results panel is read-only.
4. Results panel is a flat list and has no `..` item.
5. Current result's real path is visible somewhere in the panel UI.
6. `Enter` on a result file loads its parent directory as a local filesystem panel and selects the file.
7. `F3` views the real file.
8. `F4` edit is disabled in search results.
9. `Enter` on a result directory opens that real directory as a local filesystem panel.
10. Copying from search results to a local filesystem panel works.
11. Copying into search results is blocked with a clear message.
12. Move, delete, rename, and create-folder are disabled in search results.
13. Auto-refresh watcher is not started for search results.
14. Existing local filesystem panels still render, sort, select, copy, move, delete, create, refresh, and watch as before.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual checks:

1. Open `Alt+F7`, validate each field, cancel, and confirm no search starts.
2. Search `*.cs` recursively from a test folder and verify result count.
3. Search current directory only and verify child folders are not searched.
4. Search text with case-sensitive on/off.
5. Search text with whole words against `Customer` and `CustomerId`.
6. Search with `Not containing`.
7. Search with access-denied or disappearing files and verify the search continues.
8. Stop a long recursive search, confirm cancellation, and verify the previous panel remains open without partial results.
9. Copy a file from search results to a normal folder.
10. Try copy/delete/edit/create/move in search results and verify disabled behavior.
