## Goal

All application-owned UI is composed from one shared active full-screen surface
and zero or more modal overlays. The visible UI is always rebuilt from current
application state at the current viewport; no application UI restores a saved
screen snapshot.

## Composition contract

- The application creates exactly one `UiCompositionHost` and its associated
  `ModalDialogHost` at rendering bootstrap. Application UI receives those
  shared services through normal dependencies and never creates a local host.
- `UiCompositionHost` is the sole owner of frame creation, stable-frame retry,
  viewport change detection, and resize-event consumption. The root runtime
  and modal sessions use its same input contract.
- The active `IUiSurface` renders the complete underlay. Overlays then render
  their own current state in nesting order. Closing an overlay or surface
  schedules a complete redraw of the remaining composition.
- An overlay only supplies `Render(UiRenderContext)`. It neither opens a frame,
  captures/restores pixels, renders an underlay, nor reads input from
  `ScreenRenderer`.
- Editor, viewer, and full-screen help are temporary surfaces. Their dialogs
  are overlays on the same shared modal host.

## Input, viewport, and layout

- Composition input methods consume resize input and return only semantic
  non-resize input. Empty polling detects a changed viewport and redraws only
  when recovery is required.
- A render is stable only after its complete frame flushes without interruption
  and its viewport still matches the frame viewport. `LastStableViewport` is
  updated only then.
- Each interactive layer derives layout from `UiRenderContext.Size` or
  `UiRenderContext.Viewport` during rendering. Mouse hit testing uses only the
  layout from the most recent stable rendered frame; input is ignored until
  such layout exists.

## Application-owned UI

- Ordinary dialogs, including settings, search, copy/move/rename options,
  create-folder, attributes, drive, and directory-shortcut dialogs, are modal
  overlays and preserve their own state across viewport changes.
- Search and file-operation progress are long-lived overlays. Cancel,
  conflict, rename, and other confirmation dialogs are nested overlays above
  the relevant progress overlay.
- FTP and SFTP connection manager and editor dialogs use
  `ModuleUiServices.ModalDialogs`, including nesting. Built-in modules do not
  use `SaveScreen` or `RestoreScreen` for their UI.

## Snapshot boundary

`ScreenRenderer.Capture` and `Restore` are permitted only for renderer and
console-driver internals, shell underlay and terminal-screen interoperability,
external-console/child-process support, renderer tests, and retained public
compatibility APIs. They are prohibited in application-owned dialogs, popups,
progress UI, editor, viewer, help, commands, and built-in module UI.

## Acceptance criteria

- Grow and shrink while any visible surface or overlay is active redraw all
  visible layers at the final viewport with no stale cells.
- Closing nested overlays redraws their parent and all lower layers from state.
- Keyboard, modifier, and mouse input after one or more resize events is not
  lost; stable empty polling does not continuously redraw.
- Dialog, progress, editor, viewer, help, command, FTP, and SFTP production
  code has no private frame, snapshot lifecycle, raw `ScreenRenderer` input,
  raw console key polling, underlay callback, or local composition host.
- Source-contract tests scan production source with a small documented
  allowlist for the snapshot boundary and bootstrap host creation.
- Integration tests inspect final fake-console cells for grow and shrink of
  modal, hidden-panel, nested-progress, and temporary-surface compositions.

## Verification

Run:

```bash
dotnet build
dotnet test
```

Run source checks for snapshot, frame, raw-input, underlay callback, raw-console
input, and composition-host construction usages. Every remaining occurrence
must be inside the documented boundary.
