## Goal

Make copy dialog options selectable by mouse through the shared form-dialog control
model, and make `Paranoid` copy automatically retry read-side failures without
showing a modal wait prompt.

## Context

`Paranoid` copy already validates partial destination files and can roll back to a
safe matching offset before resuming. The current failure path still treats read
errors like ordinary copy errors and can force a user decision before retrying.

Mouse support has been added control by control. New controls should not need an
extra per-control opt-in to basic mouse handling. The controls registry should state
that mouse support is part of the expected contract for reusable controls.

## Scope

1. Add mouse selection for copy dialog option rows, including copy conflict options.
2. Make mouse support mandatory for registered reusable controls unless a specific
   control documents why it cannot receive mouse input.
3. Record the mouse-support rule in the controls registry.
4. In `Paranoid` copy, treat source-open and source-read failures as automatic retry
   cases.
5. Wait before each automatic read-side retry. Use a fixed one-minute production
   delay while keeping tests deterministic.
6. After a read-side retry, reopen source and destination, reread actual destination
   state, rerun tail validation, and continue only from the analyzer result.
7. Avoid triggering rollback search repeatedly for transient source errors. Rollback
   must be driven by verified content mismatch, not by read/open failures.
8. Keep write errors interactive: show the existing operation error and wait for the
   user decision.

## Non-goals

1. Retry settings UI.
2. Automatic retry for write failures.
3. Automatic retry for non-paranoid copy modes.
4. Provider-to-provider paranoid resume.
5. Changing validation window sizes or rollback limits.
6. Broad form-dialog redesign.

## Acceptance criteria

- Copy dialog conflict options can be selected with the mouse.
- The controls registry says reusable controls must support mouse interaction by
  default.
- `Paranoid` copy does not show a modal retry dialog for source-open or source-read
  failures.
- `Paranoid` copy waits before retrying a read-side failure.
- Repeated read-side failures do not repeatedly roll back destination content.
- After the source becomes readable again, `Paranoid` copy reruns tail validation and
  resumes from the safe offset.
- Write failures still use the existing interactive error path.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused tests:

1. Copy dialog mouse-selects `Paranoid`.
2. Paranoid copy automatically retries a transient source-read failure.
3. Paranoid copy does not truncate repeatedly across repeated source-read failures.
4. Paranoid copy write failures still ask the conflict resolver.
