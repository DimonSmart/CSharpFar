# 0065 - Debian/Ubuntu APT distribution and installation

## Intent

CSharpFar publishes Debian packages through a signed static APT repository on
GitHub Pages and provides a simple Debian/Ubuntu installation path through that
repository while keeping the transparent manual APT setup documented for users
who want to inspect or perform each repository step themselves.

## Related Specifications

- 0057

## Behavior

The public GitHub Pages site exposes both the APT repository and a root-level
POSIX `sh` installer:

```text
https://dimonsmart.github.io/CSharpFar/install.sh
https://dimonsmart.github.io/CSharpFar/apt/
```

The installer supports Debian/Ubuntu-based systems with `apt-get`. It installs
the CSharpFar repository keyring into `/etc/apt/keyrings/csharpfar.gpg`, writes
the repository source list to `/etc/apt/sources.list.d/csharpfar.list`, runs
`apt-get update`, installs or upgrades the `csharpfar` package, and prints
`csharpfar --version`.

The installer works when run as root or as a normal user with `sudo`. If root
privileges cannot be obtained, a required command is missing, or the current
package architecture is unsupported, it exits with a clear error.

CSharpFar APT packages are currently published for `amd64` only. The installer
uses `dpkg --print-architecture` and rejects unsupported architectures before
modifying APT configuration.

Only `amd64` is published because the release matrix currently produces only
`linux-x64`. Another Debian architecture must be added only with its matching
Unix publish runtime and release asset.

The installer is idempotent. Re-running it refreshes the keyring and source
list, updates the package index, and installs or upgrades `csharpfar`.

The installer must not run `csharpfar --self-test` automatically. Self-test is a
separate diagnostic command documented for manual use.

## Architecture And Patterns

The installer is POSIX-compatible and can be executed as:

```bash
curl -fsSL https://dimonsmart.github.io/CSharpFar/install.sh | sh
```

It does not use `apt-key`. It uses an APT source entry with `signed-by` pointing
to `/etc/apt/keyrings/csharpfar.gpg`.

The GitHub Pages publication artifact contains the root installer alongside the
`apt/` repository tree.

GitHub Releases remain the source of package history. The Pages workflow
rebuilds the APT repository from every non-draft `csharpfar_*_amd64.deb`
release asset so rerunning the workflow does not discard older package
versions.

The Debian package is built from the verified self-contained `linux-x64`
publish output. This keeps the portable archive and Debian package on one Unix
build model and installs the existing `csharpfar` executable as
`/usr/bin/csharpfar`.

The repository uses suite `stable` and component `main`. CSharpFar currently has
one release channel and one directly maintained component, so additional suites
or components are not part of the current distribution model.

The dedicated repository private key and optional passphrase are stored as
GitHub Actions secrets `APT_GPG_PRIVATE_KEY` and `APT_GPG_PASSPHRASE`. Private
key material is not stored in git, GitHub Releases, or the Pages artifact. The
workflow exports only the public key into the repository.

If the automatic Pages rebuild after `Release` fails or Pages must be rebuilt,
the `APT Pages` workflow can be run manually with `workflow_dispatch`. It
redownloads all matching Debian release assets, regenerates and signs the
complete repository, and deploys one Pages artifact containing the site root and
`apt/` subtree.

GitHub Pages must use GitHub Actions as its source. If documentation is later
published on the same Pages site, its generated files must be merged into the
same site artifact before deployment.

## Non-Goals

- The installer does not support non-APT distributions.
- The installer does not configure third-party repositories.
- The installer does not modify APT configuration outside the CSharpFar keyring
  and source-list files.
- The installer does not run interactive diagnostics automatically.
- The APT repository does not define multiple release channels or components
  until the release process provides distinct product meanings for them.

## Acceptance Criteria

- A user can install CSharpFar with one command:
  `curl -fsSL https://dimonsmart.github.io/CSharpFar/install.sh | sh`.
- After installation, `csharpfar --version` is available.
- The installer does not use `apt-key`.
- The keyring is installed at `/etc/apt/keyrings/csharpfar.gpg`.
- The source list is installed at `/etc/apt/sources.list.d/csharpfar.list`.
- The source list uses `signed-by=/etc/apt/keyrings/csharpfar.gpg`.
- The installer derives architecture from `dpkg --print-architecture`.
- Re-running the installer does not break an existing installation.
- Unsupported architectures and unsuitable systems fail with clear messages.
- README documents the one-command installer, version verification, separate
  self-test diagnostics, and the manual APT fallback.
- Rerunning the APT Pages workflow preserves older non-draft Debian release
  assets in the regenerated repository.
- Repository private key material is available only through GitHub Actions
  secrets and is not published.

## Verification

- Validate `docs/install.sh` with POSIX shell syntax checking.
- Verify the GitHub Pages artifact contains `install.sh`,
  `apt/csharpfar-archive-keyring.gpg`, `apt/dists/`, and `apt/pool/`.
- Smoke-test installation in a clean Debian or Ubuntu environment by installing
  prerequisites, running the one-command installer, and checking
  `csharpfar --version`.
