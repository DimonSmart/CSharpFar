Type: Spec
Status: Current
Replaces:

## Goal

Add syntax highlighting to the built-in CSharpFar editor using TextMate grammars.

The first implementation should color visible editor text without changing editor
text, cursor movement, selection, undo/redo, search, save, bookmarks, or Far-like
editing commands from `0046` and `0049`.

The implementation must introduce a small editor-domain syntax highlighting
surface first, then back it with TextMateSharp. The editor core must not depend
directly on TextMateSharp types outside the TextMate-specific implementation.
The TextMate-specific implementation is an internal library boundary, not a
public plugin API, FarNet compatibility layer, or separate editor model.

## Context

`0046` delivered the built-in editor model, rendering flow, editor-domain events,
selection state, undo/redo, find/replace, bookmarks, status rendering, scrollbar,
and left syntax color spans as follow-up work.

`0049` extends the editor command set with more Far-like editing commands. Syntax
highlighting must not change those command semantics.

`0013` and `0038` matter for this work only where the implementation needs a
menu, status, or reusable control entry point for language/theme selection.

Original Far Manager gets rich syntax highlighting mostly through FarColorer.
FarColorer is powerful, but it is native, plugin-oriented, Colorer/HRC-based, and
brings a larger integration surface than needed for the first CSharpFar
implementation.

TextMate grammars are a better first fit for CSharpFar because:

1. they have an existing .NET implementation through TextMateSharp;
2. bundled grammars cover many common languages;
3. the tokenization model returns ranges and scopes, which map naturally to editor
   color spans;
4. the implementation can stay inside the C#/.NET application boundary;
5. custom grammars and themes can be added later without designing a general
   plugin system.

The TextMateSharp repository documents these relevant traits for the candidate
dependency: it is a .NET port of `vscode-textmate`, TextMate grammars use the
Oniguruma regular-expression dialect, NuGet packages exist for `TextMateSharp`
and `TextMateSharp.Grammars`, bundled grammars cover 50+ languages, bundled
themes include `Dark+`, and the project is MIT licensed. The implementation
phase must still verify redistribution requirements for TextMateSharp, bundled
grammars/themes, and the Oniguruma wrapper/native dependency.

Important limitations:

1. TextMate tokenization is lexical. It does not provide semantic highlighting,
   AST, LSP diagnostics, formatting, or autocomplete.
2. TextMateSharp grammar loading is documented for JSON grammar files.
3. Cross-grammar injections are not currently supported at the level documented by
   TextMateSharp, so the minimum language set must not rely on injections.
4. TextMate themes often use RGB colors and font styles. CSharpFar must map these
   to the current console rendering capabilities.
5. Some grammars can be expensive on very long lines or pathological regular
   expressions, so tokenization needs limits and clear fallback behavior.

Reference:
- https://github.com/danipen/TextMateSharp

## Scope

### Editor syntax model

Add editor-domain types that are independent from TextMateSharp:

```csharp
public interface IEditorSyntaxHighlighter
public sealed class EditorSyntaxHighlightRequest
public sealed class EditorSyntaxHighlightResult
public readonly record struct EditorColorSpan
public sealed class EditorSyntaxLanguage
public sealed class EditorSyntaxTheme
public sealed class EditorSyntaxHighlightCache
```

Rules:

1. `EditorColorSpan` describes a visible text range and a `CellStyle` or editor
   style token.
2. Syntax highlighting must be read-only. It must never mutate the text buffer,
   selection, undo history, dirty revision, bookmarks, or cursor.
3. The editor renderer asks for color spans for visible lines.
4. Selection rendering has priority over syntax highlighting.
5. Search result rendering, if added later, has priority over syntax highlighting
   but lower priority than cursor rendering.
6. Plain editor colors remain the fallback when no highlighter is available.
7. The syntax surface belongs to the internal editor domain and must not become a
   public highlighter plugin abstraction in this spec.

### TextMateSharp implementation

Add a TextMate-specific implementation behind the editor-domain interface:

```csharp
public sealed class TextMateEditorSyntaxHighlighter : IEditorSyntaxHighlighter
public sealed class TextMateGrammarRegistry
public sealed class TextMateThemeMapper
public sealed class TextMateLanguageSelector
```

Responsibilities:

1. Load bundled grammars through `TextMateSharp.Grammars`.
2. Select grammar by file extension, filename, or explicit editor setting.
3. Tokenize lines through TextMateSharp.
4. Preserve TextMate rule state per line so multiline comments and strings work.
5. Map TextMate scopes and theme rules to console `CellStyle`.
6. Hide TextMateSharp types behind TextMate-specific classes.

The implementation must not leak TextMateSharp types into `EditorSession`,
`FileEditor`, or the generic renderer.

### Language detection

The first implementation supports automatic language detection by:

1. file extension;
2. exact filename, for files such as `Dockerfile`;
3. optional first-line hints for later use, for example shebang detection.

Minimum languages for acceptance:

1. C#;
2. JSON;
3. XML;
4. Markdown;
5. PowerShell;
6. YAML.

The implementation may support more languages through the bundled grammar package,
but tests only need to lock the minimum set.

### Theme and color mapping

Add editor settings:

```csharp
Editor.SyntaxHighlightingEnabled = true
Editor.SyntaxTheme = "Dark+"
Editor.SyntaxLanguage = "auto"
Editor.SyntaxMaxLineLength = 20000
Editor.SyntaxTokenizationTimeoutMs = 50
Editor.SyntaxMaxSynchronousLines = 300
Editor.SyntaxUseTrueColor = "auto"
Editor.SyntaxFallbackToPlainText = true
```

Rules:

1. Theme names come from the TextMate grammar/theme registry.
2. If the renderer supports true RGB color, use theme colors directly.
3. If the renderer supports only the current console palette, map theme colors to
   the nearest available palette color.
4. The current `CellStyle` model is palette-based. Stage 1 should not add a broad
   true-color renderer unless it is small and isolated; palette mapping is the
   expected first path.
5. If a theme cannot be loaded, fall back to a default dark theme.
6. If color mapping fails for a token, render that token with the editor default
   style.
7. Do not hard-code C# colors in the renderer.

### Rendering integration

Change editor rendering so `DrawTextLine` applies style layers in this order:

1. base editor style;
2. syntax color spans;
3. selection style;
4. cursor position.

Do not add syntax logic directly inside low-level screen writing. The renderer
should receive resolved spans and choose final cell styles.

Tabs and virtual columns must keep the existing editor behavior. Syntax spans are
based on logical text indexes. Rendering maps them to visual columns using the
same tab expansion logic already used by the editor.

### Tokenization cache

Add a cache per editor session.

Cache data:

1. document revision;
2. line index;
3. source line text hash or revision marker;
4. TextMate rule stack before the line;
5. TextMate rule stack after the line;
6. computed color spans or token style data.

Invalidation rules:

1. ordinary cursor movement does not invalidate syntax cache;
2. editing a line invalidates that line and later lines until rule state becomes
   stable again;
3. format changes do not invalidate syntax cache unless text changes;
4. changing language or theme invalidates all syntax cache;
5. changing palette or true-color mode remaps cached token styles without
   retokenizing where possible.

The first implementation may retokenize from the nearest previous cached stable
line instead of implementing complex incremental parsing. It must stay bounded
and avoid retokenizing the whole file on every key press.

### Performance and fallback

Tokenization must not make the editor feel frozen.

Rules:

1. Visible lines may be tokenized synchronously if the budget is small.
2. Tokenization beyond the visible range should be background or deferred.
3. Very long lines are rendered as plain text after `Editor.SyntaxMaxLineLength`.
4. If a grammar throws or times out, disable highlighting for the current session
   and show a clear status message or diagnostics entry.
5. If a grammar is unknown, use plain text without error dialogs.
6. Syntax highlighting must never block save, close, search, copy, paste, or undo.

### User controls

Add minimal controls:

1. setting to enable or disable syntax highlighting globally;
2. editor-local toggle for the current session;
3. language selection menu for the current session;
4. theme selection setting or menu;
5. visible status indicator when syntax highlighting is disabled due to error,
   unknown language, or file limits.

Exact key bindings are not part of this spec unless an existing editor command
inventory already reserves a safe chord. Prefer menu/settings integration over
new global key chords.

### Custom grammars and themes

Add paths for future user customization:

```csharp
Editor.SyntaxUserGrammarsPath
Editor.SyntaxUserThemesPath
```

Stage 1 may only define settings and folder structure. Loading user grammars and
themes can be implemented if it is small and testable, but it is not required for
the first accepted version.

Required behavior for Stage 1:

1. missing custom folders are ignored;
2. invalid custom grammar or theme files do not break built-in grammars;
3. diagnostics mention the file that failed to load.

### Diagnostics

Add internal diagnostics for syntax highlighting:

1. selected grammar;
2. selected theme;
3. whether true-color or palette mapping is used;
4. grammar load failures;
5. theme load failures;
6. tokenization timeout or line-length fallback;
7. disabled state.

Diagnostics may be exposed through logs or a simple editor status/debug path.
Do not add a large UI for diagnostics in this spec.

## Non-goals

Do not implement in this spec:

1. FarColorer HRC/HRD compatibility;
2. native Colorer integration;
3. semantic highlighting based on Roslyn or LSP;
4. autocomplete;
5. diagnostics from compilers or language servers;
6. formatting or code actions;
7. outline/function list;
8. bracket matching beyond optional simple visual pairing;
9. syntax highlighting in the F3 viewer;
10. user-editable key bindings;
11. a plugin API for third-party highlighters;
12. a separate public editor highlighter model outside CSharpFar's internal
    editor domain;
13. full VS Code theme fidelity;
14. exact rendering of TextMate font styles such as italic or underline if the
    console renderer cannot represent them;
15. compatibility wrappers for temporary syntax-highlight implementation details.

## Acceptance criteria

### Editor behavior

- Opening a supported source file shows syntax coloring in the editor.
- Unknown file types open normally as plain text.
- Disabling syntax highlighting returns the editor to plain text rendering.
- Changing the theme updates visible syntax colors without changing document text.
- Changing the language retokenizes the document with the selected grammar.
- Selection rendering remains readable and takes priority over syntax colors.
- Cursor rendering remains visible.
- Search, replace, copy, paste, undo, redo, save, bookmarks, and Far-like editing
  commands keep existing behavior.

### Language coverage

Automated tests cover at least:

- C#;
- JSON;
- XML;
- Markdown;
- PowerShell;
- YAML.

Tests do not need to verify exact RGB values for every token. They should verify
that expected token categories produce non-default styles and stable spans.

### Cache and performance

- Cursor-only movement does not retokenize unchanged lines.
- Editing one line invalidates the changed line and dependent following lines.
- Multiline comments or strings keep correct highlighting across line boundaries.
- Lines longer than `Editor.SyntaxMaxLineLength` render as plain text.
- Tokenization timeout or grammar failure does not close the editor.
- The editor remains usable when syntax highlighting is disabled for a session.

### Settings and diagnostics

- `Editor.SyntaxHighlightingEnabled = false` disables highlighting globally.
- `Editor.SyntaxLanguage = "auto"` uses automatic detection.
- Invalid theme name falls back to a default theme.
- Invalid grammar or unknown grammar falls back to plain text.
- Diagnostics expose selected grammar, selected theme, fallback state, and last
  syntax error.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused automated checks:

- Generic renderer tests with fake `EditorColorSpan` values.
- Syntax style layering tests: base style, syntax style, selection style, cursor.
- TextMate grammar selection by extension and filename.
- TextMate theme mapping to `CellStyle`.
- C# tokenization smoke test.
- JSON tokenization smoke test.
- XML tokenization smoke test.
- Markdown tokenization smoke test.
- PowerShell tokenization smoke test.
- YAML tokenization smoke test.
- Unknown file type fallback.
- Invalid theme fallback.
- Long-line fallback.
- Tokenization exception fallback.
- Cache invalidation after editing one line.
- Multiline comment/string state propagation.
- Regression tests for existing editor commands from `0046` and `0049`.

Manual checks:

1. Open a `.cs` file and verify keywords, comments, strings, and types are
   visually distinct.
2. Open `.json`, `.xml`, `.md`, `.ps1`, and `.yml` files.
3. Toggle syntax highlighting off and on.
4. Change theme and verify the editor redraws.
5. Edit inside a multiline comment and verify following lines update.
6. Move the cursor through a large file and verify the editor stays responsive.
7. Use selection, copy, paste, undo, redo, search, replace, save, and bookmarks
   while highlighting is enabled.
8. Open an unknown extension and verify the editor behaves as plain text.
9. Open a file with a very long line and verify fallback does not freeze the UI.
