# 0001 - CSharpFar file manager

Type: Spec
Status: Current

## Context

CSharpFar is a console two-panel file manager written in C# for .NET 10. Its
primary user model follows Far Manager: two file panels, a command line,
function-key actions, command and directory history, file operations, a viewer,
and an editor.

The product supports separate Windows and Unix/Linux executable hosts over a
shared application layer. The implementation must keep direct control over
console input, rendering, shell command execution, and the screen buffer so that
Far-like panel visibility and command-output scenarios remain possible.

## Scope / Behavior

- The application provides left and right file panels with independent current
  directories, cursor positions, selections, sorting, and scrolling.
- A readable local directory remains navigable when individual child entries
  cannot be inspected. Such entries are omitted without changing filtering,
  sorting, parent-directory, summary, or rendering behavior.
- Keyboard operation is primary. Mouse input may enhance behavior but must not
  be required for core workflows.
- The command line runs shell commands in the current directory of the active
  panel, preserves command history, and shows command output in the same
  console buffer.
- `Ctrl+O` toggles panel visibility. When panels are hidden, the user sees the
  console output underneath the UI; toggling panels back restores panel state.
- The UI includes a bottom function-key bar and status areas that remain inside
  the visible console viewport.
- The internal viewer opens files from panels for read-only inspection.
- The internal editor opens text files for editing and saving.
- Settings and histories are persisted across application restarts.
- The solution is layered so that UI orchestration, console access, file-system
  models, shell execution, file operations, viewer behavior, editor behavior,
  settings, and histories are not concentrated in a single entry-point file.
- Platform selection happens in explicit host/platform composition roots. The
  application layer owns shared application bootstrap, commands, settings,
  histories, menus, and application services, but does not directly choose
  Windows or Unix implementations for console access, shell execution, file
  launching, credentials, volumes, mount points, or terminal screen mode.
- Windows and Unix/Linux hosts are separate executable projects that reference
  the shared application layer and their platform service implementations.
- Linux x64 releases are distributed both as a portable archive and as an
  `amd64` Debian package that installs the `csharpfar` command. The Debian
  package is built from the same verified self-contained Unix publish output as
  the portable archive.
- The project publishes a signed APT repository for the `stable` suite and
  `main` component. Repository metadata and its public signing key are
  available from the project GitHub Pages site, while private signing material
  remains outside source control and published artifacts.
- Console rendering uses an internal rendering layer and reusable console UI
  components rather than a third-party TUI framework.

## Non-goals

- No plugin system is required for the first product scope.
- Full Far Manager macro compatibility is outside the current scope.
- Archive panels are outside the current scope.
- A full hex editor is outside the current scope.
- A full IDE-like code editor is outside the current scope.
- Exact compatibility with every Far Manager hotkey is outside the current
  scope unless another spec names the hotkey explicitly.
- Full Linux terminal parity is outside the current scope; the Unix/Linux host
  may use minimal platform services while the architecture remains explicit.
- macOS support is outside the current scope.
- Linux package architectures other than `amd64` and distribution formats
  other than the portable archive and Debian package are outside the current
  scope.

## Acceptance criteria

- The application can be launched as a .NET 10 console application on Windows
  through the Windows host.
- The solution provides a separate Unix/Linux host that builds for Linux without
  constructing Windows-only platform services.
- A user can navigate two file panels, switch the active panel, run shell
  commands, use command history, and return to panel navigation.
- Opening a readable local directory succeeds and shows its accessible children
  even when another direct child is inaccessible.
- `Ctrl+O` hides and restores panels without losing panel state or command-line
  text.
- Shell command output remains visible through `Ctrl+O` after the command
  completes.
- The viewer and editor can be opened from panel items.
- Settings and histories survive restart.
- The renderer keeps all visible UI inside the console viewport and avoids
  unwanted console scrolling during UI redraw.
- Release builds produce a Windows archive, a Linux portable archive, an
  `amd64` Debian package, and checksums covering all three assets.
- The published APT repository retains Debian packages from prior GitHub
  releases, exposes signed repository metadata, and supports installing and
  updating the `csharpfar` package through current APT clients.
- An APT-installed `csharpfar` command supports `--version` and `--self-test`.

## Verification

- Publish the Unix host as a self-contained single-file `linux-x64` binary and
  verify `csharpfar --version` and `csharpfar --self-test` before packaging.
- Build and inspect the Debian package, extract it into a temporary root, and
  run the installed binary's self-test from that root.
- Build the APT repository with a throwaway signing key and verify its signed
  metadata, package index, public key, package architecture, and pool filename.
- On a clean Debian or Ubuntu environment, add the repository with a dedicated
  `signed-by` keyring, run `apt update`, install `csharpfar`, and verify its
  version and self-test commands.
