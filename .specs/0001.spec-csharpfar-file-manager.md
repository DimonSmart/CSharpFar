# ТЗ: создание файлового менеджера CSharpFar

Source: `.worklog/0001.spec-csharpfar-file-manager.md`.

## 1. Цель проекта

Создать консольный двухпанельный файловый менеджер **CSharpFar** на C# и .NET 10, внешне и по основным сценариям похожий на Far Manager.

Проект должен быть пригоден для поэтапной разработки через Codex, Claude Code или аналогичный coding agent. Каждый этап должен давать рабочий результат, который можно запустить, проверить вручную, поправить и только после этого переходить к следующему этапу.

Главная цель первой версии: получить удобную консольную оболочку для Windows с двумя файловыми панелями, командной строкой, историей команд, базовыми файловыми операциями, просмотром и редактированием текстовых файлов.

## 2. Основные принципы

1. **Windows-first.** Основная целевая платформа: Windows Terminal / классическая консоль Windows / PowerShell / cmd.exe.
2. **.NET 10.** Целевой фреймворк: `net10.0`.
3. **C# only.** Основная логика пишется на C#.
4. **Без плагинной системы в первой версии.** Архитектура не должна мешать будущему расширению, но плагины не реализуются.
5. **Клавиатурное управление.** Основные сценарии должны работать без мыши.
6. **Пошаговая разработка.** После каждого этапа проект должен собираться, запускаться и иметь понятные критерии проверки.
7. **Far-like UX, но не полное клонирование.** Не нужно повторять все детали Far Manager. Нужно воспроизвести рабочую модель: две панели, командная строка, функциональные клавиши, истории, файловые операции, просмотрщик, редактор.
8. **Ctrl+O обязателен.** Нужно поддержать режим скрытия панелей, при котором под интерфейсом видно то, что было в консоли после выполнения shell-команд.

## 3. Не цели первой версии

В первой версии не нужно реализовывать:

1. плагины;
2. полноценную систему макросов Far Manager;
3. архивные панели;
4. FTP/SFTP;
5. сложную систему прав доступа;
6. полноценный hex editor;
7. полноценный IDE-like редактор кода;
8. полную совместимость с Far Manager по горячим клавишам;
9. поддержку всех терминалов Linux/macOS на уровне Windows.

Эти возможности можно оставить как backlog после стабильного MVP.

## 4. Технологический стек

### 4.1. Платформа

* C#
* .NET 10
* Windows-first console application
* Nullable reference types enabled
* Treat warnings as errors желательно включить после стабилизации базовой архитектуры

### 4.2. Рекомендуемый тип проекта

Создать solution:

```text
CSharpFar.sln
/src
  /CSharpFar.App
  /CSharpFar.Core
  /CSharpFar.Console
  /CSharpFar.FileSystem
  /CSharpFar.Shell
  /CSharpFar.Viewer
  /CSharpFar.Editor
/tests
  /CSharpFar.Tests
```

Минимально допустимо начать с меньшего количества проектов, но код должен быть разделён по слоям так, чтобы UI, файловая модель, shell-интеграция и операции над файлами не были свалены в один `Program.cs`.

### 4.3. Внешние зависимости

На старте желательно использовать минимум зависимостей.

Разрешены:

* стандартная библиотека .NET;
* `Microsoft.Extensions.*` для DI, logging и configuration;
* `System.Text.Json`;
* `Microsoft.Data.Sqlite` только если будет принято решение хранить историю и настройки в SQLite;
* тестовые библиотеки `xUnit` или `NUnit`.

Не использовать на первом этапе:

* Terminal.Gui;
* Spectre.Console как основной UI-движок;
* полноценные сторонние редакторы;
* сторонние файловые менеджеры или TUI-фреймворки.

Причина: для режима `Ctrl+O` нужен точный контроль над console buffer, теневым display buffer и способом запуска shell-команд. Сторонние TUI-фреймворки могут усложнить это поведение.

## 6. Ключевое требование: режим Ctrl+O

### 6.1. Поведение

`Ctrl+O` переключает режим видимости панелей:

* если панели видимы, они скрываются;
* пользователь видит содержимое консоли, которое было под интерфейсом;
* командная строка и панели не должны затирать этот вывод;
* повторное `Ctrl+O` возвращает панели;
* состояние панелей, курсор, выделение и текущие каталоги сохраняются.

Это один из принципиальных сценариев Far Manager. Для CSharpFar он должен быть заложен в архитектуру с самого начала.

### 6.2. Как должны выполняться shell-команды

Когда пользователь вводит команду в нижней строке:

1. приложение временно скрывает панели или освобождает область вывода;
2. команда запускается в текущем каталоге активной панели;
3. stdout/stderr команды выводятся в тот же console buffer, а не только в отдельный внутренний лог;
4. после завершения команды панели снова рисуются поверх консоли;
5. если пользователь нажимает `Ctrl+O`, он видит вывод выполненных команд.

### 6.3. Техническая реализация для Windows

Нужен отдельный слой `IConsoleDriver`:

```csharp
public interface IConsoleDriver
{
    ConsoleSize GetSize();
    ConsoleKeyInfo ReadKey(bool intercept);
    void WriteAt(int x, int y, ReadOnlySpan<char> text, ConsoleColor? foreground = null, ConsoleColor? background = null);
    void ClearRegion(Rect region);
    void SetCursorPosition(int x, int y);
    void SetCursorVisible(bool visible);
    ScreenSnapshot Capture(Rect region);
    void Restore(ScreenSnapshot snapshot);
}
```

Для Windows-реализации желательно использовать P/Invoke к Win32 Console API там, где стандартного `System.Console` недостаточно:

* чтение/сохранение участка screen buffer;
* восстановление участка screen buffer;
* управление курсором;
* корректная работа с цветами.

`ScreenRenderer` должен владеть теневым буфером отображения: UI рисует полный логический кадр, renderer сравнивает новый кадр с предыдущим и физически обновляет только изменившиеся горизонтальные участки. Код приложения не должен вычислять и передавать dirty regions для обычной перерисовки; `Rect` остаётся геометрией компонентов и областей `Capture`/`Restore`, а не протоколом инкрементального repaint.

Минимальный MVP может временно использовать упрощённый подход:

* перед отрисовкой панелей сохранять собственный underlay buffer;
* при `Ctrl+O` восстанавливать последнюю сохранённую подложку;
* после выполнения shell-команды обновлять подложку.

Но архитектурно код должен быть готов к полноценной Windows screen buffer реализации.

### 6.4. Что нельзя делать

Нельзя реализовывать `Ctrl+O` как простую очистку экрана.

Нельзя запускать команды только с redirected output, если из-за этого пользователь потом не видит реальный вывод в консоли под панелями.

Нельзя использовать alternate screen buffer как единственный экранный режим, если это ломает сценарий “панели скрылись, под ними виден shell output”.

## 7. Основной пользовательский интерфейс

### 7.1. Структура экрана

По умолчанию экран состоит из:

```text
┌────────────────────────────┐┌────────────────────────────┐
│ Left panel                 ││ Right panel                │
│                            ││                            │
│ file list                  ││ file list                  │
│                            ││                            │
└────────────────────────────┘└────────────────────────────┘
C:\current\path> command line
1Help 2User 3View 4Edit 5Copy 6RenMov 7Mkdir 8Delete 9Menu 10Quit
```

Точная псевдографика может отличаться, но визуально это должен быть двухпанельный console file manager.

### 7.2. Панели

Каждая панель должна показывать:

* текущий путь;
* список файлов и каталогов;
* имя;
* размер;
* дату изменения;
* атрибуты, если помещаются;
* выделение активной строки;
* выбранные файлы;
* строку `..` для перехода в родительский каталог.

### 7.3. Активная панель

* `Tab` переключает активную панель.
* Активная панель визуально отличается от неактивной.
* Все операции по умолчанию выполняются из активной панели в неактивную.

### 7.4. Навигация

Обязательные клавиши:

| Клавиша               | Действие                                                                               |
| --------------------- | -------------------------------------------------------------------------------------- |
| `Up` / `Down`         | Перемещение курсора                                                                    |
| `PageUp` / `PageDown` | Постраничная прокрутка                                                                 |
| `Home` / `End`        | Начало / конец списка                                                                  |
| `Enter` на каталоге   | Войти в каталог                                                                        |
| `Enter` на файле      | Выполнить файл или открыть по системной ассоциации, можно отложить до отдельного этапа |
| `Backspace`           | Перейти в родительский каталог                                                         |
| `Tab`                 | Переключить панель                                                                     |
| `Ctrl+O`              | Скрыть / показать панели                                                               |
| `Esc`                 | Закрыть текущий диалог или режим                                                       |

### 7.5. Функциональные клавиши

Минимальная раскладка:

| Клавиша   | Действие                                         |
| --------- | ------------------------------------------------ |
| `F1`      | Help, можно заглушку на первых этапах            |
| `F2`      | User menu, можно заглушку на первых этапах       |
| `F3`      | View                                             |
| `F4`      | Edit                                             |
| `F5`      | Copy                                             |
| `F6`      | Rename / Move                                    |
| `F7`      | MkDir                                            |
| `F8`      | Delete                                           |
| `F9`      | Menu / Settings, можно заглушку на первых этапах |
| `F10`     | Exit                                             |
| `Alt+F8`  | Command history                                  |
| `Alt+F11` | View/Edit history                                |
| `Alt+F12` | Directory history                                |

## 8. Командная строка

### 8.1. Поведение

В нижней части экрана должна быть командная строка.

Требования:

* ввод обычного текста;
* `Enter` запускает команду;
* команда выполняется в каталоге активной панели;
* после выполнения команда добавляется в историю;
* история сохраняется между запусками приложения;
* `Alt+F8` открывает окно истории команд;
* выбранная команда из истории вставляется обратно в командную строку;
* `Up` / `Down` в командной строке могут листать историю, если это не конфликтует с навигацией по панели.

### 8.2. Shell

Настройка shell:

```json
{
  "shell": {
    "kind": "cmd",
    "executable": "cmd.exe",
    "argumentsFormat": "/c {0}"
  }
}
```

В будущем можно добавить PowerShell:

```json
{
  "shell": {
    "kind": "powershell",
    "executable": "powershell.exe",
    "argumentsFormat": "-NoLogo -NoProfile -Command {0}"
  }
}
```

### 8.3. Вывод команд

Вывод shell-команд должен попадать в console buffer так, чтобы его можно было увидеть через `Ctrl+O`.

Для MVP допустимо:

* перед запуском команды временно скрыть панели;
* вывести приглашение и команду;
* дождаться завершения процесса;
* оставить вывод в консоли;
* затем перерисовать панели поверх.

После `Ctrl+O` этот вывод должен быть виден пользователю.

## 9. Файловая модель панели

### 9.1. `FilePanelItem`

Минимальная модель:

```csharp
public sealed class FilePanelItem
{
    public required string Name { get; init; }
    public required string FullPath { get; init; }
    public required bool IsDirectory { get; init; }
    public long? Size { get; init; }
    public DateTime LastWriteTime { get; init; }
    public FileAttributes Attributes { get; init; }
    public bool IsParentDirectory { get; init; }
}
```

### 9.2. `FilePanelState`

```csharp
public sealed class FilePanelState
{
    public required string CurrentDirectory { get; set; }
    public List<FilePanelItem> Items { get; } = new();
    public HashSet<string> SelectedPaths { get; } = new(StringComparer.OrdinalIgnoreCase);
    public int CursorIndex { get; set; }
    public int ScrollOffset { get; set; }
    public SortMode SortMode { get; set; }
    public bool SortDescending { get; set; }
    public string? FilterMask { get; set; }
}
```

### 9.3. Сортировка

Обязательные режимы:

* name;
* extension;
* size;
* last write time.

Каталоги по умолчанию показываются перед файлами.

### 9.4. Выделение

Минимальные операции:

| Клавиша               | Действие                                                    |
| --------------------- | ----------------------------------------------------------- |
| `Insert`              | Выделить / снять выделение текущего элемента и перейти вниз |
| `Gray *` или `Ctrl+A` | Выделить / снять выделение всех файлов на панели            |
| `Esc`                 | Снять выделение, если нет активного диалога                 |

Операции копирования, перемещения и удаления работают с выделенными файлами. Если выделения нет, операция применяется к элементу под курсором.

## 10. Файловые операции

### 10.1. Общие требования

Все операции должны:

* работать с одним файлом;
* работать с выделенной группой файлов;
* обрабатывать ошибки доступа;
* показывать понятные сообщения об ошибках;
* не падать от исключений `IOException`, `UnauthorizedAccessException`, `PathTooLongException`;
* иметь возможность отмены через `Esc` или `CancellationToken`, где это применимо.

### 10.2. Копирование `F5`

При нажатии `F5`:

1. определить список источников: выделение или текущий элемент;
2. открыть диалог назначения;
3. по умолчанию предложить путь текущего каталога противоположной панели;
4. выполнить копирование;
5. показать прогресс;
6. обновить обе панели после завершения.

Требования:

* рекурсивное копирование каталогов;
* сохранение структуры каталогов;
* обработка конфликтов:

  * overwrite;
  * skip;
  * overwrite all;
  * skip all;
  * cancel;
* отображение текущего файла;
* отображение общего прогресса по количеству файлов и по байтам, если размер заранее посчитан;
* корректная работа с большими файлами через потоковое копирование.

### 10.3. Перемещение / переименование `F6`

При нажатии `F6`:

* если выбран один файл и пользователь вводит новое имя без пути, выполнить rename;
* если указан путь, выполнить move;
* если выбрано несколько файлов, выполнить move в указанный каталог;
* по умолчанию предложить каталог противоположной панели.

### 10.4. Создание папки `F7`

* открыть маленький input dialog;
* создать папку в каталоге активной панели;
* если папка уже существует, показать ошибку;
* после создания обновить панель и поставить курсор на новую папку.

### 10.5. Удаление `F8`

Минимальное поведение:

* запросить подтверждение;
* удалить файл или выделенную группу;
* для каталогов выполнять рекурсивное удаление;
* после удаления обновить панель.

Опционально для Windows:

* удаление в корзину через Windows API;
* `Shift+Del` удаляет без корзины.

В MVP можно реализовать физическое удаление с явным предупреждением.

## 11. Внутренний просмотрщик `F3`

### 11.1. MVP

Просмотрщик должен:

* открывать файл из активной панели;
* показывать текст в полноэкранном режиме внутри консоли;
* поддерживать прокрутку вверх/вниз;
* поддерживать `PageUp`, `PageDown`, `Home`, `End`;
* закрываться по `Esc` или `F10`;
* не изменять файл.

### 11.2. Кодировки

Минимально:

* UTF-8 with BOM/no BOM;
* UTF-16 LE with BOM;
* system default encoding для Windows, если возможно.

Следующий уровень:

* Windows-1251;
* ручной выбор кодировки через диалог.

### 11.3. Большие файлы

Для MVP допустимо ограничить размер файла для полного чтения, например 20 MB.

Если файл больше лимита:

* показать предупреждение;
* читать файл постранично или открыть в streaming viewer;
* не загружать гигабайтный файл целиком в память.

### 11.4. Поиск

На следующем этапе:

* `F7` или `/` открывает поиск;
* поиск вперёд;
* переход к следующему совпадению;
* подсветка текущего совпадения, если это не усложняет рендер.

## 12. Встроенный редактор `F4`

### 12.1. MVP

Редактор должен:

* открывать текстовый файл;
* показывать содержимое в полноэкранном режиме;
* позволять вводить текст;
* поддерживать Backspace, Delete, Enter, стрелки;
* сохранять файл по `F2`;
* закрываться по `Esc` или `F10`;
* если есть несохранённые изменения, спрашивать подтверждение.

### 12.2. Модель текста

Для первой реализации можно использовать простую модель:

```csharp
List<string> lines
int cursorLine
int cursorColumn
int scrollLine
int horizontalOffset
bool isDirty
```

Не нужно на первом этапе делать rope/piece table. Это можно добавить позже, если редактор станет узким местом.

### 12.3. Line endings и кодировки

При загрузке файла нужно определить:

* `CRLF`;
* `LF`;
* последнюю пустую строку;
* BOM, если есть.

При сохранении желательно сохранять исходный стиль line endings и BOM.

### 12.4. Ограничения MVP

В MVP редактор может отказаться открывать очень большие файлы, например больше 10 MB, с понятным сообщением.

## 13. Истории

### 13.1. Виды историй

Нужно поддержать:

1. историю команд: `Alt+F8`;
2. историю открытых файлов viewer/editor: `Alt+F11`;
3. историю каталогов: `Alt+F12`.

### 13.2. Хранение

Для начала использовать JSON-файлы в пользовательском профиле:

```text
%APPDATA%\CSharpFar\settings.json
%APPDATA%\CSharpFar\command-history.json
%APPDATA%\CSharpFar\directory-history.json
%APPDATA%\CSharpFar\file-history.json
```

Portable mode:

```text
./CSharpFar.config/settings.json
./CSharpFar.config/command-history.json
```

Portable mode включается, если рядом с exe есть файл:

```text
CSharpFar.portable
```

### 13.3. Формат истории команд

```json
[
  {
    "command": "dotnet build",
    "workingDirectory": "C:\\Projects\\Demo",
    "timestampUtc": "2026-05-06T12:00:00Z"
  }
]
```

### 13.4. Диалоги истории

Окно истории должно позволять:

* выбрать запись;
* вставить её в командную строку;
* удалить запись, можно отложить;
* закрыть окно без выбора.

## 16. Обработка ошибок

Приложение не должно падать при обычных ошибках файловой системы.

Ошибки показывать пользователю в диалоге:

* нет доступа;
* файл уже существует;
* файл занят другим процессом;
* путь не найден;
* слишком длинный путь;
* ошибка чтения каталога;
* ошибка копирования;
* ошибка удаления.

Для неожиданных ошибок:

* писать лог в файл;
* показывать короткое сообщение;
* по возможности продолжать работу.

Логи:

```text
%APPDATA%\CSharpFar\logs\csharpfar-yyyyMMdd.log
```
