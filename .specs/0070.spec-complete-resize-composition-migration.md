## Goal

Application-owned console UI always rebuilds its visible composition from the
current application state and current viewport. It never restores a captured
application screen after a dialog, full-screen surface, or resize.

## Composition ownership

- Rendering bootstrap creates exactly one shared `UiCompositionHost` and one
  associated `ModalDialogHost`. Application components receive those instances
  through explicit dependencies; a modal host is not a service locator for the
  composition host.
- The main panels and hidden-panel command-line mode are the root surface.
  Editor, viewer, and help are temporary surfaces. Dialogs, popup UI, and
  progress UI are modal overlays, including FTP and SFTP connection editors.
- The composition host exclusively owns frame creation, stable-frame retry,
  resize recovery, and rendering of the active surface followed by visible
  overlays. Layers derive layout from `UiRenderContext` and retain their last
  stable layout for input hit testing.
- Closing an overlay or temporary surface redraws the remaining composition
  from state at the current viewport.

## Input and resize

- Resize events are consumed by composition input and never reach
  component-specific handlers.
- If a viewport change is detected after a semantic key, mouse, modifier, or
  future non-resize input event has been read, the host completes resize
  recovery and returns that already-read event. It must not replace it by a
  later event.
- Empty polling redraws only when the viewport requires recovery. Follow-mode
  viewer polling uses its surface input session and explicitly renders after
  state changes.

## Snapshot boundary

Application-owned UI must not capture or restore screen snapshots, open a
private frame, read raw screen/console input, or render an underlay callback.
Snapshots remain limited to renderer and console infrastructure, shell and
terminal interoperability, external child processes, renderer tests, and
documented public compatibility APIs. Built-in FTP and SFTP UI does not use its
compatibility snapshot API.

## Acceptance criteria

- Grow and shrink rebuild the active surface, each overlay, and the top-layer
  cursor in one stable composition; no default or stale cells remain.
- Text, focus, cursor, valid clamped scrolling, selected values, viewer/editor
  position, search state, and progress state survive resize.
- Help opened from the viewer overlays the viewer as a surface; closing it
  restores the viewer, not the panels. Editor opened from viewer similarly
  restores the viewer.
- Source-contract tests scan production source with a narrow documented
  allowlist for snapshot, frame, raw-input, underlay, and host-construction
  boundaries.

## Verification

Automated tests inspect final fake-console cells, not merely callback counts,
for grow and shrink of panels plus compare dialog, hidden panels plus modal,
editor plus modal, viewer plus modal, viewer plus help, FTP/SFTP editors, and
nested progress UI. Composition-host tests cover semantic key, mouse, and
modifier preservation across implicit viewport changes.

dotnet build
