## Goal

CSharpFar application-owned UI uses one shared composition stack for full-screen
surfaces and modal overlays. Resize recovery, layer close, and nested UI redraw
the visible composition from current state instead of restoring pixels captured
from an earlier viewport.

## Legacy lifecycle removal

Application-owned dialogs, popups, progress UI, editor, viewer, and help do not
use `ScreenRenderer.Capture` or `ScreenRenderer.Restore` to preserve their
underlay. They do not open private frames, read directly from `ScreenRenderer`,
or accept underlay-rendering callbacks. Production dialog APIs receive the
shared `ModalDialogHost`; raw-screen constructors and static host lookup are
not part of production UI wiring.

Low-level snapshot APIs remain available for shell underlay, terminal-screen
interoperability, external-console and child-process behavior, renderer tests,
and public compatibility APIs where removal would be an incompatible module API
change.

## Surface and overlay behavior

The main panels or hidden command-line screen are the root application surface.
Editor, viewer, and full-screen help are temporary full-screen surfaces opened
through the shared `UiCompositionHost`. Ordinary dialogs and progress UI are
modal overlays opened through the shared `ModalDialogHost`.

Nested dialogs are additional overlays above the parent. Closing a child
overlay redraws the parent overlay over the active surface at the current
viewport. Closing a temporary surface redraws the previously active surface at
the current viewport.

## Progress lifecycle

File-operation and search progress UI is a long-lived overlay in the shared
modal stack. Conflict and cancel confirmation dialogs open above progress
through the same `ModalDialogHost`. Background work publishes progress state;
UI-thread polling updates the overlay and reads input through composition input
methods.

## Resize and input

Composition input polling detects viewport changes even when the input queue is
empty. Resize input events are consumed by the composition host. Ordinary input
is not lost after one or more resize events. Once the viewport is stable,
repeated empty polling does not render repeatedly.

Every render callback calculates layout from `UiRenderContext.Size` or
`UiRenderContext.Viewport`. Mouse hit testing uses layout produced by the most
recent successful render.

## Built-in modules

Built-in FTP and SFTP UI uses `ModuleUiServices.ModalDialogs` for connection
manager and connection editor dialogs. Built-in modules do not use
`ModuleUiServices.SaveScreen` or `RestoreScreen` for application-owned UI.

## Acceptance criteria

- Resizing a modal over panels, hidden command line, editor, viewer, progress,
  or another modal redraws every visible layer using the final viewport.
- Closing a nested modal after resize redraws its parent at the current
  viewport and leaves no stale child cells.
- Closing a temporary editor, viewer, or help surface after resize redraws the
  previous surface at the current viewport.
- Progress remains below conflict and cancel overlays and redraws after those
  overlays close.
- Dropdown popups are rendered as part of the current composition and do not
  capture or restore an underlay snapshot.
- File-operation UI does not read keys through `System.Console`.
- Production code has no static `ModalDialogHost.For` or `UiCompositionHost.For`
  lookup and does not create a second host for a dialog.
- Grow and shrink resize paths are covered by automated tests that inspect the
  resulting buffer, not only callback counts.

## Verification

Run:

```bash
dotnet build
dotnet test
```

Source checks verify that application-owned UI has no remaining dialog,
progress, editor, viewer, help, command, FTP, or SFTP use of snapshot
restoration, private frames, raw-screen dialog constructors, static host lookup,
or raw `System.Console` input.
