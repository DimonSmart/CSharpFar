## Goal

Add a built-in FTP/FTPS panel source so a user can connect to an FTP server,
browse remote directories in a normal panel, and use the existing Far-style
workflows for viewing, copying, creating folders, renaming, and deleting remote
items where the server and provider support those operations.

## Context

Spec `0029` added the source-aware panel model and an SFTP provider. Spec `0030`
added panel-local retry behavior for source load failures. FTP/FTPS should use
the same `IFilePanelSource`, `PanelLocation`, provider capabilities, file
operation, and panel retry mechanisms.

FTP and FTPS are not SFTP variants. FTP uses separate command and data
connections, can run in passive or active data connection modes, and may be
plain-text or protected with TLS. FTPS needs TLS certificate validation instead
of SSH host key validation. The provider should therefore be separate from
`SftpFilePanelSource`, even if small UI or credential helper code can be shared.

The intended library is `FluentFTP`, added to `CSharpFar.FileSystem`. The exact
package version is an implementation-time choice and must be locked in the
project file.

## Scope

### Source identity and connection model

1. Add `PanelSourceId.Ftp(connectionId)` with values in the form
   `ftp:{connectionId}`.
2. Add an `FtpConnectionInfo` model with at least:
   - id;
   - display name;
   - host;
   - port;
   - username;
   - remote root path;
   - credential id when a password is saved;
   - connection security mode;
   - data connection mode;
   - data connection TLS setting;
   - expected TLS certificate fingerprint for FTPS modes;
   - optional active-mode local port range;
   - `ShowInDriveSelection`.
3. Use project-owned enums in Core instead of exposing FluentFTP types through
   application models. Required connection security values are:
   - plain FTP;
   - explicit FTPS;
   - implicit FTPS;
   - auto.
4. Required data connection values are:
   - auto passive;
   - passive;
   - active.
5. Store FTP connection metadata separately from SFTP metadata unless a small
   shared store removes real duplication without changing the meaning of the
   existing SFTP store. Do not silently require SFTP saved-connection migration
   for this feature.
6. Do not store plaintext passwords in metadata. Reuse the DPAPI-backed
   credential store with FTP-specific credential ids.

### FTP/FTPS provider

Add `FtpFilePanelSource` in `CSharpFar.FileSystem`, implementing
`IFilePanelSource` and using FluentFTP.

Provider behavior:

1. Remote paths use normalized absolute POSIX-style paths owned by the FTP
   provider. Empty paths normalize to the configured remote root path.
2. Capabilities include enumerate, open read, open write, create directory,
   delete, rename, copy from, copy to, and refresh.
3. Watch and edit are not exposed for FTP/FTPS.
4. Directory enumeration maps FluentFTP directory items to `FilePanelItem`
   values with the FTP source id. Provider paths must not be passed to local
   `System.IO` APIs.
5. `OpenReadAsync` and `OpenWriteAsync` return streams whose FTP client lifetime
   is tied to the stream lifetime, matching the SFTP ownership pattern.
6. Recursive delete is supported when requested by the existing delete flow and
   must observe cancellation between items.
7. Same-source rename works when the server allows it.
8. FTP-to-FTP copy between different connections is not supported in this MVP.
9. Cross-provider move is not supported in this MVP and must remain blocked by
   capabilities or a clear unsupported-operation message.
10. Provider exceptions are translated to user-readable `IOException`,
    `UnauthorizedAccessException`, or `DirectoryNotFoundException` values.
    Panel enumeration failures use the panel error/retry behavior from spec
    `0030`.

### FTPS security

1. Plain FTP is supported only when the user explicitly chooses it. The mode name
   in UI must make the lack of TLS obvious.
2. Explicit FTPS connects to the configured port and upgrades with TLS.
3. Implicit FTPS starts with TLS from connection open.
4. Auto mode is an explicit user choice. It must not be the only way to connect.
5. For FTPS modes, validate the server certificate through FluentFTP's
   certificate validation hook.
6. On first FTPS connection without a saved fingerprint, show the certificate
   fingerprint and require explicit trust before opening the panel.
7. Saved connections persist the accepted certificate fingerprint. A changed
   fingerprint blocks reconnect with a clear warning.
8. `ValidateAnyCertificate` or equivalent accept-any behavior is not allowed as
   saved connection behavior.
9. Data connection TLS is configurable. For FTPS modes it defaults to enabled.

### Data connection mode

1. Default to auto passive mode.
2. Passive mode must be available as an explicit choice.
3. Active mode must be available as an explicit choice.
4. Active mode may include an optional local port range. Invalid ranges fail
   validation before a connection attempt.
5. Custom NAT address resolution is out of scope unless active mode cannot be
   implemented without it.

### UI integration

1. Add a Plugins menu entry for FTP/FTPS connection management. FTP/FTPS is a
   first-party native CSharpFar plugin. Keep the existing SFTP entry.
2. Extend `Alt+F1` and `Alt+F2` source selection so it can show saved FTP/FTPS
   connections with labels that distinguish plain FTP from FTPS.
3. Add FTP/FTPS connection actions equivalent to the SFTP flow:
   - create a temporary connection;
   - save a connection;
   - connect to a saved connection;
   - edit a saved connection;
   - delete a saved connection;
   - enter or re-enter a password.
4. Add a console dialog in the existing dialog style with fields for:
   - connection name;
   - host;
   - port;
   - username;
   - password;
   - remote root path;
   - save connection;
   - save password;
   - show in drive menu;
   - connection security;
   - data connection mode;
   - data connection TLS;
   - active port range when active mode is selected;
   - certificate trust/fingerprint when an FTPS validation probe requires it.
5. After a successful connection, the chosen panel opens the configured remote
   root path and the panel title/status makes the FTP/FTPS source obvious.
6. Shared SFTP/FTP dialog or manager helpers are allowed only when names and
   responsibilities remain unsurprising. Do not create a broad remote-provider
   framework just to prepare for S3.

### File operations

1. Local-to-FTP and FTP-to-local copy stream through provider
   `OpenReadAsync`/`OpenWriteAsync` and report progress through the existing
   progress UI.
1. Provider copy supports only `CopyMode.Normal` and conflict decisions `Ask`,
   `Overwrite`, `Skip`, and `Rename` until provider metadata, seekable resume
   validation, and per-file salvage continuation are explicitly supported.
   `Reliable`, `Fast salvage`, and `OnlyNewer` must fail with clear unsupported
   messages if passed to the service.
2. Remote delete is permanent. Do not route FTP/FTPS deletes through the Windows
   recycle bin.
3. Remote view (`F3`) must work through the same provider-aware viewer path used
   by SFTP.
4. Remote edit (`F4`) remains out of scope.
5. Existing local and SFTP behavior must keep working.

## Non-goals

1. S3 or any object-storage provider.
2. A generic third-party plugin loader.
3. Merging SFTP and FTP into one protocol class.
4. Broad connection-manager consolidation beyond what is needed for this FTP
   provider.
5. Remote file watching.
6. Remote edit with temp-file synchronization.
7. FTP-to-FTP copy between different connections.
8. FXP or server-to-server FTP transfer.
9. FTP-specific resume through REST commands.
10. Cross-provider move between local, SFTP, and FTP.
11. Client-certificate authentication for FTPS.
12. Proxy support.
13. Manual non-UTF8 listing/path encoding selection unless implementation finds
    it is required for basic FluentFTP behavior.
14. Custom active-mode NAT address resolver unless implementation proves active
    mode cannot work without it.
15. Compatibility wrappers that preserve old path-only APIs only to avoid
    updating call sites.

## Acceptance criteria

1. Existing local filesystem and SFTP tests still pass.
2. `PanelSourceId.Ftp(connectionId)` exists and source identity participates in
   panel items and operations.
3. `Plugins` contains separate SFTP and FTP/FTPS entries.
4. `Alt+F1` and `Alt+F2` show saved FTP/FTPS connections marked by protocol
   security mode, plus an action to open the FTP/FTPS manager or connection
   dialog.
5. A temporary plain FTP connection can open a remote root in the active panel
   when the user explicitly selects plain FTP.
6. A temporary explicit FTPS connection can open a remote root after certificate
   trust is accepted.
7. A temporary implicit FTPS connection can open a remote root after certificate
   trust is accepted.
8. A saved FTP/FTPS connection stores metadata without plaintext secrets and can
   reconnect after app restart.
9. A changed saved FTPS certificate fingerprint blocks reconnect with a clear
   warning.
10. Passive or auto passive data connections work for browse and file copy.
11. Active mode validates its optional local port range before connecting.
12. Remote navigation supports entering directories and going to parent
    directories without assuming Windows drive roots.
13. `F3` views a remote FTP/FTPS file.
14. `F5` copies files from local to FTP/FTPS and from FTP/FTPS to local with
    progress and cancellation.
15. `F7` creates a remote directory when the FTP/FTPS panel is active.
16. `F8` deletes remote files/directories with the existing confirmation style
    and without using the recycle bin.
17. Single-source FTP/FTPS rename works when the server allows it.
18. Unsupported cross-provider operations are disabled or show a clear
    unsupported-operation message.
19. FTP connection, authentication, TLS, permission, missing-file, timeout, and
    network errors are shown as user-readable messages and do not crash the app.
20. No FTP/FTPS operation reaches `System.IO` with a remote provider path.

## Verification

Automated checks:

1. FTP source id and model validation tests.
2. FTP connection store tests proving metadata contains no plaintext password.
3. FluentFTP config mapping tests for plain FTP, explicit FTPS, implicit FTPS,
   auto, passive, auto passive, active, data TLS, and active port range.
4. FTP provider path normalization tests.
5. FTPS certificate fingerprint tests for first accept, matching reconnect, and
   changed fingerprint rejection.
6. Menu and drive-selection tests proving FTP/FTPS appears without removing
   SFTP.
7. Fake-source file operation regression tests for local-to-provider and
   provider-to-local copy if existing SFTP tests do not already cover the needed
   paths.

Manual checks:

1. Connect to a known test FTP server with plain FTP.
2. Connect to a known test server with explicit FTPS.
3. Connect to a known test server with implicit FTPS.
4. Save a connection, restart the app, and reconnect.
5. Browse remote directories, including root and parent navigation.
6. View a remote file with `F3`.
7. Copy local to remote and remote to local with visible progress.
8. Cancel a larger copy and verify partial operation reporting.
9. Create, rename, and delete a remote test directory/file.
10. Change or reject an FTPS certificate fingerprint and verify the connection
    is blocked with a clear message.
11. Verify unsupported operations show clear messages.
