# 0051 SPIKE Editor UTF-8 four-byte cursor movement

Type: spike

## Goal

Learn why moving the cursor in the built-in editor can fail or behave
unexpectedly on Unicode characters encoded as four bytes in UTF-8.

## Hypothesis

The editor stores text as .NET strings and moves the cursor by UTF-16 `char`
indexes. Unicode scalar values outside the BMP, such as emoji and many historic
scripts, occupy two UTF-16 code units. Cursor movement, deletion, rendering, or
status display may therefore stop inside a surrogate pair instead of treating it
as one logical character.

## Experiment

1. Create or load a UTF-8 file containing BMP characters and non-BMP characters
   such as `U+1F600`, `U+1F642`, and `U+10348`.
2. Inspect editor model operations that move by one character, delete by one
   character, render lines, and report the current character code.
3. Add focused automated coverage for cursor movement across non-BMP
   characters.
4. Fix the smallest editor-domain surface that lets cursor movement treat
   surrogate pairs predictably.

## Constraints

Do not add a compatibility layer or a separate Unicode abstraction unless the
current editor model cannot be fixed directly.

Keep the change inside the internal editor model and renderer paths from `0046`
and `0049`.

Do not change syntax highlighting behavior except where it consumes editor text
positions affected by the fix.

## Done when

- the result is documented;
- the recommendation is written;
- follow-up is clear;

## Result

The hypothesis was confirmed.

The editor stored line positions as UTF-16 indexes, which is still the internal
representation, but cursor movement and single-character deletion also advanced
by one UTF-16 `char`. A non-BMP Unicode scalar value such as `U+1F642` occupies
two UTF-16 code units, so `RightArrow`, `LeftArrow`, `Backspace`, and `Delete`
could place the cursor inside a surrogate pair or remove only half of the
character.

Implementation summary:

- Added `EditorUnicode` for editor-local Unicode scalar boundaries over the
  existing UTF-16 string storage.
- Changed `EditorSession.MoveLeft`, `MoveRight`, `DeleteBack`, and
  `DeleteForward` to move/delete whole Unicode scalar values.
- Normalized editor buffer and cursor positions so direct editor operations do
  not keep the cursor inside a surrogate pair.
- Changed editor status reporting to show the full scalar code point, for
  example `U+1F642/128578`, instead of a surrogate half.
- Changed status column display to count non-BMP scalar values as one logical
  character while preserving virtual columns after line end.
- Changed editor visual-column calculation to use terminal display cell width:
  emoji and other wide scalars occupy two screen cells, while logical editor
  columns still count them as one symbol.
- Added a custom editor cursor style for wide scalars. When the cursor is on a
  wide surrogate-pair glyph, the editor highlights the whole glyph range and
  hides the native one-cell console cursor so it does not blink over only half
  of the rendered glyph.
- Added editor-level blinking for the custom wide-glyph cursor. While waiting
  for input on a wide glyph, the editor polls input and toggles the full-glyph
  cursor style on the same blink interval instead of using the native one-cell
  cursor.
- Fixed line splitting at the end of a line containing non-BMP characters.
  `InsertText` no longer normalizes the post-insert cursor before the buffer is
  changed, so `Enter` can move to the newly-created next line instead of being
  clamped back to the current line.
- Added automated regression coverage for movement, selection, deletion, status,
  screen cursor column, wide-glyph cursor styling, wide-glyph cursor blinking,
  line splitting at end of line, and saving after moving across a 4-byte UTF-8
  character.
- Added focused coverage that distinguishes emoji display width from a non-emoji
  supplementary scalar such as `U+10348`.
- Created an ignored manual sample file at
  `artifacts/editor-utf8-four-byte.txt` containing `U+1F642`, `U+1F600`, and
  `U+10348`.

Deviations:

- The fix does not convert the editor position model from UTF-16 indexes to a
  full grapheme-cluster model. It keeps the current editor storage and prevents
  cursor/editing operations from splitting surrogate pairs.
- The custom wide-glyph cursor is rendered and blinked by the editor instead of
  the native console cursor while the cursor is on that glyph.
- Combining marks, zero-width joiner emoji sequences, East Asian width, and
  terminal-specific display-width rules remain outside this spike.

## Recommendation

Keep this narrow scalar-boundary fix. It addresses the reported 4-byte UTF-8
case without changing the editor storage model or introducing a compatibility
layer.

If later reports involve composed grapheme clusters or terminal cell widths,
create a separate spike before changing the position model more broadly.

## Follow-up

- Consider a later editor text-width spike if the renderer needs correct
  grapheme cluster or East Asian width behavior across terminals.
