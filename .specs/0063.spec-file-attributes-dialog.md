## Intent

CSharpFar provides a Far-like dialog for viewing and changing metadata of the
current or selected local file-system items. The dialog is capability-based:
the application asks the active metadata provider which attributes and times
are visible and editable, and the UI does not hard-code platform-specific
attribute policy.

## Behavior

The attributes command works from the active file panel. If the panel has a
selection, the command targets the selected items; otherwise it targets the
current item. The parent-directory entry `..` is not changed. The first version
supports only `PanelSourceId.Local`; non-local panels show the message `File
attributes are supported only for local files.` and do not open the dialog.

The command reads metadata through `IFileMetadataService`. Single-item metadata
uses the item path directly. Multi-item metadata uses a merged snapshot where
values that differ between targets are represented as indeterminate or blank in
the dialog. Applying changes refreshes the active panel and keeps the cursor on
the same item name when possible. Partial apply failures are not rolled back;
the user sees changed and failed counts plus per-item error messages.

File metadata uses stable application model identifiers rather than UI-owned
bit lists:

- `FileAttributeId`: `ReadOnly`, `Hidden`, `System`, `Archive`, `Temporary`,
  `NotContentIndexed`, `Compressed`, `Encrypted`, `SparseFile`, `Offline`,
  `ReparsePoint`, and `Directory`.
- `AttributeEditState`: `Unchecked`, `Checked`, and `Indeterminate`.
- `FileAttributeDescriptor`: id, label, optional hotkey, visibility,
  editability, and optional disabled reason.
- `FileMetadataSnapshot`: path, display name, directory flag, raw
  `FileAttributes`, optional creation, write, and access times, optional owner
  display name, provider-supplied attribute descriptors, and optional typed Unix
  metadata.
- `FileMetadataChangeSet`: changed attributes, optional changed times, and
  explicitly changed Unix permission bits.
- `FileMetadataApplyResult`: processed count, changed count, and structured
  per-path apply errors.

Unix permissions use a separate typed model rather than extending
`FileAttributeId` or introducing provider-owned string identifiers. The model
contains owner, group, and others read/write/execute bits; set-user-ID,
set-group-ID, and sticky bits; numeric uid/gid; resolved owner/group names; and
per-bit tri-state edit state. Owner and group are displayed but are not
editable.

The dialog shows the target display name, supported attributes, write,
creation, and access times, and the owner. Unavailable owner is displayed as
`<not available>`. Owner editing is not part of this dialog.

Time fields use `dd.MM.yyyy HH:mm:ss` with invariant parsing and formatting.
For a single file, existing values are prefilled. For multiple files, equal
values are prefilled and differing values are blank. Blank time fields mean
"leave unchanged" on apply. Each editable time field has its own `Original`,
`Current`, and `Blank` actions placed to the right of that field: original
restores the value captured for that field when the dialog opened, current sets
that field to the current local time, and blank clears that field. Providers
expose whether creation, write, and access times can be edited; unsupported
fields are disabled and do not show editable per-field time actions.

Windows providers use .NET file attributes for ordinary flags. The first
Windows version makes `Read only`, `Hidden`, `System`, `Archive`, `Temporary`,
and `Not indexed` editable. It may show but not edit `Directory`, `Reparse
point`, `Compressed`, `Encrypted`, `Sparse`, and `Offline`. The Windows
`System properties` action opens shell properties through `UseShellExecute =
true` and the `properties` verb.

Linux providers read and write POSIX permission bits through managed Unix file
mode APIs. The dialog shows a tri-state owner/group/others permission matrix,
special permission bits, owner, group, and the mode as four octal digits followed
by two spaces and the nine-character symbolic permission string without a file
type prefix. Symbolic execute positions use Unix-compatible `s`/`S` for setuid
and setgid and `t`/`T` for sticky. Multi-selection marks a permission bit
indeterminate when selected items differ and displays `Mode: <mixed>` instead of
a partially mixed mode; it applies only bits explicitly changed by the user.
Owner/group and numeric ids are shown only when equal across the selection;
mixed values are displayed as unavailable or mixed.

When Linux Unix mode was read successfully, the ordinary `Read only` attribute remains
visible for compatibility but is disabled with the reason `Use Unix permissions
to edit write bits.` Permission changes are made only through the Unix matrix.
If Unix metadata cannot be read, the provider retains the minimal editable
`Read only` behavior so the dialog remains useful.

Linux owner and group names are resolved through native APIs. Failed identity
resolution falls back to the numeric uid or gid, and metadata read failures do
not prevent the dialog from opening.

macOS providers preserve existing ordinary attribute behavior. Hidden is
editable only when the provider explicitly supports a native hidden capability;
generic Unix logic must not implement hidden by renaming.

## Architecture And Patterns

Metadata read and write behavior lives behind `IFileMetadataService` and
platform/provider implementation classes. `IFileSystemService` is not expanded
with metadata editing operations merely for this dialog.

`FileAttributesDialog` is an application-level wrapper over reusable UI
components. It uses `ModalDialogRenderer`, `ScrollableFormDialog`,
`CheckBoxRow` or tri-state checkbox rows, `TextInputRow`, reusable button rows
or reusable input-plus-button form rows, `SeparatorRow`, and `LabelRow`. It must
not locally implement dialog frames, buttons, checkboxes, scrollbars, focus
navigation, or ordinary form mouse hit testing.

The first key binding is `Shift+F9 = Attr`. `Ctrl+A` remains select-all or
command-line selection and is not repurposed by this feature.

If multiple selected files are supported immediately, the reusable UI library
owns a tri-state checkbox primitive whose visible states are `[ ]`, `[x]`, and
`[-]`. Space or Enter changes `Unchecked -> Checked -> Unchecked` and
`Indeterminate -> Checked`.

## Non-Goals

- Recursive attribute changes for directories.
- Windows ACL editing.
- Owner or group editing.
- Linux `chattr` or `lsattr` editing.
- Linux inode flags, including immutable, append-only, nodump, noatime, sync,
  and nocow. A future implementation must use native ioctl/statx APIs rather
  than command parsing and must explicitly confirm dangerous flag changes.
- Extended attribute or ACL editing.
- NTFS compression or encryption editing.
- Alternate data streams.
- FTP, SFTP, archive, or virtual-panel attributes.
- Rename-based hidden toggling on Linux.
- Background mass operation progress.

## Acceptance Criteria

- A local file or directory can open the attributes dialog.
- The dialog shows target name, provider-supported attributes, times, and owner.
- Unsupported attributes are absent or visibly non-editable.
- Windows can edit `Read only`, `Hidden`, `System`, `Archive`, `Temporary`, and
  `Not indexed`.
- Linux files and directories show owner/group/others rwx permissions, setuid,
  setgid, sticky, owner, group, and mode as `0644  rw-r--r--` when Unix metadata
  is available, without an `ls -l` file type prefix.
- The symbolic mode uses Unix-compatible `s`/`S` and `t`/`T` special-bit
  notation, and a multi-selection with differing permissions shows
  `Mode: <mixed>`.
- Linux permission changes use managed Unix file mode APIs and change only bits
  explicitly edited by the user; no shell command fallback is used.
- Multi-selection merges each Unix permission bit independently and preserves
  indeterminate bits that the user did not edit.
- Linux owner and group are read-only and degrade to numeric identifiers or
  unavailable values without preventing the dialog from opening.
- Linux `Read only` is non-editable when the permission matrix is available and
  retains its prior minimal behavior when Unix metadata cannot be read.
- Windows behavior remains unchanged apart from carrying the optional Unix
  model through shared metadata types.
- UI code uses reusable dialog/form components and does not reimplement frames,
  buttons, checkboxes, scrollbars, focus, or form hit testing.
- Platform policy is isolated in metadata provider/service code, not in the
  dialog or command.
- Access and unsupported-operation failures are shown through structured apply
  errors.
- Applying successful changes refreshes the active panel and preserves cursor
  position by file name when possible.
- Specs and focused tests cover metadata service behavior, dialog change-set
  behavior, command routing, and tri-state checkbox behavior when used.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused checks:

- Temp-file metadata tests prove `ReadOnly` can be set and cleared.
- Platform-specific descriptor tests verify editable flags by OS and skip when
  not applicable.
- Dialog tests prove cancel does not apply changes, blank times are unchanged,
  per-field `Current` uses an injectable clock, and per-field `Original`
  restores opening values.
- Multi-file tests prove differing attributes become `Indeterminate`.
- Mapping tests cover ordinary and special Unix mode bits and round trips.
- Formatter tests cover octal and nine-character symbolic mode output, including
  setuid, setgid, sticky, and the two-space display separator; dialog tests cover
  mixed-mode display.
- Unix metadata tests prove permission merge, same/mixed owner and group, and
  apply semantics for checked, unchecked, indeterminate, and empty changes.
- Unix-only temp-file integration tests prove managed mode read and write for a
  file and skip on Windows.
- Command tests prove local-only behavior, parent-directory no-op behavior,
  panel refresh, and cursor preservation.
