# 0017 - Current-console program launch

Type: Spec
Status: Current
Replaces:
- none

Related:
- 0009
- 0016
- 0022

## Goal

Change external program launch behavior so executable files and typed commands run
like Far Manager: the child console process writes directly to the same console,
CSharpFar waits for it to exit, and then CSharpFar redraws its UI.

## Context

`0009` opened every file through Windows Shell. That is still right for documents,
folders, images, PDFs, URLs, and other association-based objects, but it is not the
desired behavior for console programs. For executable files and commands, CSharpFar
should not redirect stdout/stderr and should not ask the shell to create a separate
window. The child process should inherit the current console and write directly to
the visible console buffer.

The current rendering model in `0016` already has a clear distinction between UI
rendering mode and shell output underlay. The terminal screen mode from `0022` makes
that boundary physical in VT-supported terminals: full-screen CSharpFar UI lives in the
alternate screen buffer, while external program output lives in the main screen buffer
and remains part of terminal history. This VT-first model is also the intended direction
for future Unix-like terminal portability.

## Scope

- Keep command-line execution in current-console mode: `UseShellExecute = false`,
  no stdin/stdout/stderr redirection, no new console window, and wait for process
  exit.
- Before every current-console child process starts, ensure the terminal is in main
  screen mode. In VT-supported terminals this means leaving alternate screen mode before
  process execution and scrolling the main screen buffer viewport to the bottom before
  synchronizing the renderer with that visible main screen viewport, then writing the
  prompt and child output.
- After the child process exits, return to alternate screen mode only when panels should
  be visible; remain in main screen mode when both panels are hidden.
- While a current-console child process is running, `Ctrl-C` is delivered to the
  child process as a normal console control event so a long-running child can be
  stopped without closing CSharpFar.
- When opening files from a panel, run `.exe` and `.com` directly in the current
  console and wait for exit.
- When opening `.bat` or `.cmd`, run through `cmd.exe /c` in the current console
  and wait for exit.
- Keep non-executable files on Windows Shell association behavior.
- Make the Application hide/restore the UI around current-console file launches,
  matching the existing command-line execution path.
- Preserve the existing shared Enter and double-click open flow.
- Cover launch selection and process start options with unit tests that do not
  start real external programs.

## Non-goals

- Do not implement Far Manager file association tables.
- Do not add a compatibility path for the old Shell-only executable launch.
- Do not redirect external program output into CSharpFar UI widgets.
- Do not parse arbitrary command-line arguments for files opened from the panel.
- Do not implement detached/background launch behavior.
- Do not add cross-platform file launch behavior beyond current Windows-oriented
  requirements.

## Acceptance criteria

- Typed commands still run via the configured shell in the current console and wait
  for exit.
- Typed commands and executable panel launches write to the terminal main screen buffer,
  not to the alternate screen buffer used for CSharpFar UI, and start from the visible
  bottom of that main screen buffer without first blanking the visible main viewport.
- Pressing `Ctrl-C` while a current-console child process is running interrupts
  that child process without terminating CSharpFar.
- `.exe` and `.com` opened from a panel use `UseShellExecute = false`, do not
  redirect standard streams, do not create a new window, and wait for exit.
- `.bat` and `.cmd` opened from a panel use `cmd.exe /c` with the same current
  console behavior.
- Non-executable files still use Shell association with `UseShellExecute = true`
  and `Verb = "open"`.
- Opening executable files hides/restores the CSharpFar UI through the terminal screen
  mode boundary and refreshes panels after the process exits.
- Unit tests verify routing without launching real external programs.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual checks:

1. Run a console `.exe` from a panel and confirm its output appears in the same
   console before panels are redrawn.
2. Run a long-running command such as `ping 127.0.0.1 -t`, press `Ctrl-C`, and
   confirm the child command stops while CSharpFar redraws and continues running.
3. Run a `.cmd` or `.bat` file from a panel and confirm it uses the same console.
4. Open a document file and confirm it still uses the system association.
