# 0054 - Paranoid copy auto retry

## Goal

Make `Paranoid` copy automatically retry read-side failures without showing a modal
wait prompt.

## Context

`Paranoid` copy already validates partial destination files and can roll back to a
safe matching offset before resuming. The current failure path still treats read
errors like ordinary copy errors and can force a user decision before retrying.

Copy dialog mouse interaction is part of the reusable control contract owned by
`0038`.

## Scope

1. In `Paranoid` copy, treat source-open and source-read failures as automatic retry
   cases.
2. Wait before each automatic read-side retry. Use a fixed one-minute production
   delay while keeping tests deterministic.
3. After a read-side retry, reopen source and destination, reread actual destination
   state, rerun tail validation, and continue only from the analyzer result.
4. Avoid triggering rollback search repeatedly for transient source errors. Rollback
   must be driven by verified content mismatch, not by read/open failures.
5. Keep write errors interactive: show the existing operation error and wait for the
   user decision.

## Non-goals

1. Retry settings UI.
2. Automatic retry for write failures.
3. Automatic retry for non-paranoid copy modes.
4. Provider-to-provider paranoid resume.
5. Changing validation window sizes or rollback limits.
6. Changing reusable form-dialog mouse behavior.

## Acceptance criteria

- `Paranoid` copy does not show a modal retry dialog for source-open or source-read
  failures.
- `Paranoid` copy waits before retrying a read-side failure.
- Repeated read-side failures do not repeatedly roll back destination content.
- After the source becomes readable again, `Paranoid` copy reruns tail validation and
  resumes from the safe offset.
- Write failures still use the existing interactive error path.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Focused tests:

1. Paranoid copy automatically retries a transient source-read failure.
2. Paranoid copy does not truncate repeatedly across repeated source-read failures.
3. Paranoid copy write failures still ask the conflict resolver.
