# 0054 - Reliable copy auto retry

Related:
- 0019
- 0030
- 0038

## Goal

Make `Reliable` copy automatically retry read-side and destination write
failures without showing a modal wait prompt.

## Context

`Reliable` copy already validates partial destination files and can roll back to a
safe matching offset before resuming. The current failure path still treats read
errors like ordinary copy errors and can force a user decision before retrying.

Copy dialog mouse interaction is part of the reusable control contract owned by
`0038`.

## Scope

1. In `Reliable` copy, treat source-open, source-read, destination-open, and
   destination-write failures as automatic retry cases.
2. Wait before each automatic retry using fixed internal retry delays while
   keeping tests deterministic.
3. After a retryable failure, reopen source and destination, reread actual
   destination state, rerun tail validation, and continue only from the analyzer
   result.
4. Avoid triggering rollback search repeatedly for transient source errors. Rollback
   must be driven by verified content mismatch, not by read/open failures.
5. Write failures must not continue from an in-memory stream position. If retry
   is exhausted, show the existing operation error and wait for the user
   decision or fail the file according to the active operation mode.

## Non-goals

1. Retry settings UI.
2. Automatic retry for non-reliable copy modes.
4. Provider-to-provider paranoid resume.
5. Changing validation window sizes or rollback limits.
6. Changing reusable form-dialog mouse behavior.

## Acceptance criteria

- `Reliable` copy does not show a modal retry dialog for source-open,
  source-read, destination-open, or destination-write failures.
- `Reliable` copy waits before retrying a retryable failure.
- Repeated read-side failures do not repeatedly roll back destination content.
- After the source becomes readable again, `Reliable` copy reruns tail validation and
  resumes from the safe offset.
- Destination write failures are retried through reopen, reread, and
  tail-validation before the interactive error path is used.

## Verification

Focused tests:

1. Reliable copy automatically retries a transient source-read failure.
2. Reliable copy does not truncate repeatedly across repeated source-read failures.
3. Reliable copy retries destination write failure with tail validation.
