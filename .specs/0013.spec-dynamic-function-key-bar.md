# 0013 SPEC Dynamic function key bar

Source: `.worklog/0013.spec-dynamic-function-key-bar.active.md`.

Type: spec

Replaces:
- none

Related:
- 0012

## Goal

Make the bottom function key bar a real part of the command and key binding system.

The bar must show the actions that are available for the current function-key modifier state. When the user holds `Alt`, the row changes to `Alt+F...` actions. When the user holds `Control`, the row changes to `Ctrl+F...` actions. Releasing the modifier restores the plain function-key row.

The behavior should feel close to Far Manager: the row is not just static decoration; it follows the active key layer and must not advertise actions that the current application state cannot execute.

## Context

The current bottom row is rendered by `StatusBarRenderer` from a hard-coded list:

```text
1Help 2UserMn 3View 4Edit 5Copy 6RenMov 7MkFold 8Delete 9ConfMn 10Quit
```

Actual key handling lives separately in `Application.HandleKey`. That code already supports several modified function-key commands:

1. `Alt+F1` selects the left panel volume.
2. `Alt+F2` selects the right panel volume.
3. `Alt+F7` starts search.
4. `Alt+F8` opens command history.
5. `Alt+F11` opens file history.
6. `Alt+F12` opens directory history.
7. `Ctrl+F3` sorts by name.
8. `Ctrl+F4` sorts by extension.
9. `Ctrl+F5` sorts by last write time.
10. `Ctrl+F6` sorts by size.

The renderer and the handler are therefore allowed to drift. The lower row can show `F7 MkFold` while `Alt+F7` is the command the user is preparing to invoke. It also cannot react to provider capabilities from search results or other read-only panels.

The class name `StatusBarRenderer` is also surprising because the project already has panel status rows. The bottom command row should be named as a function-key or key-bar renderer, not as a generic status bar.

## Scope

Add a small command binding model that is the source of truth for function-key commands and their labels.

Required key layers:

1. plain function keys;
2. `Alt` + function keys;
3. `Control` + function keys.

The model must support `F1` through `F12` because existing commands already use `Alt+F11` and `Alt+F12`. The renderer may clip or omit trailing keys on narrow terminals, but the data model must not be limited to `F10`.

Each binding should carry:

1. a stable command id;
2. the key chord;
3. a short label for the bottom bar;
4. the command execution path;
5. an availability predicate when the command depends on active state or panel capabilities.

Use explicit names such as `FunctionKeyBinding`, `FunctionKeyLayer`, `FunctionKeyBarRenderer`, or similarly precise names. Avoid naming this model as a generic status bar or a generic menu if it only describes function-key commands.

### Modifier state

The application must track the currently pressed key-bar modifier state independently from completed key chords.

Rules:

1. Pressing `Alt` alone changes the bottom row to the `Alt` layer.
2. Pressing `Control` alone changes the bottom row to the `Control` layer.
3. Releasing the modifier restores the plain layer.
4. Modifier-only events must not execute commands.
5. Modified function keys still execute their command normally, for example `Alt+F7` runs search.
6. Repeated modifier events should not cause unnecessary full-screen redraws when the visible layer did not change.
7. If both `Alt` and `Control` are pressed, show the plain layer unless a real `Ctrl+Alt` function-key layer is introduced by a later spec.

The Windows console input path may need a new input event for modifier-state changes because the current logical input model mostly exposes completed `ConsoleKeyInfo` keys and skips key-up events. This should be added deliberately to the console input layer instead of inferring modifier state from arbitrary command keys.

### Rendering

Replace the static bottom-row labels with labels resolved from the active function-key layer.

Rendering rules:

1. Keep the existing Far-like number and label color split.
2. Fill the full bottom row background as today.
3. Show only commands that exist in the active layer.
4. For unavailable commands, either hide the label or render it with a disabled style; choose one behavior and keep it consistent.
5. Do not show a label for a command that the handler would reject in the same state.
6. Keep one-row rendering; no wrapping.
7. Clip cleanly on narrow terminals without ellipsis.
8. Do not resize panels or the command line when the key-bar layer changes.

Suggested plain layer:

```text
1Help 2UserMn 3View 4Edit 5Copy 6RenMov 7MkFold 8Delete 9ConfMn 10Quit
```

Suggested `Alt` layer:

```text
1Left 2Right 7Search 8History 11FHist 12DHist
```

Suggested `Control` layer:

```text
3SortNm 4SortExt 5SortTm 6SortSz
```

Labels may be shortened if the final renderer needs tighter text, but they must remain recognizable and consistent with the actual command.

### Command handling

Function-key command execution should use the same binding table that feeds the key bar, or a single command catalog that both the handler and renderer consume.

Rules:

1. Avoid duplicating the same key chord in separate switch blocks and renderer arrays.
2. Keep non-function-key commands such as `Ctrl+O`, `Ctrl+Q`, `Ctrl+A`, arrows, `Enter`, and command-line text routing outside this spec unless a small helper is needed to keep `HandleKey` readable.
3. Keep menu command ids reusable where they already exist, but do not force all panel commands through the top-menu command system if that makes the change larger than needed.
4. Preserve existing behavior for plain function keys and existing modified function-key commands.
5. Preserve the rule that `Alt+F7` is search and plain `F7` is create folder.
6. Preserve the rule that `F9` opens the top menu.

### Availability

The bar should reflect the same state checks as command execution.

Required examples:

1. On local filesystem panels, plain `F4`, `F5`, `F6`, `F7`, and `F8` keep their current behavior.
2. On search results panels, commands disabled by provider capabilities from spec `0012` must not be advertised as available.
3. `F4 Edit`, `F6 RenMov`, `F7 MkFold`, and `F8 Delete` should not look available when the active panel source is read-only and the command would be blocked.
4. `F5 Copy` from search results may remain available when the active source supports `CopyFrom` and the passive panel supports `CopyTo`.
5. `Ctrl+F3` through `Ctrl+F6` remain available only when the active panel can be sorted.

## Non-goals

Do not implement in this spec:

1. a full shortcut help system for every key combination;
2. dynamic hints for non-function-key commands such as arrows, `Tab`, `Ctrl+O`, `Ctrl+Q`, `Ctrl+A`, or `Ctrl+*`;
3. a plugin API for third-party commands;
4. a configurable keyboard layout or user-editable key bindings;
5. a `Shift` function-key layer unless there are real existing `Shift+F...` commands to show;
6. a `Ctrl+Alt` layer;
7. mouse hover help for the bottom bar;
8. compatibility wrappers for the old hard-coded renderer data.

## Acceptance criteria

- Plain bottom row still shows the current `F1` through `F10` actions.
- Holding `Alt` changes the bottom row to the `Alt` function-key layer.
- Releasing `Alt` restores the plain layer.
- Holding `Control` changes the bottom row to the `Control` function-key layer.
- Releasing `Control` restores the plain layer.
- `Alt+F7` still starts search and plain `F7` still creates a folder.
- `Alt+F1`, `Alt+F2`, `Alt+F8`, `Alt+F11`, and `Alt+F12` keep their current behavior.
- `Ctrl+F3`, `Ctrl+F4`, `Ctrl+F5`, and `Ctrl+F6` keep their current sorting behavior.
- The key bar is rendered from the same binding data that command dispatch uses.
- The key bar does not show file operation commands as available when provider capabilities would block them.
- Modifier-only key events do not insert command-line text and do not execute commands.
- Narrow terminal rendering stays one row and clips cleanly.
- Existing menu, command line, panel navigation, and dialog key handling keep their current behavior.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual checks:

1. Start the app in a real Windows console.
2. Hold and release `Alt`; verify the bottom row changes and returns without opening dialogs.
3. Hold and release `Control`; verify the bottom row changes and returns without running commands.
4. Press `Alt+F7`; verify search opens and plain `F7` still opens create-folder.
5. Press `Ctrl+F3` through `Ctrl+F6`; verify sorting still works and the row labels match the active layer.
6. Open a search results panel and verify disabled file operations are not advertised as available.
7. Resize the console to a narrow width and verify the bottom row remains a single clipped row.
