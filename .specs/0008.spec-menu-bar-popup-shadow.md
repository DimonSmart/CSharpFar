# 0008 - Menu bar, popup rendering, and shadows

Type: Spec
Status: Current

## Context

CSharpFar needs a top menu bar, drop-down menus, and shared popup/dialog
rendering behavior so modal UI elements have consistent layout, shadowing,
colors, hit testing, keyboard control, and command dispatch.

## Scope / Behavior

- `F9` activates the top menu bar.
- Mouse activation on the top menu bar opens the corresponding menu.
- Keyboard navigation supports moving between menu groups, moving through menu
  items, accepting a command, and canceling.
- Menu state is separate from command execution. Menu definitions provide
  command identifiers, labels, enabled state, and grouping.
- Executing a menu item dispatches through the same command path as equivalent
  hotkeys where such commands exist.
- Accepting an enabled menu item closes the open menu before dispatching the
  command, so commands that open dialogs are shown without the drop-down menu
  remaining open behind or above them.
- Drop-down menu layout clips or scrolls predictably when the screen is too
  small.
- The leftmost top-level menu group is `File`. Its first menu contains the
  commands `View`, `Edit`, and `Attributes`, and these items dispatch through
  the same command path as the equivalent file viewer, editor, and attributes
  commands.
- Popups and dialogs use a shared renderer for borders, titles, content
  background, and optional shadow.
- Popup shadow rendering never writes outside the console viewport.
- Overlay rendering restores the underlying UI after popup close.
- Cursor visibility and position are controlled so popups do not leave the
  cursor in an unexpected location.
- Menu colors and popup colors come from the active palette.
- The menu contains the product's main command groups and exposes available
  commands without requiring a separate implementation path.

## Non-goals

- This spec does not require implementing every Far Manager menu item.
- This spec does not require arbitrary nested submenus unless a later spec adds
  them.
- This spec does not require shadows in terminals where the configured palette
  cannot express them clearly.

## Acceptance criteria

- `F9` opens the menu bar and `Esc` closes it.
- Arrow keys and mouse input can choose menu groups and items.
- Accepting an enabled item invokes the corresponding command.
- The menu is closed before the accepted command is executed.
- If the accepted command opens another popup or dialog, that popup or dialog
  uses the main UI as its underlay; the previously open menu drop-down is not
  visible behind, above, or inside the new popup or dialog.
- Disabled items cannot be executed.
- The first top-level menu item is `File`, and opening it shows `View`, `Edit`,
  and `Attributes` commands.
- Popups and dialogs share border, title, color, and shadow behavior.
- Closing menus or popups redraws the underlying screen correctly.
- Rendering remains inside the viewport at narrow and short console sizes.

## Verification

- Menu definition tests or focused UI tests prove `File` is the first top-level
  menu group and contains `View`, `Edit`, and `Attributes`.
- Command routing tests prove those menu items invoke the same commands as the
  corresponding file viewer, editor, and attributes shortcuts.
- Controller tests prove an accepted enabled menu item closes the menu before
  command dispatch.
- Application-level menu command tests prove a dialog opened from a menu item
  does not capture the closed drop-down menu in its underlay.
