## Goal

Identify architectural cleanup opportunities in the current application: leftover
experimental code, unused files, duplicated techniques, misplaced responsibilities,
and classes that have grown beyond a predictable purpose.

## Hypothesis

The project has accumulated useful but uneven feature work. The most likely cleanup
points are in the application orchestration layer, repeated dialog/list interaction
logic, rendering helpers, and feature-specific services that may belong in shared
modules.

## Experiment

Review the solution structure, project dependencies, large classes, duplicate code
patterns, unused or suspicious files, and active worklog context. Prefer concrete
findings with file paths and small follow-up proposals over broad rewrites.

## Constraints

- Do not change production behavior during the spike.
- Do not remove files or refactor code as part of this review.
- Treat active worklog documents as current product intent.
- Keep recommendations small enough to be implemented and verified separately.

## Done when

- architecture hotspots are documented;
- likely dead or leftover files are listed or ruled out;
- duplication and misplaced responsibility candidates are identified;
- recommendations are prioritized with follow-up scope.

## Result

Static architecture review completed.

Findings:

1. `Application` is the main hotspot. It is about 3,200 lines and owns startup
   state, rendering, command dispatch, function-key binding creation, mouse
   handling, command-line completion, search-results panel behavior, file
   operation UI orchestration, current-console program launch, shell underlay
   handling, auto-refresh, and direct `System.Console` fallbacks. The class name
   no longer predicts the number of responsibilities inside it.
2. `FileOperationService` is the second main hotspot. It owns operation
   dispatch, copy-plan building, recursive traversal, conflict policy,
   copy/move/delete/create implementations, reparse-point handling,
   tail-validated resume integration, metadata preservation, and progress state.
   The copy path is especially dense.
3. Several dialog/list patterns are duplicated:
   - `HistoryDialog`, `DirectoryHistoryDialog`, `FileHistoryDialog`, and
     `UserMenuDialog` are nearly the same scrollable list dialog with different
     item projection and title.
   - `SearchDialog`, `FileOperationDialog`, and `SettingsDialog` each carry
     local scrollable-form state, input cursor math, scrollbar handling, and
     row rendering helpers.
   - text input rendering helpers are repeated in multiple dialogs.
4. The input model is split. Newer UI paths use `ConsoleInputEvent`, mouse
   events, resize events, and scrollbar handlers. Older viewers/simple dialogs
   still use key-only `ReadKey`, and `IConsoleDriver.ReadKey` is explicitly
   marked as kept for dialog compatibility. This is not dead code, but it is a
   second input path that can drift from resize/mouse-aware behavior.
5. `Application` still reaches directly into `System.Console` through local
   `SysConsole` wrappers and `TryReadConsoleKey`. Those calls are tied to
   current-console command execution and prompt cleanup, but the console driver
   is the more predictable owner for this behavior.
6. Mask groups are placed under `Core.Highlighting`, but the mask matcher,
   search service, copy filtering, settings, and highlighting all use the same
   `MaskGroup` concept. This makes file-mask infrastructure depend on a
   highlighting namespace even when the caller is not doing highlighting.
7. `FilePanelState` mixes local-panel state and search-result panel metadata.
   The current capability flags are useful, but search-specific properties
   (`SearchRequest`, `SearchWasCancelled`, full-path display, display title)
   make the model less predictable if another virtual source is added.
8. `FilePanelState.FilterMask` appears unused in source and tests.
9. `AppSettings.Ui.Theme` and `AppSettings.Shell.Kind` appear unused at runtime;
   `Shell.Executable`, `Shell.ArgumentsFormat`, `Ui.Palette`, and
   `Ui.ConfirmDelete` are the settings actually read by the app.
10. `tests/CSharpFar.Tests/UnitTest1.cs` is named like a template leftover while
    containing only generic smoke tests. It may still pass, but the file name and
    coverage value are weak.
11. Test doubles are duplicated across application-level tests, especially
    `NoOpShellService` and `NoOpFileOperationService`.
12. Text decoding is inconsistent between search and viewer/editor paths.
    `FileSystemSearchService` has explicit BOM/UTF-8/ANSI fallback logic, while
    `TextFileReader` falls back to `Encoding.Default`. This is a shared domain
    concern and should not live as two independent implementations.
13. The ignored `artifacts/` folder contains manual screenshots, publish output,
    and an input diagnostic log. These are not tracked by git, so they are local
    cleanup candidates rather than source cleanup.

## Recommendation

Recommended order:

1. Start with low-risk cleanup: remove unused `FilePanelState.FilterMask`,
   remove or wire `Ui.Theme` and `Shell.Kind`, and rename or delete the
   template-like `UnitTest1.cs` after moving any useful assertions.
2. Extract a reusable scrollable list dialog for command history, directory
   history, file history, and user menu.
3. Extract shared text input/form helpers for modal dialogs before adding more
   fields to `SearchDialog`, `FileOperationDialog`, or `SettingsDialog`.
4. Move current-console prompt/underlay/direct-console handling out of
   `Application` into a console-layer service or driver extension.
5. Split `FileOperationService` along real responsibilities:
   `CopyPlanBuilder`, copy conflict policy, file content copier, delete/move
   execution, metadata preservation, and progress state.
6. Move mask group contracts/default groups out of `Core.Highlighting` into the
   file-mask area, then make highlighting consume them.
7. Before adding another virtual panel source, replace search-specific fields on
   `FilePanelState` with a small source/location model.
8. Unify text decoding/BOM/binary detection into one service used by search,
   viewer, editor, and quick view.

## Follow-up

- Create one cleanup spec for unused settings/state/test leftovers.
- Create one refactor spec for scrollable list dialogs.
- Create one refactor spec for modal form helpers.
- Create an ADR or spike before changing the panel source model.
- Verification during this spike was static review only; no production code was
