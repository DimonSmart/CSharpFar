# 0066 - Folder and file-set comparison

Related:
- 0001
- 0008
- 0010

## Context

CSharpFar needs read-only comparison commands for two local folder trees shown
in the left and right panels. Users need two different workflows:

- folder structure comparison, where files match by relative path from the two
  selected roots;
- file-set comparison, where files match by name-derived keys while folder
  structure is ignored.

The visible result of either comparison is normal item selection in the
existing left and right panels.

The commands do not open a separate result panel, result table, or detailed
comparison dialog. After the comparison, CSharpFar shows one informational
summary dialog and leaves differing items selected in the panels.

## Scope / Behavior

- The application exposes commands named `Compare folders` and `Compare file
  sets` from the main UI.
- Both commands compare the local left panel with the local right panel,
  regardless of which panel is active.
- The terms `left` and `right` refer to panel position, not active/passive
  status.
- If either side is not a local directory, the command shows a clear error and
  does not start the comparison.
- Comparison is read-only. It never copies, deletes, moves, synchronizes, or
  modifies files.
- The comparison engine is independent from Far UI controls and is directly
  unit-testable.
- The engine returns structured comparison results internally, but these
  results are used only to calculate panel selection and the final summary.
- No detailed comparison-result view is opened.
- Panel locations, contents, sort order, cursor positions, and scroll positions
  remain unchanged after comparison.
- Existing selections are snapshotted before comparison because they may be
  used as scan input.
- After comparison finishes, previous selections in both panels are replaced
  by the comparison-result selection.
- If the comparison is cancelled or fails before producing a valid result,
  existing panel selections remain unchanged.
- Both commands support recursive scanning, selected-items-only scanning,
  maximum depth, include masks, exclude masks, and name comparison mode.
- Depth `0` means files directly in the scan root only. Depth `1` means root
  files plus files in immediate child folders. Unlimited depth means full
  recursive traversal.
- Include masks decide which files are compared. Exclude masks skip matching
  files and prevent scanning matching directories. Exclude wins over include.
- Masks are semicolon-separated and are applied to relative paths where
  practical.
- Name comparison can use system default, case-sensitive, or case-insensitive
  matching. System default is case-insensitive on Windows and case-sensitive on
  Unix-like hosts.
- Directory symbolic links are not recursively followed by default. File
  symbolic links may be read as files. Unsupported or unreadable entries are
  represented as errors.

Folder structure mode:

- Files match by normalized relative path from the selected root.
- Supported statuses are `Equal`, `Different`, `Left only`, `Right only`, and
  `Error`.
- Fast comparison compares file size and last-write time using the selected
  timestamp tolerance and does not read file content.
- Content comparison compares size first, then compares streams byte by byte in
  chunks. It does not load complete files into memory.
- Timestamp tolerance values are `Exact`, `2 seconds`, and `1 hour`. Content
  comparison treats content equality as equality and does not use timestamp
  tolerance.

File-set mode:

- Files match independently of relative path by one of these strategies:
  `File name`, `File name + size`, or `File name + content hash`.
- `File name` is the default file-set strategy.
- Content-hash matching uses streaming SHA-256 hashing and does not load
  complete files into memory.
- Supported statuses are `Equal`, `Different`, `Left only`, `Right only`,
  `Ambiguous`, `Duplicate`, and `Error`.
- If exactly one matching file exists on each side, the engine compares those
  files.
- If multiple matching files exist on one or both sides and the selected
  strategy cannot safely decide pairs, the result is ambiguous or duplicate.
  The engine must not silently choose an arbitrary duplicate.
- Content-only matching of files with different names is outside the first
  version.

Selected-items-only behavior:

- When disabled, each side scans the current panel directory.
- When enabled and one side has selected items, that side scans its selected
  items and the other side scans corresponding item names when they exist, or
  its current directory when corresponding roots cannot be resolved.
- When enabled and both sides have selected items, each side scans selected
  roots independently.
- A selected file is included as one file. A selected directory contributes
  files according to recursion and depth options.
- Unresolvable selected items are reported clearly instead of crashing the
  command.

## Panel selection result

Comparison results are represented using the normal selection state of the
existing panels.

For files directly displayed in the current panels:

- `Equal` selects neither side.
- `Left only` selects the file in the left panel.
- `Right only` selects the file in the right panel.
- `Different` selects the corresponding files in both panels.
- `Error` selects the affected file on the side where the error occurred.
- `Ambiguous` and `Duplicate` select all involved files that are directly
  displayed in either panel.

For recursive results:

- A nested file cannot be selected directly when it is not currently displayed
  in the panel.
- In this case, the nearest currently displayed ancestor item is selected.
- Normally this is the first-level directory below the current panel root.
- If comparison started from an explicitly selected file or directory, that
  selected root is used as the visible marker when appropriate.
- If several differences map to the same visible directory, the directory is
  selected only once.
- A directory selected by comparison means that one or more differences or
  errors exist somewhere below that directory.
- Comparison-generated directory markers may be selected even when the manual
  `Select folders` option is disabled.
- The parent-directory entry (`..`) is never selected.

Applying comparison selection must:

- clear previous result selection on both panels;
- populate both `SelectedPaths` and `SelectedLocations` consistently;
- update each panel's selected-item count and selected-file-size summary;
- preserve panel cursor and scroll positions;
- request one normal panel redraw.

Selection only indicates that an item contains or represents a non-equal
result. Different result categories do not use separate colors in the first
version.

## Summary dialog

After a successful comparison, the application shows one informational dialog
with an `OK` button.

The dialog does not contain an item list and does not allow opening individual
comparison rows.

For a comparison containing differences, the summary includes applicable
lines in this form:

```text
Matched: 110
Left only: 5 (100 KB)
Right only: 5 (300 KB)
Different: 23 (left 100 KB, right 500 KB)
Ambiguous or duplicate: 2 groups
Errors: 1
```

If every result is equal, the dialog clearly states that no differences were
found. The summary may also report elapsed time and compared-file totals.

## Comparison options dialog

Each comparison command opens an options dialog titled for that command. The
dialog shows the left and right panel paths so the user can confirm the
comparison context before starting it.

Options are arranged in the following visual sections:

- `Scan`: include subfolders, selected items only, and `Depth`;
- `Filters`: include and exclude masks, with a visible semicolon-separated-mask
  hint;
- `Comparison`: the comparison method, applicable timestamp tolerance, name
  comparison, and, only for file-set comparison, the file-set matching method.

The dialog does not repeat the command mode in its content. It uses the same
dialog-button style for `Compare` and `Cancel`. `Timestamp tolerance` is shown
only for fast comparison; content comparison does not present an inapplicable
tolerance control. A custom depth value is shown directly below `Depth` only
when the `Custom` depth option is selected.

## Non-goals

- Folder synchronization.
- Copy, delete, update, rename, or repair actions from comparison results.
- A full visual text diff viewer.
- A one-sided duplicate finder.
- Content-only matching of files with different names.
- Legacy-style independent checkboxes whose combined comparison semantics are
  unclear.

## Acceptance criteria

- A user can start `Compare folders` and `Compare file sets` from the UI when
  both panels point to local directories.
- Folder structure comparison reports equal, different, left-only, right-only,
  and per-file error rows.
- File-set comparison ignores folder structure and reports equal, different,
  left-only, right-only, ambiguous or duplicate, and error rows.
- Fast comparison uses size and modified time only.
- Content comparison reads equal-size files in chunks and detects byte
  differences.
- Recursive scanning, selected-items-only scanning, maximum depth, include
  masks, and exclude masks affect the scan result.
- Duplicate file-set keys are never silently paired.
- Per-file errors are included in the result model and do not fail the whole
  comparison.
- A successful comparison does not open a detailed result view.
- A successful comparison replaces the prior selections with selection markers
  for non-equal results and shows one summary dialog.
- The selection result maps direct files and recursive differences to visible
  files or ancestor directories as specified, never selecting `..`.
- Applying the result preserves panel location, items, sorting, cursor, and
  scroll state, and keeps `SelectedPaths`, `SelectedLocations`, and the panel
  summary consistent.
- Cancelled or failed comparisons leave both panel selections unchanged.
- The comparison options dialog shows panel context, grouped scan, filter, and
  comparison options, and hides timestamp tolerance for content comparison.
- Core comparison behavior is covered by unit tests.

## Verification

- Unit tests create temporary folders and verify result statuses and keys for
  folder structure comparison, file-set comparison, masks, depth, timestamp
  tolerance, duplicates, content hashing, errors, and case sensitivity.
- Unit tests verify comparison-result selection for direct files, nested files,
  selected roots, duplicates, and errors, including state preservation and
  selection-summary consistency.
- Manually verify that the options dialog groups controls, shows both panel
  paths, and updates visible controls when comparison method or depth changes.
- Build and test the solution after implementation.
