# 0022 - Hidden console history scroll

Type: Spec
Status: Current

- none

## Goal

When panels are hidden through `Ctrl+O`, the user can scroll the console viewport up to
inspect older command output. CSharpFar must not immediately pull the viewport back down
just because only the viewport origin changed.

When the user resumes interaction with CSharpFar, the application returns the visible
viewport to the bottom of the console buffer and keeps the existing command-line and panel
rendering behavior.

In VT-supported interactive terminals, CSharpFar separates full-screen UI from shell
history by using the terminal alternate screen buffer for application UI and the main
screen buffer for command output. This is the preferred model because it matches common
terminal behavior across Windows Terminal, conhost with VT support, Linux, and macOS
terminals, and keeps future Unix-like portability from depending on Windows-only console
screen-buffer resizing.

## Context

Viewport-bound rendering from `0016` correctly treats `Left`, `Top`, `Width`, and `Height`
as part of the physical rendering target. That is required while panels are visible.

In hidden-panel mode, however, a user scrolls console history by changing the visible
viewport origin. Treating that origin-only change as a redraw trigger immediately restores
the hidden screen and redraws the command line, making old shell output hard to inspect.

The previous Windows-only scrollback implementation remains only a fallback for terminals
where VT alternate screen mode is unavailable. A VT-supported session must not use manual
console buffer resizing as the primary way to separate CSharpFar UI from shell output.

## Scope

- Distinguish hidden-panel origin-only viewport moves from real width/height changes.
- Add an application-level terminal screen mode abstraction that can enter and leave
  the alternate screen buffer without making the renderer decide which screen buffer is
  active.
- In VT-supported terminals, use `ESC[?1049h` for application screen mode whenever at
  least one panel is visible, including partially hidden panel layouts.
- In VT-supported terminals, use `ESC[?1049l` for main screen mode when both panels are
  hidden, before launching any external command, and during terminal restoration.
- When CSharpFar explicitly enters both-panels-hidden mode in a VT-supported terminal,
  it leaves alternate screen mode, scrolls the main screen buffer viewport to the bottom,
  captures the current visible main screen buffer into the renderer state, updates the
  cached viewport, shows the cursor, and then renders only the command line.
- The main-buffer scroll-to-bottom action belongs to explicit state transitions: entering
  both-panels-hidden mode, preparing an external command, and user input after hidden
  scrollback inspection. Ordinary command-line-only render, resize handling, and
  origin-only viewport movement must not pull the viewport down by themselves.
- Pure modifier-key state changes such as pressing or releasing only `Ctrl`, `Shift`,
  or `Alt` are not command input in both-panels-hidden mode. They must not scroll the
  main screen buffer, edit the command line, or trigger command-line-only rendering.
- When real keyboard or mouse input resumes CSharpFar interaction after hidden-mode
  scrollback inspection, the application scrolls the main screen buffer viewport to the
  bottom and synchronizes the renderer with the current visible main screen viewport
  before rendering the command line.
- After every VT-supported transition from alternate screen mode to main screen mode,
  CSharpFar must synchronize the renderer with the current visible main screen viewport
  before the first renderer-based draw. This synchronization is a capture-only operation:
  it must not clear the viewport and must not restore an older underlay over the main
  screen buffer.
- Do not emit VT screen-mode escape sequences when output is redirected or VT output
  mode cannot be enabled.
- Restore the terminal on normal exit, exceptions, and Ctrl+C handling paths by leaving
  alternate screen mode, showing the cursor, resetting attributes, and restoring original
  Windows console modes when possible.
- Leave origin-only viewport moves alone while panels are hidden.
- Scroll the viewport back to the bottom before ordinary hidden-mode input edits the
  command line.
- Scroll the viewport back to the bottom before returning from hidden mode to panel mode.
- Add a console-driver API for explicit bottom scrolling.
- Cover the behavior with fake-driver unit tests.

## Non-goals

- Do not pin the CSharpFar command line while the user is viewing old output.
- Do not change visible-panel resize behavior.
- Do not add a new scrollback UI or a custom command-output viewer.
- Do not emulate Windows console history scrolling on non-Windows platforms.
- Do not use WinAPI `CreateConsoleScreenBuffer` as the primary implementation path.
- Do not remove the legacy scrollback/buffer-size path while it is still needed as the
  unsupported-terminal fallback.

## Acceptance criteria

- Hidden panels plus origin-only viewport movement does not redraw the command line or
  restore stale underlay.
- Pressing or releasing only `Ctrl`, `Shift`, or `Alt` while both panels are hidden and
  the user is viewing scrollback leaves the viewport and visible content unchanged.
- In VT-supported terminals, visible panels, menus, dialogs, viewer, editor, and other
  full-screen UI render in the alternate screen buffer and do not enter main-buffer
  scrollback.
- In VT-supported terminals, both-panels-hidden mode leaves the alternate screen buffer
  and keeps shell output scrollback available in the main screen buffer.
- The first `Ctrl+O` after application startup shows the hidden-mode command line
  immediately and leaves the main screen buffer viewport at the bottom.
- After `hidden panels -> command output -> show panels -> hide panels`, the visible
  main screen command output remains visible except where CSharpFar redraws its command
  line; command-line-only rendering must not replace the rest of the viewport with
  blanks.
- In VT-supported terminals, partially hidden panels remain in alternate screen mode.
- A transition from partial panel visibility to both-panels-hidden mode uses the same
  main-buffer scroll-to-bottom behavior as `Ctrl+O`.
- Before every current-console external command launch, CSharpFar leaves alternate screen
  mode and scrolls the main screen buffer viewport to the bottom so the child process
  writes to the visible bottom of the main screen buffer. The renderer is synchronized
  with that visible main screen viewport before CSharpFar writes the command prompt.
- After a current-console external command exits, CSharpFar re-enters alternate screen
  mode only when panels should be visible, otherwise it remains in the main screen buffer.
- In VT-supported terminals, legacy console scrollback resizing is not used for
  visible/hidden panel transitions.
- In VT-unsupported or redirected-output environments, no screen-mode escape sequences
  are emitted and the existing legacy scrollback behavior remains the fallback.
- Repeated terminal restoration calls are safe.
- Hidden panels plus actual size change still redraws the hidden command line.
- After hidden scroll up, ordinary command input returns the viewport to bottom and edits
  the command line without blanking the visible main screen buffer.
- After hidden scroll up, `Ctrl+O` returns the viewport to bottom and redraws panels.
- Visible panels still redraw on viewport-origin changes.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual check:

- Run app in a real Windows console.
- Execute enough output to create scrollback.
- Press `Ctrl+O`, scroll up, and verify the viewport stays up.
- Press a character and verify the viewport returns down and edits the command line.
- Scroll up again, press `Ctrl+O`, and verify panels return at the bottom.
- In Windows Terminal or a VT-capable conhost, show panels, hide panels, scroll up, and
  verify old CSharpFar panels are not visible in main-buffer scrollback.
- Run a long-output command while panels are hidden and verify the output remains
  scrollable.
- Run a command while panels are visible and verify command output appears in main-buffer
  history, then panels redraw cleanly afterward.
- Exit from visible panels and verify the shell returns with visible cursor and reset
  attributes.
