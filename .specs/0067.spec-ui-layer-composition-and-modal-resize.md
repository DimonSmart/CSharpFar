# 0067 - UI layer composition and modal resize

Type: Spec
Status: Current

Related:

- 0004

## Goal

Application-owned console UI is rendered through one shared composition host so
that a viewport change redraws the active full-screen surface and every modal
overlay as one current frame. Closing a layer rebuilds the UI from state rather
than restoring pixels captured from an earlier viewport.

## Layer model

- A **surface** is an opaque full-screen UI: the main application screen,
  hidden-panels command-line surface, viewer, editor, or full-screen help.
- Surfaces and overlays share one ordered layer stack. The last surface in that
  stack is active; earlier surfaces and every overlay below that active surface
  are not drawn.
- An **overlay** is drawn only when it is above the active surface in the same
  stack. Eligible overlays are rendered in stack order and support nesting.
- Surface and overlay lifetimes are explicit disposable LIFO scopes. Repeated
  disposal is safe; invalid out-of-order disposal is rejected deterministically.

## Composition and frames

- The application owns exactly one shared `UiCompositionHost` and one shared
  `ModalDialogHost`; they are created during bootstrap and passed through normal
  dependencies. UI components do not use a static service locator.
- `UiCompositionHost` is the only owner of the render-until-stable retry loop.
  It snapshots the active surface and eligible overlays before opening a frame,
  opens the active surface frame, renders that surface followed by the snapshot
  overlays, retries the whole composition when the frame is interrupted, drains
  stale resize events without losing ordinary input, and records the final
  stable viewport. Layers cannot be changed while that snapshot is rendering.
- All layer layout uses `UiRenderContext.Viewport` and `UiRenderContext.Size`
  from the already opened frame. Layer renderers do not open nested frames or
  reread console size for the current layout.
- The topmost rendered layer determines cursor state. On overlay close the
  composition redraw restores the cursor from the remaining top layer.

## Modal behavior

- Ordinary modal dialogs, progress dialogs, popup restoration, and nested
  dialogs must use the shared `ModalDialogHost`.
- A modal session renders the full composition. Its input methods consume
  `ConsoleResizeInputEvent`, perform a resize-recovery composition render, and
  return only dialog-relevant input. A viewport change detected without an
  explicit resize event is recovered before returning user input.
- Dialogs own only dialog state, current-frame layout, rendering of their own
  layer, input handling, and their result. They neither render panels/viewer/
  editor nor accept `RenderUnderlay` callbacks. Production dialog APIs accept
  the shared `ModalDialogHost`; raw-screen constructors, `Show(ScreenRenderer)`
  paths, and parallel legacy dialog lifecycles are not retained.

## Surface behavior

- The main application surface encapsulates normal panels and hidden-panels
  command-line mode. It preserves the shell-underlay, terminal-screen,
  bottom-pin, and scroll-position rules of 0004.
- Editor, viewer, and full-screen help are temporary surfaces. Their dialogs
  use the shared modal stack. Disposing a temporary surface redraws the prior
  active surface.

## Snapshot policy

`ScreenRenderer.Capture` and `Restore` remain low-level operations for shell
underlay, external-console/child-process interoperability, terminal-screen
mode, renderer tests, and public compatibility APIs. They must not be used to
restore application-owned modal dialogs, progress dialogs, editor, viewer,
help, popups, or nested dialog UI.

## Acceptance criteria

- Resizing while a dialog is over panels, hidden command line, editor, viewer,
  progress UI, or another dialog redraws every visible layer using the final
  viewport.
- Resize events never reach normal dialog-specific input handlers.
- Removing an overlay or temporary surface leaves no stale cells and redraws
  the correct lower composition.
- Main application, editor, viewer, and help do not run independent stable
  render loops.
- Modal APIs have no dialog-specific underlay rendering callback.
- Reusable, application, progress, editor, viewer, help, and built-in module
  dialogs all use the shared modal or temporary-surface lifecycle.
- Hidden-panel shell output and bottom-pin behavior remain governed by 0004.

## Verification

Run:

```bash
dotnet build
dotnet test
```

Focused tests cover composition ordering and LIFO lifetimes, interrupted-frame
retry, resize-event draining without input loss, modal resize recovery,
nested-modal restoration, and panels/dialog resize integration for both larger
and smaller viewports.
