# 0031 SPEC Large File Viewer

Type: spec

Replaces:
- none

Related:
- 0014
- 0016
- 0020

## Goal

Make F3 viewing usable for large, growing, and binary files without reading the whole file
into memory before the viewer opens.

## Context

`FileViewer` currently reads text through `TextFileReader.ReadLines(filePath)`, so it
loads all lines into a `string[]` before rendering and rejects files larger than the
shared 10 MB limit. `TextFileReader` also reads the file into a full in-memory list.

This blocks viewing large logs and makes memory use proportional to file size instead of
the visible screen area and a bounded cache. Quick View remains a small preview surface
from `0014`; full F3 viewing needs a separate large-file path.

## Scope

- Add a large-file viewer path under `CSharpFar.App.Viewer`.
- Read file bytes by offset through an `IFileByteReader` implementation based on
  `RandomAccess.ReadAsync`.
- Open files with sharing suitable for active logs:
  `FileShare.ReadWrite | FileShare.Delete`.
- Add a bounded block cache with recent-block reuse.
- Decode visible text lazily, supporting UTF-8, UTF-16 little endian, UTF-16 big endian,
  and the system ANSI encoding for non-UTF-8 files.
- Tolerate damaged text by using replacement fallback instead of failing the viewer.
- Render only visible lines and horizontal slices.
- Keep vertical navigation responsive for start, end, page movement, line movement, and
  percent/line-oriented jumps where existing UI commands expose them.
- Preserve horizontal offset while the viewer remains open.
- Add a follow mode for growing files. Follow stays active at the end of the file and is
  disabled when the user scrolls away.
- Keep memory bounded by screen height, block size, and cache capacity.
- Keep small files on the existing `FileViewer` path unless that would duplicate large
  file behavior.

## Non-goals

- Editing files in the viewer.
- Search inside the large-file viewer.
- Syntax highlighting.
- Persisting viewer positions between application runs.
- A full binary editor. Binary display may be read-only hex/text-safe output.
- Quick View scrolling or unlimited Quick View previews.
- Background full-file indexing as a required MVP feature.
- Compatibility shims for the old 10 MB F3 limit.

## Acceptance criteria

- F3 can open a file larger than 10 MB.
- Opening a large text file does not call `TextFileReader.ReadLines`.
- Opening the end of a large file reads only bounded blocks near the tail.
- Up, Down, PageUp, PageDown, Home, End, horizontal scrolling, and F10 work consistently
  in the small and large viewer paths.
- Follow mode appends newly written lines when the file grows and the viewer is at EOF.
- Scrolling away from EOF disables follow mode.
- Binary or damaged text does not crash the viewer.
- Memory use is bounded by cache settings and visible rows, not by file size.
- Quick View keeps its bounded preview behavior.

## Outcome

Implementation summary:
- Added the large-file F3 path under `CSharpFar.App.Viewer`.
- Added `IFileByteReader` and `RandomAccessFileByteReader` using offset-based
  `RandomAccess.ReadAsync` and `FileShare.ReadWrite | FileShare.Delete`.
- Added `BlockCache` with fixed block size and LRU eviction.
- Added `LineScanner` for lazy visible-line decoding, BOM detection, UTF-8 validation,
  UTF-16 LE/BE BOM handling, ANSI fallback, damaged-byte replacement, tail scanning, and
  line-number seeking through `SparseLineIndex`.
- Added `LargeFileViewerState` and `LargeFileViewer` with bounded rendering, horizontal
  scroll, Home/End/Page/line movement, `G` line/percent jumps, `F` follow mode, and a
  binary hex display.
- Changed `FileViewer` so files above the small-file threshold are delegated to
  `LargeFileViewer` instead of being rejected by `TextFileReader.MaxFileSizeBytes`.
- Kept `TextFileReader.MaxFileSizeBytes` for existing bounded preview/editor uses.
- Added tests for files above the old 10 MB limit, tail opening, binary hex mode, follow
  status, UTF-16 BOM scanning, and damaged text.
- Documented the F3 large-file path and Quick View limitation in `README.md`.

Deviations:
- The MVP uses synchronous viewer control flow around async read primitives because the
  current application command/viewer stack is synchronous.
- Follow mode is implemented by polling while the viewer is idle; it updates to the tail
  when file length changes and disables on ordinary vertical movement.
- There is no background full-file indexer. Line-number jumps build the sparse index lazily
  from the nearest known checkpoint.
- ANSI fallback follows the repository's existing `Encoding.Default` behavior.
- `TextFileReader.MaxFileSizeBytes` was not moved into user configuration in this pass;
  the F3 hard stop is removed, while existing bounded preview/editor checks keep using the
  current constant.

Follow-up:
- Add a dedicated manual QA pass for appending to a live log in a real Windows console.
- Consider a background indexer if repeated jumps to far line numbers become common.
- Add search inside the large-file viewer as a separate feature.
