# 0041 SPEC Native FarNet domain model

Type: spec

Replaces:
- 0037
- 0039
- 0040

Related:
- 0029
- 0035

## Goal

Make the FarNet-like module model the native extension model inside CSharpFar.
External FarNet DLLs and built-in features should register the same domain
objects: module actions, tools, commands, panels, explorers, module managers,
and host API services.

## Context

Specs `0039` and `0040` added a `CSharpFar.Plugin.FarNet` adapter on top of the
existing `ICSharpFarPlugin` model. That proved binary binding to CSharpFar's
`FarNet.dll` shim, but it inverted the intended architecture: FarNet modules were
loaded as a plugin inside CSharpFar's own plugin abstraction.

The desired direction is different. CSharpFar's plugin/domain model should be
FarNet-like at the host level. The host should use a native module registry and
call `ModuleTool`, `ModuleCommand`, `Panel`, and `Explorer` objects directly.
Existing SFTP and FTP/FTPS plugins may stay statically referenced for now, but
their public integration should move toward this module model instead of
remaining the primary domain shape.

## Scope

- Add a native FarNet module registry and host services used by the app.
- Move FarNet DLL discovery/loading out of the user-visible `ICSharpFarPlugin`
  catalog and into app composition.
- Project registered `ModuleTool` actions directly into the top `Plugins` menu
  and disk menus.
- Dispatch selected FarNet module actions directly from the application host,
  not through `CSharpFar.Plugin.FarNet` as a proxy plugin.
- Keep the `FarNet.dll` shim identity and real-FarNet-DLL binding behavior from
  `0040`.
- Keep current SFTP/FTP `ICSharpFarPlugin` implementations working during this
  migration.
- Add compatibility tests proving that the FarNet loader is not exposed as a
  CSharpFar plugin item but still loads and runs real FarNet modules.

## Non-goals

- Removing `ICSharpFarPlugin` entirely in this slice.
- Rewriting SFTP and FTP/FTPS implementations to `ModuleTool + Panel/Explorer`
  in this slice.
- Full FarNet API compatibility.
- Loading original `FarNet.dll` as the host.
- Adding a security sandbox for modules.

## Plan

1. Retire the active specs that describe the old proxy-plugin direction.
2. Split the FarNet loader into a native module service with registry,
   diagnostics, command prefixes, and panel adapter behavior.
3. Remove `FarNetPlugin` from the app's plugin list and wire the module service
   into `PluginManager`/menu/disk-menu projection at the host level.
4. Keep a small bridge only where the current app still expects menu items and
   disk selection commands, but make the bridge app-owned rather than a plugin
   pretending to be user code.
5. Preserve existing loader tests and add tests for native projection.
6. Run `dotnet build` and `dotnet test`.

## Pause points

- Stop if native host integration requires rewriting SFTP/FTP in the same pass.
- Stop if app-side projection would need `CSharpFar.App` references from the
  FarNet shim assembly.

## Acceptance criteria

- `Program.cs` does not register FarNet loading as an `ICSharpFarPlugin`.
- FarNet module actions are projected into menus by app-owned host code.
- Real FarNet DLL loading tests still pass.
- Existing SFTP/FTP plugin behavior remains available through the current
  plugin manager.
- FarNet diagnostics remain user-visible.
- No `CSharpFar.App` reference is added to `CSharpFar.FarNetCompat`.

## Outcome

Implementation summary:
- Retired `0037`, `0039`, and `0040` because their active direction treated
  FarNet as a separate plugin/proxy layer instead of the native extension model.
- Replaced the user-visible `FarNetPlugin : ICSharpFarPlugin` shape with
  `FarNetModuleHost`, `FarNetModuleHostOptions`,
  `FarNetModuleHostServices`, `FarNetModuleMenuItem`, and
  `FarNetModuleOpenResult`.
- `Program.cs` now passes `new FarNetModuleHost()` separately to
  `Application`; the plugin list contains only the current SFTP and FTP/FTPS
  plugins.
- `Application` initializes FarNet host services directly and projects FarNet
  module actions into the top `Plugins` menu and disk menus using app-owned
  `ModuleMenuProjection` and `ModuleOpenCommand`.
- Disk selection now supports `VolumeSelectionAction.OpenModule` for native
  FarNet disk-menu actions alongside existing local volumes and SFTP/FTP plugin
  items.
- FarNet tests now call `FarNetModuleHost.OpenFromPluginMenu`,
  `OpenFromDiskMenu`, and `OpenFromCommandLine` directly instead of wrapping
  calls in `PluginOpenInfo`.
- Added app-level coverage proving native module menu projection and native
  FarNet disk action dispatch.
- Recorded the durable correction in `AGENTS.local.md`: FarNet compatibility is
  the native extension domain model, not a proxy plugin layer.

Deviations:
- The physical project and assembly are still named
  `CSharpFar.Plugin.FarNet` in this slice. The runtime/API shape is no longer an
  `ICSharpFarPlugin`, and `Program.cs` no longer registers it as a plugin.
  A broad scripted repo rename was rejected as too risky by the tool approval
  reviewer, so the project/folder rename should be done later as an explicit
  narrow file-move change.
- SFTP and FTP/FTPS remain `ICSharpFarPlugin` implementations, as allowed by
  this spec. They were not rewritten to `ModuleTool + Panel/Explorer`.

Follow-up:
- Rename the FarNet host project/folder/namespace from
  `CSharpFar.Plugin.FarNet` to a native name such as `CSharpFar.FarNetHost` or
  `CSharpFar.FarNet` using explicit file moves.
- Convert SFTP and FTP/FTPS to native `ModuleTool + Panel/Explorer` modules in
  a separate slice, then retire the old `ICSharpFarPlugin` abstraction when no
  built-in code depends on it.
- Decide whether FarNet panel tools should become enabled by default after
  manual sample-module smoke testing.
