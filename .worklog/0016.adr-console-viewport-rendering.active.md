# 0016 ADR Viewport-bound console rendering

Type: adr

Replaces:
- 0015

Related:
- 0002
- 0013

## Goal

Зафиксировать решение по рендерингу консольного UI при resize: кадр должен быть
привязан к конкретному visible viewport консоли, а не только к ширине и высоте окна.

Решение должно сохранять buffered rendering из `0002`, убрать повреждение нижней строки
функциональных клавиш и не возвращать мигание полного direct redraw.

## Context

Буферизация из `0002` убрала обычное мигание: приложение строит полный логический кадр в
back buffer, а `ScreenRenderer` отправляет в консоль только изменившиеся горизонтальные
участки. Эта модель остается правильной, но Windows console resize добавил отдельный
класс артефактов.

При непрерывном изменении размера visible viewport (`srWindow`) может двигаться внутри
screen buffer прямо во время рендера. Ширина и высота не описывают физическую область
экрана полностью. Два viewport с одинаковым размером, но разными `Top` или `Left`,
показывают разные строки и столбцы буфера.

Обнаруженные проблемы:
- записи, основанные на устаревших `Console.WindowTop` / `Console.WindowLeft`, могли
  попадать вне текущего visible window и заставлять Windows скроллить buffer;
- постановка курсора после frame flush могла скроллить viewport даже тогда, когда сами
  записи были безопасны;
- interrupted frame мог все еще применить pending cursor position;
- `front buffer` считался валидным только по размеру, поэтому origin-only сдвиг viewport
  мог оставить renderer в уверенности, что нижняя строка уже корректна;
- ожидание стабилизации размера уменьшало часть мигания, но не решало сдвиги
  `srWindow.Top` / `srWindow.Left` при той же ширине и высоте.

Нижняя строка функциональных клавиш страдала особенно заметно, потому что она находится
на последней видимой строке. Любой scroll на одну строку или stale front buffer сначала
ломает именно ее.

## Options considered

### Оставить size-based buffering и дольше ждать после resize

Pros:
- минимальное изменение кода;
- все UI-компоненты продолжают рисовать только логические `x, y`.

Cons:
- origin viewport не моделируется;
- изменение `srWindow.Top` / `srWindow.Left` при том же размере не обнаруживается;
- корректность зависит от timing;
- UI становится менее отзывчивым во время resize.

### После каждого resize очищать и полностью перерисовывать консоль напрямую

Pros:
- простая модель после события resize;
- может скрыть часть stale-buffer артефактов.

Cons:
- возвращает лишние записи и визуальное мигание, от которых уходили в `0002`;
- resize может прийти во время уже идущего flush;
- direct redraw все равно не решает безопасную постановку курсора;
- resize-specific поведение начинает протекать в верхнеуровневый UI.

### Маскировать проблему console mode настройками или cursor visibility

Pros:
- маленький diff, если конкретный terminal случайно ведет себя лучше;
- можно быстро проверить вручную.

Cons:
- `ENABLE_WRAP_AT_EOL_OUTPUT` уже управляется для UI-rendering и не был оставшейся
  причиной;
- скрытие курсора не равно безопасной постановке курсора;
- stale absolute coordinates остаются stale coordinates;
- поведение становится зависимым от реализации terminal.

### Привязать каждый frame к полному viewport snapshot

Pros:
- frame описывает реальную физическую цель: `Left`, `Top`, `Width`, `Height`;
- origin-only сдвиги становятся видимыми для invalidation logic;
- buffered rendering из `0002` остается основой;
- физические записи и cursor placement не применяются к viewport, отличному от frame
  snapshot;
- resize handling остается централизованным в console rendering layer.

Cons:
- `IConsoleDriver` становится более явным и получает viewport-aware операции;
- во время активного resize больше frames намеренно отбрасываются и повторяются;
- fake console в тестах должен моделировать viewport, а не только size.

## Decision

Использовать viewport-bound buffered rendering.

`IConsoleDriver` предоставляет `GetViewport()`, который возвращает `ConsoleViewport` с
`Left`, `Top`, `Width`, `Height`. На Windows snapshot берется из
`GetConsoleScreenBufferInfo.srWindow`. `GetSize()` остается как сокращение для layout-кода,
которому нужны только width/height.

`ScreenRenderer.BeginFrame()` захватывает один viewport snapshot для всего frame.
`front buffer` валиден только для этого же viewport. Если изменились width, height, left
или top, known front buffer инвалидируется, а следующий успешный frame отправляется как
full redraw.

`FlushFrame()` перед строками проверяет, что current viewport совпадает с frame viewport,
использует viewport-bound physical writes и перед commit back buffer в front buffer делает
финальную проверку. Если viewport изменился, frame помечается interrupted, front buffer
сбрасывается, а caller запускает render заново.

Pending cursor position применяется только после успешного flush. Cursor placement тоже
viewport-bound: абсолютная позиция вычисляется относительно frame viewport, а операция
пропускается, если current viewport уже другой. Это убирает cursor-induced scroll,
который портил последнюю строку.

`Application` рисует видимый UI через `RenderUntilStable()`. Метод повторяет interrupted
frames и после успешного render вычитывает stale resize events, сохраняя non-resize input
для диалогов и command handling.

Старые отдельные `GetViewportOrigin()` и `SetCursorPositionAbsolute()` не оставлены в
`IConsoleDriver`. Два параллельных способа работать с координатами сделали бы контракт
неожиданным и оставили бы обходной путь мимо viewport-bound checks.

## Consequences

Positive:
- frame buffer теперь описывает конкретный visible viewport, а не только размер;
- origin-only scroll заставляет renderer сделать full redraw, поэтому function-key bar
  перерисовывается;
- writes и cursor movement не могут commit-ить frame в неправильный viewport;
- UI-renderers остаются на логических координатах и не управляют dirty regions;
- resize behavior покрывается unit-тестами без sleep-based ожиданий.

Negative:
- console driver abstraction стала шире;
- во время активного resize часть frames будет намеренно отброшена;
- ручная проверка в настоящей Windows console все еще нужна, потому что timing terminal
  resize нельзя полностью смоделировать unit-тестами.

## Done criteria

- `ScreenRenderer` инвалидирует front buffer при изменении viewport origin.
- Interrupted frame не применяет pending cursor position.
- Win32 writes и cursor placement выполняются только для ожидаемого viewport.
- Нижняя строка функциональных клавиш восстанавливается full redraw после origin-only
  viewport move.
- `dotnet build` проходит без warnings.
- `dotnet test` проходит.

## Outcome

Implementation summary:
- Добавлен `ConsoleViewport`.
- Добавлены viewport-aware driver operations для physical writes и cursor placement.
- `ScreenRenderer` захватывает frame viewport, привязывает к нему front-buffer validity и
  прерывает frame при изменении viewport.
- `Application` использует retry-on-interrupt flow для видимого UI.
- Временное resize file logging удалено из runtime-кода.
- Обновлены fakes и tests для viewport-origin changes, interrupted flush, cursor behavior,
  сохранения input после resize draining и связанных stale expectations.
- Registry lookup в `TextFileInfo` явно ограничен Windows, чтобы repo checks оставались
  warning-free.

Verification:
- `dotnet build` passed with 0 warnings.
- `dotnet test` passed: 505 passed, 1 skipped.

Follow-up:
- Вручную проверить continuous resize в настоящей Windows console. Нижняя строка
  функциональных клавиш должна восстанавливаться full redraw и не оставаться сдвинутой
  или испорченной.
