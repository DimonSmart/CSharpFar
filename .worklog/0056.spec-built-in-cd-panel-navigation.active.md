## Goal

Handle `CD` and `CHDIR` typed in the built-in command line as CSharpFar built-in
commands and navigate the active local file panel to the requested directory.

## Context

Typed external commands already run from the active panel directory in the current
console. `CD` and `CHDIR` are different from ordinary external commands: Far Manager
handles them as built-in commands instead of passing them to `%ComSpec%`.

CSharpFar should follow that model. A child shell process cannot change CSharpFar's
current directory or active panel location. For `cd abc`, CSharpFar must parse the
built-in command, resolve the target relative to the active panel directory, validate
it, and load it into the active local panel.

## Scope

- Run ordinary typed shell commands from the active panel directory as before.
- Recognize standalone `CD` and `CHDIR` commands before shell dispatch.
- Resolve the requested directory relative to the active local panel directory.
- If the target directory exists, differs from the active panel directory, and can be
  opened by the local panel source, load it into the active panel.
- Support relative paths, subdirectories, absolute paths, quoted paths with spaces,
  and `/d` drive-switch syntax.
- Leave the panel unchanged when the command is not `CD`/`CHDIR`, the target is
  missing, or the target is invalid/inaccessible.
- Handle invalid or inaccessible final paths without crashing.
- Add focused tests for built-in `CD`/`CHDIR` panel navigation.

## Non-goals

- Do not attempt to read the current directory of a child shell process.
- Do not make external commands such as `pushd` change the panel after they exit.
- Do not make the parent CSharpFar process current directory follow the panel.
- Do not add compatibility behavior for the old unsynchronized panel state.
- Do not change module command dispatch or non-local panel behavior.
- Do not add a persistent shell session.

## Plan

1. Add a small built-in `CD`/`CHDIR` parser for standalone command-line input.
2. Resolve the target path against the active panel directory.
3. Load the active local panel only after validating the resolved directory.
4. Keep current-console execution behavior for all ordinary shell commands.
5. Add application tests for built-in `CD`/`CHDIR` panel navigation.
6. Run repository checks.

## Pause points

- Stop if `CD` handling requires full shell command parsing beyond standalone
  `CD`/`CHDIR`.

## Acceptance criteria

- `cd ..` moves the active panel one directory up.
- `cd someFolder` moves the active panel into the existing subdirectory.
- `cd "folder with spaces"` moves the active panel into the directory with spaces.
- `cd /d D:\SomeFolder` changes the panel drive and directory when that drive and
  directory exist.
- `chdir someFolder` behaves like `cd someFolder`.
- `dir` runs through the configured shell and leaves the panel unchanged.
- An erroneous `cd` leaves the panel unchanged.
- Commands such as `pushd D:\Work` are ordinary shell commands and do not move the
  panel after the child process exits.
- The implementation does not attempt to read child-process cwd.
- Invalid or inaccessible final paths do not crash the application.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual checks:

1. Run `cd ..` from the built-in command line and confirm the active panel moves up.
2. Run `cd "folder with spaces"` and confirm the active panel opens that folder.
3. Run `cd /d D:\SomeFolder` when such a folder exists and confirm the active panel
   changes drive and directory.
4. Run `dir` and confirm the panel stays in place.
5. Run `cd C:\FolderThatDoesNotExist` and confirm the panel stays in place.

## Outcome

Implementation summary:
- Added built-in `CD`/`CHDIR` handling in `Application.ExecuteCommand` before
  ordinary shell dispatch.
- The built-in command resolves the target against the active local panel directory,
  supports `/d`, quoted paths, relative paths, parent paths, and absolute paths, and
  loads the active panel only when the target directory exists and can be opened.
- Ordinary commands such as `dir` still run through `IShellService` with the active
  panel directory as the working directory.
- Added application tests covering `cd child`, `cd ..`, quoted folders with spaces,
  `cd /d`, `chdir`, ordinary shell dispatch for `dir`, and missing target
  directories.

Deviations:
- The initial idea of reading the final current directory from a child shell process
  was discarded. CSharpFar now follows the Far-like model: `CD`/`CHDIR` are handled
  by the application as built-in commands, and external child process cwd is not read.

Follow-up:
- Manual console checks are still useful for command-line prompt behavior around
  built-in `CD`/`CHDIR`, especially with real drive changes.
