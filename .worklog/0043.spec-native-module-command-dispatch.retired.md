## Goal

Finish the remaining native module cleanup and route command-line module
prefixes to native/FarNet module actions before falling back to the shell.

## Context

Spec `0042` removed the old `ICSharpFarPlugin` entry point model but left
three naming and behavior follow-ups:

- panel/service abstractions still used `Plugin*` names;
- SFTP and FTP/FTPS projects still had plugin-style project/folder names;
- FarNet `ModuleCommand` discovery existed in `FarNetModuleHost`, but the app
  command line did not dispatch module prefixes.

The next real compatibility target should be a small existing FarNet module,
not another synthetic hello-world fixture. `FarNet.JsonKit` is the preferred
candidate for the next spec because it exposes real JSON commands and panels
without requiring editor, spell-checking, chart windows, or scripting runtimes.
NuGet describes it as JSON helpers with `jk:` commands and array/object panels.

## Scope

- Rename remaining `Plugin*` panel/service abstractions to module terminology.
- Rename SFTP and FTP/FTPS project folders, project files, namespaces, and
  project references to module terminology.
- Replace `PanelSourceId.Plugin(...)` with module-named source IDs.
- Extend `NativeModuleCatalog` with command prefixes and command-line dispatch.
- Connect `Application.ExecuteCommand()` so a known module prefix is handled
  before shell execution.
- Make FarNet `ModuleCommand` return opened panels when a command opens one.
- Keep the top menu text `Plugins`; it is a user-facing Far/FarNet convention,
  not the removed CSharpFar plugin domain model.
- Update tests and smoke harness code for renamed abstractions and command
  dispatch.

## Non-goals

- Making `FarNet.JsonKit` fully work in this pass.
- Adding new FarNet APIs beyond what command dispatch itself needs.
- Rewriting the SFTP/FTP panel source internals as FarNet `Explorer`.
- Removing historical references in retired worklog files.
- Changing on-disk settings paths for existing SFTP/FTP saved connections.

## Plan

1. Rename `CSharpFar.Plugin.Abstractions` to a module abstraction project and
   update all references.
2. Rename `IPluginPanel`, `PluginPanelInfo`, `PluginUiServices`,
   `PluginSettingsService`, and related uses.
3. Rename SFTP/FTP project folders, project files, namespaces, and test imports.
4. Replace `PanelSourceId.Plugin` with `PanelSourceId.Module` while preserving
   existing source ID string compatibility unless a test proves migration is
   needed now.
5. Add native command-prefix registration to `NativeModuleCatalog` for built-in
   modules and FarNet `ModuleCommand` prefixes.
6. Dispatch module command lines in `Application.ExecuteCommand()` before shell
   execution and open pending panels from FarNet commands.
7. Inspect a real FarNet package candidate and record the chosen next target.
8. Run build, tests, and the real FarNet smoke harness.

## Pause points

- Stop if project renaming would require changing NuGet/package identity beyond
  local project and namespace names.
- Stop if `FarNet.JsonKit` requires broad FarNet API work before a useful
  next-slice target can be identified.

## Acceptance criteria

- Production code has no `CSharpFar.Plugin.*` namespaces or project references.
- Production code has no `IPluginPanel`, `PluginPanelInfo`,
  `PluginUiServices`, or `PluginSettingsService` names.
- SFTP/FTP projects are module-named and still build.
- Command lines with registered module prefixes are dispatched to the module
  catalog before shell execution.
- FarNet `ModuleCommand` can open a CSharpFar module panel when the command
  opens a FarNet panel.
- A real FarNet package candidate is selected and documented for the next
  compatibility slice.
- `dotnet build CSharpFar.slnx` and `dotnet test CSharpFar.slnx` pass.

## Outcome

Implementation summary:
- Renamed the remaining shared extension abstractions from plugin terminology
  to module terminology: `CSharpFar.Module.Abstractions`,
  `IModulePanel`, `ModulePanelInfo`, `ModuleUiServices`, and
  `ModuleSettingsService`.
- Renamed SFTP and FTP/FTPS project folders, project files, and namespaces to
  `CSharpFar.Module.Sftp` and `CSharpFar.Module.Ftp`.
- Replaced `PanelSourceId.Plugin(...)` with `PanelSourceId.Module(...)` while
  preserving the existing `plugin:` source ID prefix for saved panel/source
  compatibility.
- Added command-prefix registration to `NativeModuleCatalog` and dispatch from
  `Application.ExecuteCommand()` before shell execution.
- Registered built-in `sftp`, `ftp`, and `ftps` prefixes, and registered FarNet
  `ModuleCommand` prefixes from `FarNetModuleHost`.
- Updated FarNet command invocation so commands can return opened panels when
  they call `Panel.Open()`.
- Added official-FarNet-compiled tests proving app-level command dispatch and
  command-opened FarNet panels.
- Selected `FarNet.JsonKit` 0.5.1 as the next real compatibility target.

Deviations:
- `ModuleSettingsService` intentionally still writes under the existing
  `plugins/<module-id>` settings folder to avoid breaking saved SFTP/FTP
  connection data.
- `PanelSourceId.Module(...)` intentionally keeps the existing `plugin:` value
  prefix for now for the same compatibility reason.
- `FarNet.JsonKit` was inspected but not made fully functional in this pass, as
  planned. Its first missing shim API is `FarNet.CommandParameters`.

Follow-up:
- Add `FarNet.CommandParameters` and the minimal parameter parsing helpers
  needed by `FarNet.JsonKit` `jk:open`.
- Use `FarNet.JsonKit` 0.5.1 as the next real-module smoke target: load the
  package module folder, run `jk:open file=<sample.json>`, and open a read-only
  CSharpFar panel with JSON entries.
- After JsonKit panel browsing works, decide separately whether editing,
  saving, and editor APIs are in scope.
