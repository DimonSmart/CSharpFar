# 010. Улучшение файловых операций

Type: spec
Status: done

Related:
- `.worklog/0007.spec-file-panel-options.md`
- `.worklog/0008.spec-menu-bar-popup-shadow.md`

## Goal

Сделать копирование, перемещение, удаление и создание каталогов предсказуемыми и ближе к Far Manager в ежедневных сценариях:

1. перед операцией показывать понятный диалог параметров;
2. показывать прогресс текущего файла и общий прогресс;
3. поддержать отмену без закрытия приложения;
4. расширить поведение при конфликтах;
5. сохранить важные метаданные по настройке пользователя;
6. добавить фильтр по маскам для copy/move;
7. добавить удаление в корзину как управляемую настройку;
8. убрать блокирующую синхронную модель файловых операций из UI path.

## Context

Сейчас CSharpFar поддерживает базовые операции:

1. `F5` копирует выбранные элементы или текущий элемент в путь из `InputDialog`;
2. `F6` перемещает или переименовывает;
3. `F8` удаляет навсегда после подтверждения;
4. `F7` создает один каталог через `Directory.CreateDirectory`;
5. `ProgressDialog` показывает только имя текущего файла;
6. `ConflictChoice` содержит только `Overwrite`, `Skip`, `Cancel`;
7. `FileOperationService` работает поверх `File.Copy`, `File.Move`, `Directory.Move`, `File.Delete`, `Directory.Delete`;
8. UI вызывает async methods через `.GetAwaiter().GetResult()`.

Из предложений по Far Manager для первого практичного этапа стоит оставить повседневные функции: общий прогресс, ETA, скорость, отмену, настройки операции, расширенные конфликты, only newer, сохранение timestamps/attributes, опциональное копирование ACL, фильтр масок, корзину, статистику удаления и создание дерева каталогов.

Редкие и рискованные возможности надо явно убрать из этой спецификации, чтобы не раздувать реализацию: альтернативные NTFS streams, копирование в несколько назначений, копирование в `NUL`, `.diz` descriptions, append file contents, создание hard/symbolic links вместо копирования и будущие виртуальные файловые системы.

## Scope

### Operation model

Заменить строковые методы `IFileOperationService` на модель запроса и результата.

Рекомендуемые типы:

```csharp
public sealed record FileOperationRequest
{
    public required FileOperationKind Kind { get; init; }
    public required IReadOnlyList<string> Sources { get; init; }
    public string? Destination { get; init; }
    public required FileOperationOptions Options { get; init; }
}

public enum FileOperationKind
{
    Copy,
    Move,
    Delete,
    CreateDirectory,
}

public sealed record FileOperationOptions
{
    public ConflictDecisionMode DefaultConflictDecision { get; init; } = ConflictDecisionMode.Ask;
    public bool OnlyNewer { get; init; }
    public bool PreserveTimestamps { get; init; } = true;
    public bool PreserveAttributes { get; init; } = true;
    public FileSecurityMode SecurityMode { get; init; } = FileSecurityMode.Inherit;
    public SymlinkCopyMode SymlinkMode { get; init; } = SymlinkCopyMode.CopyLink;
    public bool UseRecycleBinForDelete { get; init; }
    public string? FileMask { get; init; }
}
```

Keep async naming and cancellation:

```csharp
Task<FileOperationResult> ExecuteAsync(
    FileOperationRequest request,
    IProgress<FileOperationProgress>? progress,
    IFileOperationConflictResolver conflictResolver,
    CancellationToken cancellationToken);
```

Rules:

1. `ExecuteAsync` must not block the UI thread.
2. `CancellationToken` must be checked before each filesystem operation and during buffered copy.
3. A cancelled operation may leave already completed files in place. It must not report success.
4. The service must return a structured result with counts, byte totals, skipped items and errors.
5. Do not keep the old string overloads as compatibility shims unless another spec explicitly asks for compatibility.

### Pre-operation dialogs

Replace the current one-line `InputDialog` path prompt for `F5` and `F6` with operation dialogs.

Copy dialog fields:

1. destination path, prefilled with the opposite panel directory;
2. file mask, empty by default;
3. conflict mode;
4. only newer;
5. preserve timestamps;
6. preserve attributes;
7. security mode: inherit or copy ACL;
8. symlink mode: copy link or copy target contents.

Move dialog fields:

1. destination path or new name for a single item;
2. file mask for multi-source move;
3. conflict mode;
4. only newer;
5. preserve timestamps and attributes when move falls back to copy+delete;
6. security mode when move falls back to copy+delete;
7. symlink mode when move falls back to copy+delete.

Delete dialog fields:

1. item count;
2. total size if it can be calculated reasonably;
3. delete permanently or send to recycle bin;
4. confirmation follows `AppSettings.Ui.ConfirmDelete`.

Rules:

1. Dialogs should use existing popup/dialog rendering from spec `008`.
2. Do not introduce a new UI framework or a new renderer abstraction.
3. Use existing palette and input patterns.
4. Store the last selected options in settings and use them as defaults for future operations.

### Copy dialog UI

The copy dialog is a modal Far-like window over the file panels. It should be close to Far Manager in everyday behavior, without exposing advanced Far features that are out of scope for this stage.

The dialog contains:

1. title `Copy`;
2. line `Copy <source> to:`;
3. destination path input;
4. `Access rights` radio-style row;
5. existing-file behavior row;
6. `Preserve all timestamps` checkbox;
7. `Copy contents of symbolic links` checkbox;
8. `Use filter` checkbox;
9. `Filter mask` input;
10. `{ Copy }` and `[ Cancel ]` buttons.

Target structure:

```text
┌──────────────────────────── Copy ────────────────────────────┐
│ Copy ЮТ. Для умелых рук. 1956-1990 to:                        │
│ E:\Temp\                                                      │
│                                                               │
│ Access rights: ( ) Default ( ) Copy (x) Inherit               │
├───────────────────────────────────────────────────────────────┤
│ Already existing files: Ask                                   │
│ [ ] Preserve all timestamps                                   │
│ [ ] Copy contents of symbolic links                           │
├───────────────────────────────────────────────────────────────┤
│ [ ] Use filter                                                │
│ Filter mask:                                                  │
│ *                                                             │
├───────────────────────────────────────────────────────────────┤
│                     { Copy }   [ Cancel ]                     │
└───────────────────────────────────────────────────────────────┘
```

Access rights:

1. use exactly radio-style notation: `( ) Default ( ) Copy (x) Inherit`;
2. selected state is marked by `(x)`;
3. do not replace `(x)` with `•`, `.`, or another marker.

Existing files:

1. row text must be `Already existing files: Ask`;
2. do not put a space before the colon;
3. the value must look like a switchable field, not plain text;
4. supported switchable values for copy are `Ask`, `Overwrite`, `Skip`, `Rename`, and `Append`;
5. move/rename does not show `Append` because appending is only meaningful for file copy conflicts.

Filter:

1. do not implement the full Far filter dialog;
2. do not show a `[ Filter ]` button;
3. default mask text is `*`;
4. if `Use filter` is off, mask is not applied;
5. if `Use filter` is on, copy only files matching the mask;
6. support wildcard masks such as `*`, `*.cs`, `test*.txt`, and `*.cs;*.json`;
7. split multiple masks with `;`;
8. trim whitespace around masks and ignore empty parts;
9. use case-insensitive comparison on Windows;
10. for recursive directory copy, apply the filter to files and still traverse directories whose names do not match the mask.

Not supported and not shown:

1. `[ ] Process multiple destinations`;
2. separate advanced Far-style filter window;
3. disabled `[ Filter ]` button.

Buttons:

1. bottom row contains only `{ Copy }   [ Cancel ]`;
2. `{ Copy }` is the default action;
3. `Esc` closes the dialog without copying;
4. `Tab` moves focus between interactive elements;
5. `Space` toggles checkbox/radio values.

Visual style:

1. copy dialog must read as a separate modal dialog, not a file panel;
2. dialog palette is separate from panel palette;
3. dialog background is gray;
4. primary text is black or dark gray;
5. border and internal separators use dialog border color;
6. active input/switchable field uses dark cyan background with light text;
7. disabled text is muted gray;
8. shadow is black or the darkest palette color;
9. dialog has a thin border and title embedded in the top border;
10. dialog is compact, centered, and has internal padding.

### Progress

Replace `ProgressDialog.Update(string currentFile)` with byte-based progress.

Copy progress has two phases:

1. preliminary scan;
2. actual copy.

The scan phase runs before copying and calculates total work. During this phase the dialog shows:

```text
                          Copy
 Scanning the folder
 YouTube
────────────────────────────────────────────────────────────
 Files:                                           2 651
 Folders:                                           123
 Bytes:                                  96 081 696 632
```

Rules:

1. copying has not started during scan;
2. update current scanned folder;
3. update found file count;
4. update found folder count;
5. update accumulated byte count;
6. do not show a progress bar during scan.

The copy phase starts after scan, so `Total` already has stable denominators for files and bytes.

Copy phase dialog target:

```text
                          Copy
 Copying the file
 17.09.2023_Carefour\Carrefour.wfpbundle
 to
 C:\0\17.09.2023_Carefour\Carrefour.wfpbundle
 ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░      1%

───────────────────────── Total ─────────────────────────
 Files:                                             2 / 4
 Bytes:                         18 118 235 / 1 211 723 996
 ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░      1%

──────────────────────────────────────────────────────────
 Time: 00:00:01          Remaining: 00:00:00            0B/s
```

Copy phase rules:

1. top block is current file only;
2. long paths may be shortened in the middle;
3. current file progress is based on current file size;
4. `Total` separator title is centered in the line;
5. `Files` shows processed file count and total file count;
6. `Bytes` shows copied bytes and total bytes;
7. total progress is byte-based;
8. bottom row shows elapsed time, remaining time, and speed;
9. speed may use `B/s`, `KB/s`, `MB/s`, or `GB/s`;
10. if speed is not available, show `0B/s`.

Cancellation rules:

1. pressing `Esc` during the preliminary scan cancels immediately without confirmation;
2. pressing `Esc` during actual copying shows a warning-style red confirmation dialog;
3. confirmation buttons are `{ Yes }` and `[ No ]`;
4. choosing `Yes` cancels the operation;
5. choosing `No` resumes showing copy progress;
6. actual copying is paused while the confirmation dialog is open.

Progress dialog visual rules:

1. draw a gray outer rectangle over panels;
2. draw the actual dialog frame inside it with a one-character margin;
3. keep text away from the frame with internal padding;
4. draw dark shadow to the right and bottom when possible;
5. progress bar must not use `#`;
6. filled progress uses solid block characters such as `█`;
7. empty progress uses light block/checker characters such as `░`;
8. percent is shown to the right of the bar.

Copy, copy-progress, conflict, and drive-selection modal frames should use the shared modal dialog renderer instead of each dialog hand-rolling the outer background and one-character frame margin.

Progress model:

```csharp
public sealed record FileOperationProgress
{
    public required FileOperationKind Kind { get; init; }
    public FileOperationPhase Phase { get; init; }
    public required string CurrentPath { get; init; }
    public string? CurrentDestinationPath { get; init; }
    public long CurrentBytesDone { get; init; }
    public long CurrentBytesTotal { get; init; }
    public long TotalBytesDone { get; init; }
    public long TotalBytesTotal { get; init; }
    public int ItemsDone { get; init; }
    public int ItemsTotal { get; init; }
    public int FoldersDone { get; init; }
    public double BytesPerSecond { get; init; }
    public TimeSpan? TimeRemaining { get; init; }
    public TimeSpan Elapsed { get; init; }
}
```

UI requirements:

1. show current path, destination when applicable, item count and current action;
2. show current-file progress bar;
3. show total progress bar;
4. show speed and estimated time remaining;
5. expose `Esc` or a visible cancel button/key hint to request cancellation;
6. keep the progress window as a modal operation window for this stage.

Settings:

```csharp

    public string SymlinkMode { get; set; } = "CopyLink";
```

Add it under `AppSettings.FileOperations`.

### Conflict handling

Replace `ConflictChoice` with a richer decision model.

Required decisions:

```text
Ask
Overwrite
OverwriteAll
Skip
SkipAll
Rename
RenameAll
Append
OnlyNewer
Cancel
```

Rules:

1. `OverwriteAll`, `SkipAll`, `RenameAll` apply to the rest of the current operation only.
2. `OnlyNewer` overwrites only when source last write time is greater than destination last write time.
3. `Rename` asks the user for a destination name.
4. `RenameAll` uses generated names without asking for every conflict.
5. Generated names use the existing destination directory and append ` (2)`, ` (3)`, etc. before the extension.
6. Directory/file type conflicts must never be overwritten silently.
7. The conflict dialog must show source and destination metadata: name, type, size and last write time when available.
8. `Append` is supported for copy conflicts only and appends source bytes to the existing destination file.

Conflict dialog UI:

1. title is `Warning`;
2. first line says `File already exists`;
3. show destination path;
4. show `New` and `Existing` rows with size and last write time when available;
5. show `[ ] Remember choice`;
6. buttons are `{ Overwrite } [ Skip ] [ Rename ] [ Append ] [ Cancel ]` for copy conflicts;
7. `Append` is not shown for move/rename conflicts;
8. when `Remember choice` is enabled, ordinary choices become operation-wide choices: overwrite all, skip all, rename all, append all;
9. buttons support hotkeys, left/right navigation, `Enter`, `Space`, and mouse click.

Dialog interaction:

1. modal dialogs should use shared dialog/window helpers for outer background, frame, buttons, keyboard navigation and mouse clicks;
2. avoid per-dialog custom button handling unless a dialog has genuinely different interaction rules.

### Mask filtering

Use the existing Far-like mask support from `CSharpFar.Core.FileMasks`.

Rules:

1. Empty mask means all selected sources.
2. Mask filtering applies to files inside selected directories recursively.
3. Directories are traversed when they may contain matching files.
4. Empty directories are copied only when the mask is empty.
5. The mask expression must support the existing parser behavior: `*`, `?`, ranges, comma/semicolon lists, `|` excludes, regex tokens and mask groups.
6. Do not create a second mask syntax for file operations.

### Metadata

Timestamps:

1. When `PreserveTimestamps = true`, copy creation time, last write time and last access time after file contents are copied.
2. Directory timestamps are restored after their children are processed.
3. If timestamp restore fails, record an item error and continue unless the failure prevents the operation from completing.

Attributes:

1. When `PreserveAttributes = true`, copy ordinary `FileAttributes` after file contents are copied.
2. For copied files, clear temporary attributes that would prevent writing the target before applying final attributes.
3. `ReadOnly` should be preserved by default. The old Far-specific CD/DVD read-only clearing is not part of this stage.

Security:

```csharp
public enum FileSecurityMode
{
    Inherit,
    CopyAccessControl,
}
```

Rules:

1. Default is `Inherit`.
2. `CopyAccessControl` attempts to copy Windows ACL from source to target.
3. ACL failures are reported as item errors. They must not be hidden.
4. Do not add cross-platform compatibility code.

### Symlinks and reparse points

Support only the two user-visible policies needed to avoid surprising recursion:

```csharp
public enum SymlinkCopyMode
{
    CopyLink,
    CopyTargetContents,
}
```

Rules:

1. Detect symbolic links and junction-like reparse points before recursive traversal.
2. Default to `CopyLink`.
3. `CopyLink` recreates the link when Windows allows it.
4. If link creation is not permitted, report a clear item error instead of silently copying target contents.
5. `CopyTargetContents` follows the target and copies contents.
6. Do not implement "create links instead of copying" for ordinary files in this stage.
7. Do not implement hard-link identity preservation in this stage.

### Delete

Add delete options:

1. delete permanently;
2. send to recycle bin.

Rules:

1. Default follows `AppSettings.FileOperations.UseRecycleBinForDelete`.
2. Recycle bin behavior is Windows-only and should live in the existing filesystem/platform-specific project structure.
3. If recycle bin is unavailable for a path, show a clear error or ask for permanent delete.
4. Delete progress should show item count and current path.
5. Permanent delete keeps the existing recursive delete behavior, but reports failures per item.

### Create directory

Improve `F7`:

1. allow creating nested directories such as `a\b\c`;
2. keep single-directory creation as the common case;
3. fail fast for invalid names;
4. after success, refresh the panel and place cursor on the created top-level directory when possible.

## Non-goals

Do not implement in this spec:

1. alternative NTFS data streams;
2. copy to multiple destination directories;
3. copy to `NUL`;
4. `.diz` description file preservation;
5. append / append all conflict mode;
6. creating hard links or symbolic links instead of copying ordinary files;
7. hard-link identity preservation;
8. plugin or virtual filesystem copy/move APIs;
9. archives, FTP, SFTP or cloud file operations;
10. background minimized progress that lets the user keep browsing panels during the operation;
11. compatibility wrappers for old file operation APIs;
12. read-only attribute clearing specifically for CD/DVD sources.

## Pause points

- Stop after the scan-phase design if the current panel selection model cannot represent filtered recursive sources cleanly.
- Stop before ACL copying if the target .NET APIs differ across target frameworks or require package changes.
- Stop before recycle-bin implementation if the required Windows API shape would force a broad platform abstraction.

## Acceptance criteria

1. `F5` opens a copy dialog with destination and operation options.
2. `F6` opens a move/rename dialog with operation options.
3. `F8` can delete permanently or send to recycle bin.
4. `F7` can create nested directory paths.
5. Copy and move progress show current file progress and total progress when enabled.
6. Progress shows speed, ETA and processed item count.
7. User cancellation stops the operation and reports cancellation.
8. UI is not blocked by `.GetAwaiter().GetResult()` during file operations.
9. Conflict dialog supports overwrite, overwrite all, skip, skip all, rename, rename all, only newer and cancel.
10. Rename-generated names use `name (2).ext`, `name (3).ext`, etc.
11. `OnlyNewer` compares source and destination last write times.
12. File masks use the existing Far mask parser.
13. Masked recursive copy does not copy non-matching files.
14. Timestamps are preserved when enabled.
15. Attributes are preserved when enabled.
16. ACLs are inherited by default.
17. ACLs are copied only when `CopyAccessControl` is selected.
18. Symlink/reparse-point behavior follows the selected `SymlinkCopyMode`.
19. Link creation failure is reported clearly and does not silently copy target contents.
20. Directory/file type conflicts are not overwritten silently.
21. Operation result records copied, moved, deleted, skipped, failed and cancelled counts.
22. Settings persist operation defaults under `AppSettings.FileOperations`.
23. Existing panel refresh and selection clearing still behave predictably after operations.
24. There is no implementation for alternative streams, multi-target copy, `NUL`, `.diz`, append mode or create-links-instead-of-copying.

Status: done

Implementation summary:

- Added structured file operation models: request, options, progress, result, conflict metadata and conflict decisions.
- Replaced the old `IFileOperationService` copy/move/delete/create methods with one `ExecuteAsync` contract.
- Reworked `FileOperationService` around a scan/execute flow with byte progress, cancellation, result counts, mask filtering, conflict handling, generated rename names, `OnlyNewer`, timestamp/attribute preservation, optional ACL copy attempt, symlink policy handling, recycle-bin delete and nested directory creation.
- Added `AppSettings.FileOperations` defaults and JSON persistence coverage.
- Updated copy/move/delete/create UI paths to build `FileOperationRequest`, show byte-based progress, use the richer conflict dialog and accept optional copy/move masks.
- Replaced the temporary sequential destination/filter prompts with a single Far-like copy/move dialog containing destination, access rights, conflict mode, timestamps, symlink contents and filter controls.
- Reduced progress dialog flicker by rendering progress updates on the main operation loop with throttling and frame-buffered redraws.
- Routed file-conflict dialogs through the main UI loop instead of showing them from the file-operation worker thread. This avoids a key-reading race where the progress loop could consume keys such as `O` before the conflict dialog saw them.
- Added the shared modal dialog renderer for copy/progress/conflict modal frames with a one-character outer margin.
- Added shared dialog button handling with hotkeys, left/right navigation, `Enter`, `Space`, and mouse clicks.
- Added warning-style red cancel confirmation for `Esc` during the copy phase; `Esc` during the scan phase cancels immediately.
- Reworked the conflict dialog into a warning-style modal with `Remember choice`, copy-only `Append`, and operation-wide remembered choices.
- Added a pause controller so the file-operation worker waits while the copy-cancel confirmation dialog is open.
- Updated test fakes for the new contract and added spec 010 tests for progress, masks, `OnlyNewer`, timestamps, generated rename and nested directory creation.
