## Goal

Make one small real FarNet module command work end to end: load
`FarNet.JsonKit` 0.5.1, dispatch `jk:open file=<json>`, and open a read-only
CSharpFar panel backed by the FarNet panel/explorer APIs used by that command.

## Context

The native FarNet domain model and command dispatch now reach real module
commands. `FarNet.JsonKit` is the next compatibility target because it is a
real FarNet package with command-line entry points and panels, but its first
observed blocker is still narrow: the shim is missing `FarNet.CommandParameters`.

This spec should prove that a module compiled against official `FarNet.dll`
can use FarNet command parsing and open a simple panel through CSharpFar. It is
not a full JsonKit compatibility promise.

## Scope

- Add the minimal `CommandParameters`, `ParameterOptions`, and subcommand
  helpers required by real FarNet modules using command parameter parsing.
- Keep command parsing behavior predictable for quoted strings, `name=value`
  pairs, required values, path values, bool/value conversion, and unknown
  parameter diagnostics.
- Extend the existing FarNet panel adapter only as needed for `jk:open` to show
  a read-only JSON panel.
- Keep unsupported FarNet APIs failing as controlled module failures with clear
  diagnostics.
- Run a smoke check against the real `FarNet.JsonKit` 0.5.1 module folder copied
  into the app output.

## Non-goals

- Implementing JsonKit editing, saving, or editor integration.
- Implementing broad FarNet menus, dialogs, macros, viewer, or editor APIs.
- Making every JsonKit command work.
- Replacing CSharpFar storage/settings compatibility paths.
- Adding security sandboxing for third-party FarNet modules.

## Plan

1. Inspect official FarNet metadata for `CommandParameters`,
   `ParameterOptions`, `Subcommand`, and `ModuleCommand.InvokeSubcommand`.
2. Implement the missing shim types and minimal parser behavior.
3. Add focused unit/fixture tests for command parameters and FarNet command
   dispatch through official-FarNet-compiled modules.
4. Run the real JsonKit smoke harness and iteratively add only the next small
   panel/explorer API surface needed by `jk:open`.
5. Run repo verification and record any still-unsupported JsonKit behavior.

## Pause points

- Stop if `jk:open` requires editor/viewer/dialog/macro APIs before a panel can
  be opened.
- Stop if JsonKit requires a broad mutable panel implementation before showing
  read-only items.

## Acceptance criteria

- A real `FarNet.JsonKit` 0.5.1 module folder is discovered without loading the
  original `FarNet.dll`.
- `jk:open file=<sample.json>` reaches JsonKit command code and no longer fails
  on missing `CommandParameters`.
- The command opens a CSharpFar module panel for a simple JSON file, or fails at
  a documented unsupported API outside this spec's read-only panel target.
- Unsupported APIs remain controlled diagnostics, not host crashes.
- `dotnet build CSharpFar.slnx` and `dotnet test CSharpFar.slnx` pass.

## Outcome

Implementation summary:
- Added the FarNet command parsing shim surface needed by real modules:
  `CommandParameters`, `ParameterOptions`, `Subcommand`, and
  `ModuleCommand.InvokeSubcommand`.
- Implemented minimal connection-string style command parameter parsing with
  required parameters, bool/value conversion, environment expansion, full-path
  resolution against `Far.Api.CurrentDirectory`, command-file reads, and
  unknown-parameter failures.
- Extended the panel shim just enough for JsonKit `jk:open`: `Panel.OpenChild`,
  parent/child links, official `ExplorerFunctions` values, Explorer capability
  properties including `CanGetContent`, official `PanelSortMode` and
  `PanelViewMode` values, `FarColumn`, `SetColumn`, `PanelPlan`, and
  `PanelPlan.Clone()`.
- Added official-FarNet-compiled fixture coverage for
  `InvokeSubcommand`/`CommandParameters`, including controlled unknown
  parameter failure diagnostics.
- Updated the local smoke harness to enable panel tools and print opened panel
  items.
- Verified real `FarNet.JsonKit` 0.5.1 from the copied module folder:
  `jk:open file=C:\Private\CSharpFar\artifacts\JsonKitSample.json` opens an
  `Object $` panel with `name` and `items` entries and no diagnostics.

- `artifacts/FarNetSmokeHarness/Program.cs`

Deviations:
- The smoke harness change is in ignored `artifacts/` code and is not part of
  production runtime; it was updated to validate the real JsonKit target.
- `CommandParameters` intentionally implements the documented subset needed by
  command modules. Cursor-path fallback options still throw an explicit
  unsupported API exception when no value is supplied.
- JsonKit root object items are shown as read-only panel entries. Deeper JSON
  navigation/open-file behavior was not expanded in this pass.

Follow-up:
- Add FarNet panel navigation APIs used when entering JSON object/array nodes
  if the next target is browsing inside JsonKit panels.
- Decide separately whether JsonKit editing/saving APIs are in scope; they will
  require broader `Explorer` operations and likely editor/input behavior.
