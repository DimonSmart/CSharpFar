## Goal

Finish the native FarNet domain migration by removing the remaining CSharpFar
plugin abstraction from built-in extensions and making SFTP/FTP entry points
native module actions.

## Context

Spec `0041` moved FarNet loading out of the `ICSharpFarPlugin` catalog, but left
three follow-ups: the FarNet host project still had a plugin-style name, SFTP
and FTP/FTPS still implemented `ICSharpFarPlugin`, and the old plugin manager
remained in app composition.

The user clarified that the intended architecture is not a proxy/adaptor plugin
system. The native model should be FarNet-like: built-in features and external
DLL modules expose module actions, and the application projects those actions
into menus and disk menus directly.

## Scope

- Rename the FarNet host project/folder/namespace away from
  `CSharpFar.Plugin.FarNet`.
- Add an app-side native module catalog that can combine built-in module actions
  and external FarNet module actions.
- Convert SFTP and FTP/FTPS menu and disk entry points from `ICSharpFarPlugin`
  to native module actions.
- Remove `ICSharpFarPlugin` and `PluginManager` from normal app composition when
  no built-in code needs them.
- Preserve current SFTP/FTP behavior and panel implementations.
- Keep external FarNet DLL loading working.

## Non-goals

- Full FarNet API compatibility.
- Rewriting SFTP/FTP panel sources to inherit FarNet `Explorer` in this pass.
  Their entry points must be native module actions; their panel source internals
  may remain existing `IPluginPanel` implementations until a later deeper
  panel-model refactor.
- Loading original `FarNet.dll` as the host.
- Sandboxing untrusted modules.

## Plan

1. Rename the FarNet host project with explicit file moves and update project
   references.
2. Introduce native module catalog/action types in the app layer.
3. Add built-in SFTP and FTP/FTPS native module providers that reuse existing
   connection manager/opening flows.
4. Replace `PluginManager` use in `Application` with the native module catalog.
5. Delete or quarantine old plugin abstractions only after no production code
   references them.
6. Update tests for native module menu/disk projection.
7. Run `dotnet build` and `dotnet test`.

## Pause points

- Stop if SFTP/FTP conversion requires changing their remote panel file
  operation contracts.
- Stop if removing `ICSharpFarPlugin` would require rewriting unrelated tests
  or historical fixture projects in the same pass.

## Acceptance criteria

- `Program.cs` has no `plugins:` argument and no `SftpPlugin`/`FtpPlugin`
  construction.
- No production code references `ICSharpFarPlugin` or `PluginManager`.
- Top `Plugins` menu and `Alt+F1`/`Alt+F2` still expose SFTP, FTP/FTPS, and
  external FarNet `ModuleTool` actions.
- External real FarNet module smoke tests still pass.
- SFTP/FTP panels still open through existing connection manager flows.
- `dotnet build CSharpFar.slnx` and `dotnet test CSharpFar.slnx` pass.

## Outcome

Implementation summary:
- Renamed the FarNet host project from `CSharpFar.Plugin.FarNet` to
  `CSharpFar.FarNetHost` and updated app/test references.
- Added `NativeModuleCatalog` in the app layer. It projects built-in module
  actions and external FarNet `ModuleTool` actions into the same top `Plugins`
  menu and disk menu model.
- Converted SFTP and FTP/FTPS entry points from `ICSharpFarPlugin` to
  `SftpModule` and `FtpModule` native module action classes. Their existing
  connection manager and remote panel source internals are reused.
- Removed `PluginManager`, `ICSharpFarPlugin`, plugin menu/open info types, and
  `plugin.open` command dispatch from production code.
- Kept `IPluginPanel`, `PluginPanelInfo`, `PluginUiServices`, and
  `PluginSettingsService` as transitional shared panel/service abstractions;
  they are no longer tied to `ICSharpFarPlugin` entry point loading.
- Renamed FarNet host menu projection from `PluginMenuItems` to `MenuItems`.

Deviations:
- The old plugin entry point abstractions were deleted instead of quarantined
  because production code no longer referenced them and tests could be updated
  without touching unrelated fixture behavior.
- SFTP/FTP project names still include `Plugin` in their assembly/folder names.
  Their entry point classes and IDs are now module-named; project/package
  renaming is left as a packaging cleanup.

Follow-up:
- Rename the remaining panel/service abstractions away from `Plugin*` once the
  panel model is converted to the FarNet-like domain model.
- Consider moving SFTP/FTP projects to module-named project folders.
- Add command-line dispatch through the native module catalog for FarNet
  `ModuleCommand` and built-in module prefixes.
- Continue FarNet API compatibility in focused slices: Panel/Explorer support,
  then editor/viewer/dialog APIs as separate specs.
