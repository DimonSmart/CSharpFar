# 0040 SPEC Real FarNet module DLLs through shim

Type: spec

Replaces:
- none

Related:
- 0039

## Goal

Load third-party FarNet module DLLs compiled against the official `FarNet.dll`
while binding them at runtime to CSharpFar's compatibility shim.

## Context

Spec `0039` added the first FarNet `ModuleTool` loader and a CSharpFar-owned
assembly named `FarNet.dll`. Its tests use a fixture compiled against the shim.
The next compatibility step is proving binary loading of modules compiled
against the official FarNet API package.

The original FarNet host assembly is not loaded as a host. FarNet panels and
many APIs rely on `Far.Api`, `FarNet.Works.Far2.Api`, and Far Manager lifecycle
objects. CSharpFar provides only the API surface it can map predictably.

## Scope

- Add a fixture module compiled against official `FarNet.dll` version
  `10.0.30.0`.
- Keep binding module `FarNet` references to `CSharpFar.FarNetCompat`.
- Add loader options for module root, data root, and panel-tool support.
- Add resource lookup for `Resources=true` module action names.
- Add user-visible FarNet diagnostics.
- Add a read-only FarNet `Panel`/`Explorer` adapter for simple virtual panels.
- Add `ModuleCommand` discovery and direct command-line dispatch support inside
  the FarNet plugin.

## Non-goals

- Loading original `FarNet.dll` as the host.
- Full FarNet compatibility.
- Editor, viewer, macro, full dialog, menu, or Far Manager native panel API
  support.
- Sandboxing untrusted modules.

## Plan

1. Add the official-FarNet fixture project and keep the existing shim fixture as
   regression coverage.
2. Introduce `FarNetPluginOptions` and wire it through `FarNetPlugin`.
3. Implement `.resources` lookup in `FarNetModuleManager`.
4. Add a diagnostics menu item when diagnostics exist.
5. Expand the shim with the minimum official `Explorer`, `Panel`, `FarFile`,
   and event-args shape needed by simple panel modules.
6. Add `FarNetPanelAdapter` to expose simple FarNet panels as `IPluginPanel`.
7. Discover `ModuleCommand` and support direct `Open(CommandLine, ...)`
   dispatch for tests and future host integration.
8. Add focused tests and run repo checks.

## Pause points

- Stop if official-FarNet fixture compilation requires loading the real
  `FarNet.dll` into the test process.
- Stop if panel support starts requiring `FarNet.Works.Far2.Api` instead of the
  shim-owned simple panel path.

## Acceptance criteria

- A module compiled against official `FarNet.dll` loads through the shim.
- A real `FarNet.dll` next to a module is ignored by the loader.
- `Resources=true` action names resolve from sidecar `.resources` files.
- FarNet diagnostics are visible from the CSharpFar plugin menu.
- A simple FarNet panel tool opens a CSharpFar plugin panel and lists items.
- Module command prefixes are discovered and can be invoked via
  `Open(CommandLine, ...)`.
- Existing spec `0039` tests remain green.

## Outcome

Implementation summary:
- Added `CSharpFar.Tests.Fixtures.OfficialFarNetModule`, compiled against the
  official NuGet `FarNet` package `10.0.30`, and kept the shim-compiled fixture
  from `0039` as regression coverage.
- Kept the shim assembly identity as `FarNet`, version `10.0.30.0`; the module
  load context resolves supported `FarNet` references to the CSharpFar shim and
  ignores a local `FarNet.dll` in the module directory.
- Added `FarNetPluginOptions` for module root, data root, and opt-in panel-tool
  support.
- Added sidecar `.resources` lookup for `Resources=true` actions with diagnostics
  instead of startup crashes for missing or invalid resources.
- Added a `FarNet diagnostics...` plugin menu item when diagnostics exist.
- Expanded the shim with minimal `ModuleCommand`, `Explorer`, `Panel`,
  `FarFile`, `SetFile`, and event-args API needed by simple real FarNet modules.
- Added `FarNetPanelAdapter` for simple FarNet panel tools and direct
  `ModuleCommand` discovery/dispatch inside `FarNetPlugin`.
- Added focused tests for official-FarNet-compiled DLL loading, local real
  `FarNet.dll` ignoring, resources, diagnostics, command dispatch, and panel
  projection.

Deviations:
- `EnablePanelTools` remains `false` by default. The read-only panel adapter is
  implemented and covered with opt-in tests, but the app still requires an
  explicit option before FarNet panel tools can open panels.
- `ModuleCommand` support is implemented inside `FarNetPlugin.Open()` for direct
  command-line dispatch and metadata projection. The broader CSharpFar command
  line host is not rewired in this slice.
- FarNet tests are placed in a non-parallel xUnit collection because FarNet uses
  the process-wide `Far.Api` singleton.
