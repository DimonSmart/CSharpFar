# 0033 SPEC Viewer Encoding Selection

Type: spec

Replaces:
- none

Related:
- 0012
- 0014
- 0032

## Goal

Let the F3 viewer decode text with either automatic encoding detection or a user-selected
encoding while keeping the unified streaming viewer path for small and huge files.

The viewer must not load the whole file to switch code pages. Changing encoding should
rebuild the text scanner state over the same byte reader and block cache, then redraw from
the nearest sensible byte position.

## Context

`0032` replaced the old small-file `FileViewer` path with one streaming viewer:
`FileViewer` delegates to `LargeFileViewer`, which reads bytes through
`RandomAccessFileByteReader`, `BlockCache`, and `LineScanner`.

There are now three independent encoding paths:

1. `LineScanner` detects BOM, UTF-16-looking content, binary-looking content, valid UTF-8,
   and then falls back to `Encoding.Default`.
2. `FileSystemSearchService` has its own `DetectEncoding` for content search: BOM,
   valid UTF-8, then the current Windows ANSI code page.
3. `TextFileReader` uses `StreamReader` BOM detection and falls back to `Encoding.Default`
   after a UTF-8 decoder failure.

This produces different results between search, viewer, editor, and Quick View. The first
fix should make detection common and should add manual selection to the streaming viewer
without bringing back the old in-memory viewer.

## Scope

- Add a shared encoding detection model and detector outside `LargeFileViewer` and
  `FileSystemSearchService`.
- Register `CodePagesEncodingProvider` in the layer that creates non-Unicode encodings.
- Detect encoding from a bounded sample, not from the full file.
- Keep binary detection available to the viewer so binary-looking files still open in hex.
- Add a viewer encoding mode:
  - automatic detection;
  - explicit encoding by code page/name.
- Add a small catalog of encodings for manual selection:
  - automatic detection;
  - UTF-8;
  - UTF-8 with BOM label when detected, but manual UTF-8 should not require a BOM;
  - UTF-16 LE;
  - UTF-16 BE;
  - Windows ANSI code page for the current culture;
  - Windows-1251;
  - Windows-1252;
  - CP866.
- Add a viewer command, preferably `F8`, labelled `Encoding`, that opens a modal list and
  applies the selected encoding immediately.
- Store the current encoding mode in `LargeFileViewerState`.
- Recreate `LineScanner` and `SparseLineIndex` when the encoding changes.
- Preserve `LargeFileViewMode` when possible:
  - text stays text;
  - hex stays hex;
  - selecting an explicit text encoding from hex switches to text.
- Preserve approximate position across encoding changes by using the current byte offset
  and snapping it to a valid line start for the new text scanner.
- Show the active encoding in the viewer header for text mode.
- Unify `FileSystemSearchService` and `TextFileReader` with the shared detector.
- Keep `TextFileReader.MaxFileSizeBytes` as the editor/Quick View bound only; F3 must not
  use that limit.
- Update tests and README.

## Non-goals

- Restoring the old in-memory F3 viewer path.
- Reading the whole file to auto-detect encoding.
- Per-file remembered encoding.
- Persisting default viewer encoding in application settings in this spec.
- Manual encoding selection in the editor.
- A Quick View encoding selection UI.
- Search dialog code page selection beyond the existing automatic mode.
- Full charset detection by statistics or external libraries.
- Perfect recovery from arbitrary damaged byte streams.
- Hex editing or binary search.

## Plan

1. Add Core encoding contracts, for example `TextEncodingMode`,
   `TextEncodingSelection`, and `EncodingDetectionResult`.
2. Add a shared detector that accepts a byte sample and returns:
   - `Encoding`;
   - preamble/content start length;
   - binary-looking flag;
   - UTF-16 byte order metadata needed by `LineScanner`;
   - a display name for status/header text.
3. Keep sample acquisition near each caller:
   - `LineScanner.CreateAsync` reads only the first bounded bytes through
     `BlockCache`/`IFileByteReader`;
   - `FileSystemSearchService` uses the content bytes it already read for files under the
     search size limit;
   - `TextFileReader` reads or samples through normal file APIs before reading all lines
     for editor and Quick View.
4. Change `LineScanner.CreateAsync` to accept an optional explicit encoding selection.
   Automatic mode uses the shared detector. Explicit mode uses the chosen encoding and
   still skips a matching BOM when present.
5. Move newline scanning details out of detection duplication:
   `LineScanner` should derive UTF-16 newline width and byte order from the selected
   encoding/detection result.
6. Add `LargeFileViewerState.EncodingSelection` and replace immutable scanner state with
   a controlled scanner reset method that also clears `SparseLineIndex`.
7. Add an encoding picker dialog following existing modal dialog style. The dialog returns
   either automatic mode or one explicit encoding.
8. Wire `F8` in `LargeFileViewer.RunLoop`, update the footer, and redraw immediately after
   changing encoding.
9. Replace private search/viewer/TextFileReader detection implementations with the shared
   detector.
10. Add focused tests for the detector, scanner reset, viewer picker action, search
    decoding, and TextFileReader behavior.
11. Update README viewer text.

## Pause points

- Stop before adding persisted settings for preferred encoding.
- Stop before adding editor or Quick View encoding UI.
- Stop before introducing an external charset detection library.
- Stop if sharing the detector would require `CSharpFar.Core` to depend on
  `CSharpFar.App.Viewer`; the dependency should remain in the opposite direction or use
  sample bytes as the boundary.

## Acceptance criteria

- F3 still uses the streaming viewer for every file.
- Opening a huge text file reads only bounded byte samples and visible line ranges before
  the first render.
- Automatic detection is shared by viewer, search, and `TextFileReader`.
- BOM detection handles UTF-8, UTF-16 LE, and UTF-16 BE.
- UTF-16-looking files without BOM remain readable in the viewer.
- Valid UTF-8 without BOM is detected as UTF-8.
- Invalid UTF-8 falls back to the Windows ANSI code page for the current culture, not
  blindly to `Encoding.Default`.
- Binary-looking files still open in hex mode by default.
- The viewer header shows the active text encoding in text mode.
- `F8` opens an encoding picker with automatic mode and the supported explicit encodings.
- Selecting an explicit encoding redraws the same file without restarting the application.
- Switching encoding clears stale decoded-line cache/index state.
- Switching encoding does not dispose or reopen the underlying file handle unless the
  existing viewer architecture makes that simpler and still keeps full-file loading out.
- Damaged byte sequences in text mode render with replacement characters or a clear
  status indication; they must not close the viewer.
- Control-character sanitizing from `0032` still applies after encoding changes.
- Editor and Quick View continue to enforce their bounded `TextFileReader` usage.

## Outcome

Implementation summary:
- Added shared encoding contracts and detector in `CSharpFar.Core.Text`:
  `TextEncodingSelection`, `EncodingDetectionResult`, `TextEncodingDetector`, and a viewer
  encoding catalog. The detector handles UTF-8/UTF-16 BOMs, UTF-16-looking content,
  binary-looking samples, valid UTF-8 without BOM, and Windows ANSI fallback.
- `LineScanner` now uses the shared detector from a bounded first-block sample and accepts
  automatic or explicit encoding selection. UTF-16 newline scanning was also fixed for
  delimiters split across small block boundaries.
- `LargeFileViewerState` now stores the current encoding selection and can reset
  `LineScanner` plus `SparseLineIndex` without reopening or replacing the byte reader/cache.
- `LargeFileViewer` shows the active text encoding in the header, adds `F8 Encoding`, and
  applies the selected encoding immediately. Explicit encoding selected from hex switches
  the viewer to text; other mode changes preserve the current text/hex mode.
- Added `EncodingSelectionDialog` with automatic, UTF-8, UTF-16 LE/BE, Windows ANSI,
  Windows-1251, Windows-1252, and CP866 options.
- Updated the encoding dialog to render through a single `ScreenRenderer.BeginFrame`
  pass. Cursor movement now applies a live preview and redraws the viewer under the modal
  before the dialog is painted, so the user can inspect the selected code page without
  closing the picker. `Esc`/`F10` restores the original encoding and view mode.
- `FileSystemSearchService` and `TextFileReader` now use the same detector. F3 still uses
  the streaming viewer path and does not use `TextFileReader.MaxFileSizeBytes`.
- Updated README viewer text and added focused tests for detector behavior, explicit
  scanner encodings, F8 selection, live preview before `Enter`, binary-to-text explicit
  selection, and search decoding.

  remains bounded.
- Manual QA from the spec was not run in a real terminal during this implementation.
