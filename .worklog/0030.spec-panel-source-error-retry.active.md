# 0030 SPEC Panel source error retry

Type: spec

Replaces:
- none

Related:
- 0019
- 0029

## Goal

Show panel-source read errors inside the affected panel and let the user retry
the failed panel refresh after the external problem is fixed.

## Context

The SFTP provider from spec 0029 can fail after the panel is already open: a
connection may drop, authentication may expire, a remote directory may become
unavailable, or the server may reject an operation. Local panels have similar
panel-load failures when a removable drive or configured path is unavailable.

Panel-directed actions should leave the panel visible with a clear error state
instead of crashing or silently keeping stale contents. File operation errors,
including copy failures, belong to the existing file operation/progress flow and
must not be converted into panel retry UI.

## Scope

1. Add explicit panel error state for failed directory loads.
2. Render a readable error message inside the affected panel.
3. Add a panel retry command, shown as `Retry`, that repeats the failed panel
   load or refresh.
4. Convert SFTP connection, network, permission, and remote path failures from
   panel enumeration/refresh into panel error state.
5. Keep file operation errors, including copy and move failures, in the file
   operation mechanism.
6. Keep local panel retry behavior working for unavailable local paths or
   removable drives.

## Non-goals

1. Automatic retry loops.
2. Retry policy settings or retry delays.
3. Retrying copy/move/delete failures from the panel error UI.
4. SFTP-to-SFTP copy support.
5. A generic third-party plugin error framework.
6. Compatibility wrappers for old path-only panel APIs.

## Plan

1. Inspect the current provider load and panel rendering paths.
2. Add a small error model to panel state or the panel view model.
3. Update panel loading so provider enumeration failures produce the error
   model and remember enough location data for retry.
4. Render an empty/error panel body with a `Retry` affordance.
5. Wire retry input to rerun the same panel load.
6. Add focused tests for local missing path and fake/SFTP-style provider errors.

## Pause points

- Stop if retry requires changing file operation error flow.
- Stop if the panel state needs a broad source model rewrite beyond the current
  source-aware implementation.

## Acceptance criteria

- A failed panel directory load shows an error message inside that panel.
- `Retry` refreshes the same panel location and replaces the error with normal
  contents after the source becomes available.
- SFTP panel enumeration failures are user-readable and do not crash the app.
- Copy/move errors are still reported by the file operation mechanism, not by
  the panel retry UI.
- Existing successful local and SFTP panel refresh behavior still works.

## Outcome

Implementation summary:
- Added `PanelLoadError` to panel state and try-load/try-refresh paths in
  `PanelController`.
- Changed panel-directed local drive changes, SFTP panel opens, directory
  navigation, parent navigation, and refresh to put failed directory loads into
  the affected panel instead of showing a modal error or silently ignoring the
  failure.
- Rendered panel load errors in both full and brief panel modes with the error
  message and a clickable `[ Retry ]` affordance.
- Changed the panel menu item from `Refresh` to `Retry` while a panel load error
  is present.
- Wrapped SFTP connect, enumeration, item lookup, write/read open, directory
  create/delete/rename, and stream read/write failures into user-readable
  `IOException` or `UnauthorizedAccessException` values so panel refresh and
  file operation flows receive clear errors.
- Added focused tests for local missing-source retry, fake SFTP-style provider
  retry, error rendering, and Retry hit testing.
