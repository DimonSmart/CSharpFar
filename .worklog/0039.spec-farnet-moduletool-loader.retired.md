# 0039 SPEC FarNet ModuleTool loader

Type: spec

Replaces:
- none

Related:
- 0037

## Goal

Allow CSharpFar to discover and invoke simple existing FarNet modules that expose
`ModuleTool` actions, without loading the real FarNet host.

## Context

CSharpFar already has a Far-like plugin host, but its plugins are native
`ICSharpFarPlugin` implementations. FarNet modules are .NET assemblies compiled
against `FarNet.dll` and registered by public module action classes such as
`ModuleTool`.

The first useful compatibility slice is narrow binary loading for simple tools.
It should prove assembly binding, menu projection, invocation, and diagnostics
before attempting FarNet panels, explorers, commands, editors, viewers, menus,
or full dialog APIs.

## Scope

- Add a CSharpFar-owned compatibility assembly named `FarNet`.
- Add a FarNet loader plugin that scans
  `AppContext.BaseDirectory\FarNet\Modules\<moduleName>\<moduleName>.dll`.
- Discover public non-abstract `ModuleTool` classes with `ModuleToolAttribute`.
- Project panel-menu tools into the CSharpFar `Plugins` menu and disk-menu tools
  into `Alt+F1`/`Alt+F2`.
- Invoke each tool with a fresh instance and `ModuleToolEventArgs` matching the
  menu origin.
- Implement minimal `Far.Api` support for messages, input, current directory,
  module managers, temporary file names, and error display.
- Surface unsupported FarNet APIs as clear plugin failures.

## Non-goals

- Full FarNet compatibility.
- Loading real `FarNet.dll`.
- Supporting FarNet `Panel`, `Explorer`, `ModuleCommand`, `ModuleHost`,
  editor, viewer, macro, menu, dialog, or dynamic registration APIs.
- Sandboxing untrusted modules.
- Supporting modules compiled for .NET Framework or FarNet versions newer than
  the v1 shim.

## Plan

1. Create `CSharpFar.FarNetCompat` with output assembly name `FarNet` and v1
   shim types needed by `ModuleTool`.
2. Extend reusable UI services with message buttons and single-line input.
3. Create `CSharpFar.Plugin.FarNet` with module scanning, isolated load
   contexts, menu projection, and invocation.
4. Wire the FarNet plugin into `Program.cs` and `CSharpFar.slnx`.
5. Add test fixture modules and focused tests for discovery, menu mapping,
   invocation, diagnostics, and failure cases.
6. Run repository checks.

## Pause points

- Stop if the compatibility shim needs to expose app-owned types or reference
  `CSharpFar.App`.
- Stop if real FarNet panel support becomes necessary for the first useful
  test; keep it as a separate follow-up instead.
- Stop if assembly binding would load the real `FarNet.dll`.

## Acceptance criteria

- Missing `FarNet\Modules` produces no plugin menu items and no startup error.
- A simple FarNet `ModuleTool` DLL is discovered and can run from the top
  `Plugins` menu.
- Disk-enabled tools can run from both left and right disk menus with correct
  `IsLeft` values.
- Duplicate action ids, invalid ids, missing dependencies, and unsupported APIs
  fail with controlled diagnostics.
- Unsupported FarNet action kinds and unsupported menu areas are ignored for v1.
- Existing SFTP/FTP plugins and local file-manager behavior remain unchanged.

## Outcome

Implementation summary:
- Added `CSharpFar.FarNetCompat`, a CSharpFar-owned `FarNet.dll` shim pinned to
  assembly version `10.0.30.0`.
- Added minimal FarNet `ModuleTool` API surface, `Far.Api`, message/input
  support, module managers, module data folders, and explicit
  `FarNetUnsupportedApiException` failures for unsupported APIs.
- Added `CSharpFar.Plugin.FarNet`, which scans
  `FarNet\Modules\<moduleName>\<moduleName>.dll`, loads each module in its own
  collectible `AssemblyLoadContext`, binds `FarNet` references to the shim, and
  resolves non-framework dependencies only from the module directory.
- Projected supported `ModuleToolOptions.Panels` tools into the top `Plugins`
  menu and `ModuleToolOptions.Disk` tools into disk menus.
- Wired the FarNet plugin into `Program.cs`.
- Extended `PluginUiServices` and `MessageDialog` with button-returning message
  support and added a simple FarNet input dialog.
- Added fixture FarNet module/dependency projects and `Spec039` tests for
  discovery, invocation, disk side mapping, duplicate/invalid ids, unsupported
  APIs, missing dependencies, local real/fake `FarNet.dll` avoidance, and newer
  FarNet version rejection.

Deviations:
- The loader records missing non-framework dependency diagnostics eagerly by
  inspecting assembly references. This is stricter than waiting for reflection
  or invocation to fail and matches the v1 directory-local dependency rule.
- The module load contexts are collectible so tests and future reload scenarios
  can release copied DLLs. The application still keeps contexts alive for the
  process lifetime unless the plugin is disposed.

Follow-up:
- Validate against selected real FarNet sample modules manually.
- Consider `ModuleCommand` and `Panel`/`Explorer` support only after the v1
  loader behavior is proven with real modules.
