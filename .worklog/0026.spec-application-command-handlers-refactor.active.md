# 0026 SPEC Application command handlers refactor

Type: spec

Replaces:
- none

Related:
- 0023
- 0025

## Goal

Reduce `Application` command responsibility by moving concrete user command
handlers into dedicated command files without changing behavior.

## Context

Worklog 0025 moved function-key metadata out of `Application`, but command
execution still lives in large `Application` switch blocks and `Handle*`
methods. Architecture review 0023 identifies command dispatch and application
orchestration as part of the main `Application` hotspot.

## Scope

- Add command abstractions, command context, and a command registry.
- Route function-key and menu command ids through the same registry.
- Move concrete function-key, menu, settings, panel, history, search, file, and
  volume command bodies into separate command files.
- Keep shared helpers and low-level input/rendering orchestration outside command
  files.
- Add focused command registry/dispatcher tests.

## Non-goals

- Do not change runtime behavior.
- Do not change `Program.cs`.
- Do not introduce partial `Application` files.
- Do not refactor command completion, hidden console underlay, mouse input,
  rendering, or file operation internals beyond shared helpers needed by
  commands.
- Do not add configurable command bindings or compatibility shims.

## Plan

1. Add `CSharpFar.App.Commands` abstractions, context, and registry.
2. Wire function-key command dispatch through the registry.
3. Move menu command dispatch through the registry.
4. Move concrete command bodies into one command file per user command.
5. Keep shared helpers on context or small helper services.
6. Update tests and run repository checks.

## Pause points

- Stop if command extraction requires changing input routing, rendering, current
  console underlay, or file operation semantics.

## Acceptance criteria

- `Application` no longer contains concrete command handler bodies for the
  extracted user commands.
- Function-key and menu command ids execute through one command registry.
- Existing behavior tests keep passing.
- Focused tests cover registry coverage and invalid menu arguments.
- Repository checks pass.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

## Outcome

Implementation summary:
- Added `CSharpFar.App.Commands` with command abstractions, result model,
  registry, application context, default registration, and shared command
  helpers.
- Routed function-key commands, menu commands, Ctrl+S settings, Alt+1/Alt+2
  view mode changes, and empty-command Enter through the command registry.
- Moved concrete file, history, search, panel, settings, drive, sort, app, and
  open-current-item command bodies into dedicated command files.
- Moved file operation progress/conflict/pause UI flow into
  `FileOperationUiRunner`, leaving low-level input, rendering, command
  completion, hidden console underlay, search-results refresh, and panel
  orchestration in `Application`.
- Updated `Spec008MenuTests` to use the registry dispatcher instead of
  reflecting `Application.ExecuteMenuCommand`.
- Added `ApplicationCommandRegistryTests` for registry coverage and failed menu
  results on missing arguments / unknown command ids.

Files changed:
- `src/CSharpFar.App/Application.cs`
- `src/CSharpFar.App/Commands/*`
- `tests/CSharpFar.Tests/Spec008MenuTests.cs`
- `tests/CSharpFar.Tests/ApplicationCommandRegistryTests.cs`
- Existing related 0024/0025 changes remain present and were not reverted.

Verification:
- `dotnet build`
- `dotnet test`

Deviations:
- None known. The refactor keeps compatibility shims out of scope and preserves
  the existing menu/function-key behavior through the new registry.

Follow-up:
- Consider extracting search-results panel refresh helpers only in a separate
  focused worklog if `Application` remains too broad after this command split.
