# 0015 SPIKE Console resize scroll artifact

Type: spike

Replaces:
- none

Related:
- 0002 (render buffering strategy)

## Goal

Понять и устранить артефакт прокрутки консольного буфера при изменении размера окна.
При resize пользователь видит искажение экрана: содержимое сдвигается на одну строку,
чаще всего страдает строка функциональных клавиш (нижняя строка).

## Hypothesis

Между захватом размера/координат viewport'а в начале рендеринга и финальной установкой
курсора viewport успевает сдвинуться. Windows интерпретирует новую позицию курсора
как «за пределами» и скроллит буфер, чтобы курсор оказался в видимой зоне.

## Experiment

### Шаг 1 — Временное логирование

Добавлено файловое логирование (`resize_debug.log`) в трёх точках:
- `Application.cs` — события resize и попытки рендеринга
- `ScreenRenderer.cs` — начало/конец/прерывание фрейма
- `SystemConsoleDriver.cs` — WriteAt, SetCursorPosition, OUT-OF-WINDOW записи

**Вывод:** логи подтвердили, что артефакты возникают именно при resize, а не при обычном
рендеринге.

---

### Шаг 2 — Коалесцирование resize-событий

**Попытка:** `DrainResizeEvents()` — после успешного рендера вычитываем из очереди все
накопившиеся WINDOW_BUFFER_SIZE_EVENT до полного опустошения.

**Почему не помогло:** resize-события продолжали приходить во время следующего рендера.
Дрейнинг помогал только если пользователь прекратил тянуть окно — при непрерывном
изменении размера следующий resize приходил раньше, чем завершался рендер.

---

### Шаг 3 — Прерывание рендера при изменении размера

**Попытка:** в `ScreenRenderer.FlushFrame()` перед каждой строкой проверяем
`_driver.GetSize()`. Если размер изменился — устанавливаем `FrameWasInterrupted = true`
и выходим из flush без записи в буфер. `Application.RenderUntilStable()` при
`FrameWasInterrupted == true` запускает рендер заново.

**Почему не помогло полностью:**
- `GetSize()` использовал `Console.WindowWidth/Height` — кэшированные значения CLR,
  которые обновляются с задержкой. Проверка могла пройти успешно, хотя viewport уже
  сдвинулся.
- Прерывание прекращало запись, но несколько строк до прерывания уже были записаны в
  «старый» viewport, а затем SetCursorPosition вызывался с новыми координатами.

---

### Шаг 4 — Win32-атомарное позиционирование курсора

**Попытка:** заменили `Console.SetCursorPosition` на `Win32ConsoleApi.TrySetCursorPosition`,
который читает актуальный `srWindow` через `GetConsoleScreenBufferInfo` и пропускает вызов,
если позиция выходит за пределы.

**Почему не помогло полностью:** между чтением `srWindow` внутри `TrySetCursorPosition` и
собственно `SetConsoleCursorPosition` viewport мог ещё раз сдвинуться (TOCTOU). Кроме того,
координата вычислялась относительно нового viewport, а фрейм рисовался со старым.

---

### Шаг 5 — Единый snapshot размера в `BeginFrame`

**Попытка:** добавили `FrameSize` — `ConsoleSize`, захваченный в `BeginFrame()`. Все
рендереры используют `_screen.FrameSize` вместо повторного `GetSize()`.

**Почему не помогло полностью:** размер захватывался атомарно, но **viewport origin**
(абсолютные координаты `srWindow.Left/Top`) всё ещё читался повторно при SetCursorPosition.

---

### Шаг 6 — Win32-атомарный `GetSize()` и `WriteAtWindows()`

**Попытка:**
- `GetSize()` переведён на `GetConsoleScreenBufferInfo.srWindow` (атомарно).
- `WriteAtWindows()` переведён на единственный вызов `GetConsoleScreenBufferInfo`:
  координаты и bounds-check берутся из одного и того же `sbi`.
- Аналогично исправлены `CaptureWindows()` и `RestoreWindows()`.

**Лог показал:**

```
WriteAtWindows OUT-OF-WINDOW write=[0,1389-67,1389] window=[0,1340-71,1361]
```

`Console.WindowTop = 1386` (устаревшее), реальный `srWindow.Top = 1340`. Запись шла
в строку 1389, которая на 28 строк ниже видимого окна → скролл.

После исправления `WriteAtWindows` OUT-OF-WINDOW записей больше не было.

**Почему ещё не помогло:** скролл переместился из фазы записи в фазу SetCursorPosition.

---

### Шаг 7 — Разделение логирования: WRITES vs CURSOR scrolled

**Попытка:** разбили `FlushFrame cursor moved` на два отдельных замера:
- `FlushFrame WRITES scrolled` — только запись пикселей
- `FlushFrame CURSOR scrolled` — только SetCursorPosition

**Результат лога:**

```
FlushFrame CURSOR scrolled: cursor=(42,2997) win=[0,2977-97,2998]
                          -> cursor=(21,2997) win=[0,2978-97,2999]
```

Все скроллы — **исключительно CURSOR**, не WRITES. Проблема точно локализована.

---

### Шаг 8 — Атомарный захват viewport origin в `BeginFrame` ✅ ФИНАЛЬНОЕ РЕШЕНИЕ

**Диагноз (TOCTOU):**

Пример из лога:
- Фрейм рисовался при viewport `srWindow.Top = 2977`
- Курсор должен был встать в абсолютную строку `2977 + 21 = 2998`
- `TrySetCursorPosition` читал актуальный `srWindow.Top = 2978` (уже сдвинулся)
- Вычислял `absY = 2978 + 21 = 2999 = srWindow.Bottom`
- Это нижняя граница viewport → Windows скроллировал буфер

**Решение:**

1. Добавили `IConsoleDriver.GetViewportOrigin()` — атомарно возвращает `(Left, Top)` из
   одного вызова `GetConsoleScreenBufferInfo`.
2. В `BeginFrame()` захватываем `_frameViewportOrigin = _driver.GetViewportOrigin()` —
   в тот же момент что и `_frameSize`.
3. В `EndFrame()` вычисляем абсолютную позицию курсора:
   ```csharp
   int absX = _frameViewportOrigin.Left + _pendingCursorX;
   int absY = _frameViewportOrigin.Top  + _pendingCursorY;
   _driver.SetCursorPositionAbsolute(absX, absY);
   ```
4. `SetCursorPositionAbsolute` принимает уже готовые абсолютные координаты, читает
   актуальный viewport только для bounds-check (не для вычисления координат) и при
   несовпадении просто пропускает вызов без скролла.

**Почему это работает:** координаты курсора вычислены относительно того же viewport,
что использовался при рендеринге. Если viewport успевает сдвинуться между BeginFrame и
EndFrame — курсор просто не ставится (безопасно, следующий рендер поставит его правильно).

## Constraints

- Не менять протокол рендеринга (buffered frame model).
- Не использовать ENABLE_WRAP_AT_EOL_OUTPUT отключение как обходной путь.
- Логирование должно оставаться non-blocking и не ломать тесты.

## Done when

- `FlushFrame CURSOR scrolled` не появляется в логах при непрерывном resize.
- `FlushFrame WRITES scrolled` не появляется в логах.
- Все тесты `ScreenRendererTests` проходят (14/14).
- Строка функциональных клавиш не искажается при изменении размера окна.

## Result

Причина установлена точно: **TOCTOU между захватом viewport в начале рендера и повторным
чтением viewport при SetCursorPosition**. Конкретно: viewport успевал сдвинуться на 1
строку, `absY` попадал в `srWindow.Bottom`, Windows скроллировал буфер.

Исправление: атомарный захват `(Left, Top)` viewport в `BeginFrame`, использование этих
координат для вычисления абсолютной позиции курсора в `EndFrame`, bounds-check с актуальным
viewport без пересчёта координат.

Все 14 тестов `ScreenRendererTests` прошли после исправления.

## Recommendation

1. Оставить временное логирование (`resize_debug.log`) до визуального подтверждения
   отсутствия артефактов на реальном приложении.
2. После подтверждения — удалить весь logging-код из `Application.cs`,
   `ScreenRenderer.cs`, `SystemConsoleDriver.cs`.
3. Добавить тест, проверяющий что `SetCursorPositionAbsolute` вызывается с координатами
   относительно viewport'а захваченного в `BeginFrame`, а не текущего.

## Follow-up

- Удалить временное логирование (отдельный коммит после визуального подтверждения).
- Рассмотреть добавление теста на `_frameViewportOrigin` в `ScreenRendererTests`.
- Проверить нет ли аналогичного TOCTOU в `CaptureWindows` / `RestoreWindows` при resize.

## Outcome

Implementation summary:
- Итоговая правка не ограничилась решением из шага 8. Spike верно указал на cursor phase,
  но дополнительная проблема была в том, что `front buffer` считался валидным только по
  размеру. При изменении `srWindow.Left/Top` без изменения ширины/высоты renderer мог
  пропустить полный redraw, и нижняя строка функциональных клавиш оставалась физически
  испорченной.
- Добавлен `ConsoleViewport` как единый снимок `Left/Top/Width/Height`.
- `ScreenRenderer.BeginFrame()` теперь захватывает один viewport snapshot и привязывает
  известный `front buffer` к нему. Изменение origin теперь инвалидирует front buffer и
  форсирует полный кадр.
- Flush кадра проверяет весь viewport, а не только размер. Если viewport меняется до,
  во время или после физических записей, кадр помечается interrupted, front buffer
  сбрасывается, и `Application.RenderUntilStable()` перерисовывает экран.
- Win32 driver получил viewport-bound запись и viewport-bound постановку курсора:
  операции выполняются только если текущий viewport совпадает со snapshot кадра.
- Старые отдельные `GetViewportOrigin()` / `SetCursorPositionAbsolute()` убраны из
  `IConsoleDriver`, чтобы рядом не оставался менее безопасный путь постановки курсора.
- Pending cursor больше не применяется после interrupted frame.
- `DrainResizeEvents()` больше не теряет сохраненные key events для диалогов:
  `ScreenRenderer.ReadKey()` читает pending `KeyConsoleInputEvent`.
- Временное файловое логирование `resize_debug.log` удалено из runtime-кода.
