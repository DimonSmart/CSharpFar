## Goal

Make CSharpFar's managed FarNet compatibility layer measurably converge on the
public `FarNet` 10.0.30 API while keeping `FarNet.JsonKit` 0.5.1 as the first
real module acceptance target.

## Context

`0045` made JsonKit's main panel workflows work in the smoke harness, but the
real application still constructed `FarNetModuleHost` with panel tools disabled
by default. That made `Panel.Open()` fail in ordinary runtime use even though
the method and adapter path existed.

The broader compatibility surface is also not complete. Official `FarNet.dll`
10.0.30 exposes public types and members that are still absent from
`CSharpFar.FarNetCompat`, especially panel lifecycle/navigation, editor, viewer,
history, window, settings, macro, and file-operation event APIs. Future work
needs a testable inventory instead of ad hoc discovery from runtime failures.

## Scope

- Enable FarNet panel opening in the production application path.
- Keep `EnablePanelTools` available as an explicit test opt-out, but make panel
  support the default behavior.
- Add a reflection compatibility gate comparing official FarNet 10.0.30 and
  the CSharpFar shim, with approved planned gaps.
- Use the compatibility gate to drive later implementation of missing panel,
  explorer, editor, viewer, UI, history, settings, job, and macro APIs.
- Preserve CSharpFar's native module/panel/editor domain model. FarNet remains a
  boundary facade, not a restored proxy plugin architecture.

## Non-goals

- Do not implement Far Manager native plugin ABI compatibility.
- Do not implement exact Far macro language compatibility unless a real managed
  FarNet module needs a narrow API hook.
- Do not add JsonKit-specific branches to production code.
- Do not add a separate editor adapter layer before the native editor model can
  back FarNet editor APIs.
- Do not silently no-op unsupported FarNet APIs. Unsupported APIs must remain
  explicit diagnostics or approved planned gaps.

## Plan

1. Fix runtime panel wiring so `Panel.Open()` works with default production
   `FarNetModuleHost` construction.
2. Add a regression test proving default host options can open an official
   FarNet-compiled panel command.
3. Add a reflection inventory test for missing public FarNet types and core
   members against official FarNet 10.0.30.
4. Implement future compatibility in small phases by shrinking the approved gap
   lists:
   - panel/explorer lifecycle and file operations;
   - JsonKit hardening and menu smoke coverage;
   - editor/viewer/UI facade APIs over native CSharpFar models;
   - module manager, settings, interop, jobs, and macro-adjacent APIs.
5. Run repository checks and the real JsonKit smoke harness after each coherent
   phase.

## Pause points

- Stop before adding editor-facing FarNet APIs if they cannot be backed by
  CSharpFar's native editor model.
- Stop before adding macro behavior beyond explicit managed API stubs and
  diagnostics.
- Stop before changing shared CSharpFar module abstractions in a way that affects
  FTP or SFTP panels.

## Acceptance criteria

- The production application creates a FarNet host with panel opening enabled.
- A default `FarNetModuleHostOptions` instance supports `Panel.Open()` for an
  official FarNet-compiled module command.
- The compatibility inventory test records current planned gaps and fails when
  the shim drifts from the approved inventory.
- Existing FarNet loader and JsonKit smoke paths continue to pass.
- `dotnet build CSharpFar.slnx` and `dotnet test CSharpFar.slnx` pass.

## Outcome

Implementation summary:
- Enabled FarNet panel tools by default in `FarNetModuleHostOptions`.
- Made production `Program.cs` construct `FarNetModuleHost` with panel tools
  explicitly enabled, so ordinary runtime `Panel.Open()` follows the same path
  as the JsonKit smoke harness.
- Added a regression test proving default host options can open an official
  FarNet-compiled panel command.
- Added a reflection compatibility inventory test against official FarNet
  10.0.30. The test records current approved missing public types and core
  members so future compatibility work has a measurable gate.
- Changed parent-directory rendering so the "show `..` in root folders" setting
  applies to local disk roots, while plugin/module sources show `..` when their
  source contract exposes a parent path.
- Added FarNet panel parent navigation so child panels can open their parent
  panel through the generated `..` entry.

Deviations:
- This pass does not implement the full missing FarNet 10.0.30 API surface.
  It fixes the blocking production `Panel.Open()` wiring and adds the inventory
  gate that future phases must shrink.
- The parent-navigation fix is source-contract based, not JsonKit-specific.
  FTP/SFTP roots still have no `..` unless their source returns a parent path.

Follow-up:
- Implement panel/explorer lifecycle and file-operation gaps from the approved
  missing-member list.
- Add editor/viewer/UI APIs only where they can be backed by native CSharpFar
  editor/viewer models.
- Expand JsonKit smoke coverage for menu actions and item movement.
