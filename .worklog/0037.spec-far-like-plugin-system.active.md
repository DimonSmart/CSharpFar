## Goal

Move SFTP and FTP/FTPS out of the main application and into plugin projects,
while shaping the plugin host after Far Manager's plugin model:

- plugins publish cheap metadata and menu entries;
- the host calls a single Open-like entry point with an origin;
- plugins can open a file-panel source or perform an action without opening a
  panel;
- the host provides UI, settings, panel, and message services through a startup
  context;
- protocol models and protocol NuGet dependencies are owned by the plugin that
  uses them.

This is not binary or source compatibility with Far Manager. The goal is a
similar mental model and responsibility split in C#.

## Context

`Application` currently owns the SFTP and FTP/FTPS workflows directly:
connection managers, connection dialogs, saved connection loading, credential
lookup, host-key/certificate validation, source registration, panel loading, and
menu/drive-selection wiring. Specs `0029` and `0035` implemented SFTP and
FTP/FTPS as built-in providers, which was correct for the first provider model
but now makes the application and `CSharpFar.FileSystem` too broad.

Far Manager uses plugins for commands and virtual file systems. Its public API
is built around a small set of exported functions. The relevant concepts for
CSharpFar are:

- `GetGlobalInfoW` returns plugin identity such as GUID, title, description,
  author, and version.
- `GetPluginInfoW` returns plugin menu entries, disk menu entries,
  configuration menu entries, and command prefixes. Far calls this frequently,
  so it must be cheap.
- `OpenW` receives `OpenInfo` with an `OpenFrom` origin such as plugin menu,
  left/right disk menu, command line, editor, viewer, or file panel.
- A successful panel plugin returns a panel handle. Later panel-specific
  callbacks provide panel metadata, directory items, set-directory behavior,
  copy in/out, delete, create-directory, and input/event handling.
- Far passes services to plugins through `PluginStartupInfo`: menu, dialog,
  message, panel control, screen save/restore, standard functions, help,
  editor/viewer services, and similar host-owned operations.

- https://fargroup.github.io/api/ru/structures/globalinfo.html
- https://fargroup.github.io/api/ru/structures/plugininfo.html

Important local facts:

- `ScreenRenderer` is in `CSharpFar.Console`.
- `ConsolePalette`, `PaletteRegistry`, dialog button helpers, dialog frame
  renderers, modal renderers, and dialog styles are in `CSharpFar.App.Rendering`.
  SFTP/FTP dialogs cannot move to plugin projects while those UI types remain
  app-owned.
- `VolumeSelectionItem` in Core directly references `SftpConnectionInfo` and
  `FtpConnectionInfo`; those protocol models cannot move to plugins until drive
  selection becomes plugin-item based.
- `PanelSourceId` has protocol-specific `Sftp` and `Ftp` helpers in Core. Those
  helpers keep protocol names in Core and should be replaced by a generic plugin
  source id shape.
- `CSharpFar.FileSystem` currently owns the `SSH.NET` and `FluentFTP` package
  references. Those packages should move to the corresponding plugin projects.

## Scope

### Far-like host contract

Add a plugin abstraction that mirrors Far's shape without copying the C ABI:

```csharp
public interface ICSharpFarPlugin
{
    PluginGlobalInfo GetGlobalInfo();
    PluginInfo GetPluginInfo();
    void SetStartupInfo(PluginStartupInfo startupInfo);
    PluginOpenResult Open(PluginOpenInfo openInfo);
}
```

Core concepts:

- `PluginGlobalInfo`: plugin GUID, title, description, author, version.
- `PluginInfo`: flags, plugin-menu items, disk-menu items, config-menu items,
  command prefixes.
- `PluginMenuItem`: item GUID, text, optional hot key, optional flags.
- `PluginOpenInfo`: `OpenFrom`, selected item GUID, panel side, command line
  text, and optional payload.
- `PluginOpenResult`: no panel/action completed, failed with message, or
  opened a plugin panel.
- `PluginPanelInfo`: Far-like panel metadata such as format, title, current
  directory, shortcut data, optional free size, and optional key bar labels.

Use `Guid` values for plugin and menu-item identity, like Far. Do not use
protocol-specific command id constants in Core for plugin entries.

### Project layout

Add these projects:

- `src/CSharpFar.Ui/CSharpFar.Ui.csproj`
- `src/CSharpFar.Plugin.Abstractions/CSharpFar.Plugin.Abstractions.csproj`
- `src/CSharpFar.Plugin.Sftp/CSharpFar.Plugin.Sftp.csproj`
- `src/CSharpFar.Plugin.Ftp/CSharpFar.Plugin.Ftp.csproj`

Expected dependencies:

- `CSharpFar.Core`: domain contracts and models only.
- `CSharpFar.Console`: console driver and `ScreenRenderer`.
- `CSharpFar.Ui`: reusable Far-like console UI primitives and dialogs, depending
  on `Core` only where menu or model types are truly needed and on
  `CSharpFar.Console` for rendering.
- `CSharpFar.Plugin.Abstractions`: plugin contracts, depending on `Core` and
  `Ui`.
- `CSharpFar.Plugin.Sftp`: SFTP plugin implementation, depending on `Core`,
  `Console`, `Ui`, and `Plugin.Abstractions`; owns `SSH.NET`.
- `CSharpFar.Plugin.Ftp`: FTP/FTPS plugin implementation, depending on `Core`,
  `Console`, `Ui`, and `Plugin.Abstractions`; owns `FluentFTP`.
- `CSharpFar.FileSystem`: local filesystem services and DPAPI credential store;
  no SFTP/FTP protocol packages.
- `CSharpFar.App`: host implementation and composition root; statically
  references the two plugin projects for this variant.

Do not make plugins reference `CSharpFar.App`.

### Shared UI extraction

Move reusable UI infrastructure out of `CSharpFar.App` before moving protocol
dialogs:

- `ConsolePalette`
- `PaletteRegistry`
- `PaletteStyles`
- `FarDialogStyles`
- `WarningDialogStyles`
- `DialogButton`
- `DialogButtonBar`
- `DialogFrameRenderer`
- `ModalDialogRenderer`
- `MessageDialog`
- `ConfirmDialog`

Move only the UI types needed by plugin dialogs and shared application dialogs.
Do not move application orchestration, command logic, menu controller state, or
feature-specific dialogs into `CSharpFar.Ui`.

### Host services

`PluginStartupInfo` should provide host-owned services similar in spirit to
Far's `PluginStartupInfo`, but with C# names:

- UI: message, confirmation, screen, palette, save/restore screen if needed.
- Settings: per-plugin settings directory under the existing config directory.
- Credentials: `ICredentialStore`.
- Panels: open a plugin panel on a specific side, refresh/redraw panels, inspect
  active/passive panel state.
- Commands: optional command-line prefix dispatch.

Keep this contract narrow. Do not expose `Application` or
`ApplicationCommandContext` directly.

### Plugin menu and disk menu

Replace hard-coded SFTP/FTP entries in `DefaultMenuDefinitionProvider` with
plugin-provided items:

- CSharpFar's top `Plugins` menu is the host equivalent of Far's plugin commands
  menu.
- `Alt+F1` and `Alt+F2` drive/source selection consume plugin disk-menu items.
- Selecting a plugin menu item calls `plugin.Open()` with
  `OpenFrom.PluginMenu`.
- Selecting a left/right disk-menu item calls `plugin.Open()` with
  `OpenFrom.LeftDiskMenu` or `OpenFrom.RightDiskMenu`.

To stay close to Far, the disk menu should show plugin entry points, not every
saved connection from Core. Saved sessions belong inside the plugin's own
manager/list. If keeping saved sessions directly in drive selection is still
desired, add it later as a deliberate CSharpFar extension, not as part of the
Far-like extraction.

### Plugin panels

Add `IPluginPanel` as the host-facing panel handle:

```csharp
public interface IPluginPanel : IFilePanelSource, IDisposable
{
    PluginPanelInfo GetOpenPanelInfo();
}
```

The host registers the returned panel with `FilePanelSourceRegistry` and opens
the requested `PanelLocation`. This keeps current provider-aware file
operations working while making the ownership Far-like: the plugin creates and
owns the panel instance, the host displays it and calls source operations.

Replace `PanelSourceId.Sftp(connectionId)` and `PanelSourceId.Ftp(connectionId)`
with a protocol-neutral plugin source id, for example:

```csharp
PanelSourceId.Plugin(pluginGuid, panelId)
```

The string form may remain human-readable, but Core must not need SFTP or FTP
helpers.

### Command-line prefixes

Model Far's `CommandPrefix` as plugin metadata. This plan only needs the data
contract and dispatch shape:

- SFTP may expose `sftp`.
- FTP may expose `ftp` and `ftps`.
- command-line prefix execution calls `Open(OpenFrom.CommandLine, ...)`.

Full command-line grammar is out of scope unless needed to remove existing
SFTP/FTP code from `Application`.

### SFTP plugin

Move into `CSharpFar.Plugin.Sftp`:

- `SftpConnectionInfo`
- `ISftpConnectionStore`
- `SftpConnectionStore`
- `SftpFilePanelSource` as an `IPluginPanel`
- `SftpConnectionDialog`
- `SftpConnectionManagerDialog`
- SFTP open/session command flow
- SSH host-key validation
- `SSH.NET` package reference

The plugin's `GetPluginInfo()` returns:

- one plugin-menu item, such as `SFTP...`;
- one disk-menu item, such as `SFTP`;
- command prefix `sftp`.

The plugin's `Open()` handles:

- plugin menu: show the SFTP connection manager;
- left/right disk menu: show the same manager or a connection picker, then open
  the selected connection in the requested panel;
- command line: parse enough `sftp:` input to open the manager or a connection
  target if implemented.

### FTP/FTPS plugin

Move into `CSharpFar.Plugin.Ftp`:

- `FtpConnectionInfo`
- `FtpConnectionSecurityMode`
- `FtpDataConnectionMode`
- `IFtpConnectionStore`
- `FtpConnectionStore`
- `FtpFilePanelSource` as an `IPluginPanel`
- `FtpConnectionDialog`
- `FtpConnectionManagerDialog`
- FTP/FTPS open/session command flow
- FTPS certificate validation
- `FluentFTP` package reference

The plugin's `GetPluginInfo()` returns:

- one plugin-menu item, such as `FTP/FTPS...`;
- one disk-menu item, such as `FTP/FTPS`;
- command prefixes `ftp:ftps`.

Keep FTP and SFTP in separate plugins for the first extraction. Far's NetBox
combines several network protocols into one plugin, but merging the current
CSharpFar protocol code now would create extra abstraction work unrelated to
removing `Application` responsibilities.

### Application integration

Add an app-side `PluginManager` that:

- receives statically referenced plugins from `Program.cs`;
- calls `SetStartupInfo()`;
- reads `GetGlobalInfo()` and `GetPluginInfo()`;
- validates duplicate plugin GUIDs and duplicate item GUIDs per plugin menu;
- builds menu and disk-menu adapter items;
- dispatches selected menu items to `Open()`;
- registers and opens returned plugin panels.

Update `ApplicationCommandRegistry` so app commands are not rebuilt from a
closed default list after plugins are known. Prefer a constructor or builder
that receives default commands plus plugin host commands.

Update `DriveSelectionCommand` and `VolumeSelectionItem` so Core no longer
contains FTP/SFTP-specific fields or enum values. A drive-selection item should
be either a local volume item or a plugin-open item with plugin/item identity and
target panel side.

### Composition root

`Program.cs` should construct:

- local services;
- credential store;
- plugin instances;
- plugin manager or plugin catalog;
- application host.

For this static-reference variant, there is no scanning of arbitrary plugin
folders and no dynamic assembly loading. The API shape should not prevent a
future loader, but no loader support code should be added now.

### Tests

Update tests to reference plugin projects where they test protocol-specific
types.

Add focused tests for:

- plugin metadata validation;
- plugin menu item projection into the top `Plugins` menu;
- plugin disk item projection into `Alt+F1` and `Alt+F2`;
- `OpenFrom.PluginMenu`, `OpenFrom.LeftDiskMenu`, and `OpenFrom.RightDiskMenu`
  dispatch;
- `VolumeSelectionItem` containing no FTP/SFTP protocol models;
- Core containing no `SftpConnectionInfo`, `FtpConnectionInfo`,
  `ISftpConnectionStore`, or `IFtpConnectionStore`;
- `CSharpFar.FileSystem` containing no `SSH.NET` or `FluentFTP` references;
- SFTP/FTP provider tests moved to plugin test namespace references;
- existing local/provider file-operation tests still passing.

## Non-goals

- Binary compatibility with Far Manager plugins.
- Loading third-party DLLs or assemblies from a `Plugins` directory in this
  implementation pass.
- A Lua/macro plugin layer.
- Editor, viewer, dialog, content-column, archive analysis, or process-event
  plugin APIs.
- Combining SFTP and FTP/FTPS into one NetBox-style plugin.
- Preserving saved SFTP/FTP connections directly in the drive-selection menu
  unless explicitly chosen as a CSharpFar-specific extension.
- Adding compatibility wrappers only to keep old `Application.OpenSftp...` or
  `Application.OpenFtp...` call sites alive.
- Moving all rendering from `CSharpFar.App` to `CSharpFar.Ui`; move only the
  reusable primitives needed by plugins and shared dialogs.

## Plan

1. Create `CSharpFar.Ui` and move only the reusable dialog/rendering primitives
   needed by SFTP/FTP dialogs. Update app and tests to reference the new project.
2. Create `CSharpFar.Plugin.Abstractions` with Far-like metadata, startup,
   open, menu, disk-menu, command-prefix, and plugin-panel contracts.
3. Add app-side `PluginManager`, startup-info implementation, plugin menu
   adapters, disk-menu adapters, and plugin panel opening. Keep built-in SFTP
   and FTP wired the old way until the host layer is tested.
4. Replace hard-coded SFTP/FTP command ids in the top `Plugins` menu with
   plugin-provided menu items.
5. Replace FTP/SFTP-specific `VolumeSelectionItem` and drive-selection logic
   with plugin disk-menu items and OpenFrom dispatch.
6. Add generic plugin source ids and remove protocol-specific helpers from
   `PanelSourceId`.
7. Move SFTP models, store, provider, dialogs, and open flow into
   `CSharpFar.Plugin.Sftp`. Remove SFTP fields and methods from `Application`
   and `ApplicationCommandContext`.
8. Move FTP/FTPS models, store, provider, dialogs, and open flow into
   `CSharpFar.Plugin.Ftp`. Remove FTP fields and methods from `Application`
   and `ApplicationCommandContext`.
9. Move `SSH.NET` and `FluentFTP` package references from
   `CSharpFar.FileSystem` into the plugin projects. Keep
   `System.Security.Cryptography.ProtectedData` with the DPAPI credential store
   unless that store is moved later by a separate decision.
10. Update `Program.cs`, project references, `CSharpFar.slnx`, and test project
    references.
11. Update tests for the new namespaces and plugin host behavior.
12. Run `dotnet build` and `dotnet test`.

## Pause points

- Stop after creating `CSharpFar.Ui` if moving dialog/rendering primitives starts
  pulling application orchestration or command-specific types into the UI
  project.
- Stop after plugin abstractions if the API exposes `Application` or
  `ApplicationCommandContext` directly.
- Stop before changing drive selection if product behavior must keep saved
  connections as first-level drive-menu items.
- Stop before moving both protocols if SFTP extraction reveals that the
  plugin-panel contract is missing required provider behavior.
- Stop if a plugin project needs to reference `CSharpFar.App`.

## Acceptance criteria

- `Application` has no SFTP/FTP connection manager, dialog, validation, store,
  credential, or protocol panel-opening methods.
- `ApplicationCommandContext` has no SFTP/FTP-specific methods or properties.
- `CSharpFar.Core` no longer contains SFTP/FTP connection models or connection
  store interfaces.
- `VolumeSelectionItem` contains no protocol-specific properties or enum values.
- `PanelSourceId` has no SFTP/FTP-specific helper methods.
- `CSharpFar.FileSystem` has no `SSH.NET` or `FluentFTP` package references.
- SFTP and FTP/FTPS are available through plugin-provided entries in the
  `Plugins` menu.
- `Alt+F1` and `Alt+F2` can invoke SFTP and FTP/FTPS through plugin disk-menu
  entries and open the selected connection on the requested side.
- SFTP and FTP/FTPS panels still support browse, view, local-to-remote copy,
  remote-to-local copy, create directory, delete, same-source rename, refresh,
  and provider error display as accepted in specs `0029`, `0030`, and `0035`.
- Rejected host keys and changed FTPS certificates still block connection with
  clear messages.
- Existing local filesystem, search, quick view, command registry, and menu
  behavior tests still pass.

## Verification

Commands:

```bash
dotnet build
dotnet test
```

Manual checks:

1. Open SFTP from the top `Plugins` menu and connect to a test server.
2. Open SFTP from `Alt+F1` and `Alt+F2` and verify the target side is correct.
3. Open FTP/FTPS from the top `Plugins` menu and connect to plain FTP, explicit
   FTPS, and implicit FTPS test servers if available.
4. Open FTP/FTPS from `Alt+F1` and `Alt+F2` and verify the target side is
   correct.
5. Save a SFTP and FTP/FTPS connection, restart, reconnect, and confirm no
   plaintext password appears in metadata.
6. View, copy, create, rename, and delete remote test files for both plugins.
7. Reject a new SFTP host key and a new FTPS certificate and confirm the panel is
   not opened.

## Outcome

Implementation summary:
- Added `CSharpFar.Ui` and moved reusable dialog/rendering primitives there:
  palettes, popup/frame/modal renderers, scrollbars, dialog buttons,
  `MessageDialog`, `ConfirmDialog`, and `SingleLineTextInput`.
- Added `CSharpFar.Plugin.Abstractions` with Far-like global info, plugin info,
  menu items, open info/result, startup info, settings, and plugin panel
  contracts.
- Added static `CSharpFar.Plugin.Sftp` and `CSharpFar.Plugin.Ftp` projects.
  SFTP and FTP/FTPS connection models, stores, dialogs, validation flow, and
  panel sources now live in those projects, and the protocol NuGet packages
  moved with them.
- Added app-side plugin registration/validation and generic menu/disk dispatch.
  The top `Plugins` menu and `Alt+F1`/`Alt+F2` drive selection now consume
  plugin-provided items and call `Open()` with Far-like origins.
- Replaced protocol-specific `PanelSourceId.Sftp/Ftp` with
  `PanelSourceId.Plugin(pluginId, panelId)`.
- Removed FTP/SFTP-specific connection models, store interfaces, volume
  selection fields, command ids, and direct open flows from Core/App/FileSystem.
- Finished the FTP/FTPS extraction into `CSharpFar.Plugin.Ftp`, including the
  FTP connection models/store, FluentFTP-backed panel source, connection
  manager, editor dialog, certificate validation flow, menu/disk-menu metadata,
  and protocol package ownership.
- Aligned the SFTP connection editor with the FTP editor's Far-style modal
  rendering, including the gray dialog background, visible text cursor in
  focused input fields, mouse focus/cursor placement, and checkbox mouse
  toggling.
- Added scrollable form bodies to the SFTP and FTP/FTPS connection editors so
  short consoles keep all settings reachable through the shared scrollbar
  renderer and mouse scrollbar handling.
- Reworked the SFTP and FTP/FTPS saved-connection manager dialogs to use the
  same bottom `DialogButtonBar` behavior as the copy dialog: mouse-clickable
  buttons, keyboard focus movement between buttons, list keyboard navigation,
  list mouse selection, double-click connect, and shared border scrollbars for
  long connection lists.
- When a saved-connection manager has no connections, it now shows only
  `New`/`Cancel`; `Connect`, `Edit`, and `Delete` are hidden and keyboard
  navigation does not move focus to the empty list.
- Added `Shift+Tab` as the reverse control traversal key in SFTP and FTP/FTPS
  connection dialogs and manager dialogs while keeping `Tab` as forward
  traversal.
- Added `Spec037PluginSystemTests` for metadata validation, plugin menu
  projection, disk-menu dispatch, Core protocol cleanup, and FileSystem package
  ownership.
- Added focused SFTP/FTP UI tests for cursor visibility, mouse field focus,
  checkbox toggling, short-console scrollbars, manager button mouse clicks,
  manager button keyboard selection, and double-click connect.

Deviations:
- The implementation keeps plugins statically referenced from `Program.cs`, as
  planned. There is no plugin folder scanning or dynamic assembly loading.
- Command-line prefixes are exposed in metadata, but the app does not yet parse
  typed command prefixes into plugin `Open(CommandLine, ...)` calls. This was
  left as dispatch shape only, matching the limited scope in the spec.
- Saved FTP/SFTP connections are no longer first-level drive menu items. Drive
  selection shows plugin disk-menu entry points only.
- The DPAPI credential entropy string was changed from an SFTP-specific name to
  a protocol-neutral name. Compatibility with already encrypted passwords was
  not preserved because compatibility was not requested.
- The connection manager dialogs expose saved sessions inside the plugin-owned
  manager instead of as first-level drive-menu items, matching the Far-like
  plugin menu model in this spec.

Follow-up:
- Manual checks against real SFTP/FTP/FTPS servers are still needed for
  end-to-end connection, certificate/host-key rejection, file operations, and
  saved-session reconnect flows.
- Add command-line prefix parsing when the command line feature is ready to
  invoke plugins directly.
