# 0038 SPEC History-enabled text input

Type: spec

Replaces:
- none

Related:
- 0018

## Goal

Add optional command-history-style suggestions to reusable single-line text input controls, and document the current control ecosystem so new controls are not duplicated.

## Context

CSharpFar has many one-line input fields in dialogs and plugin dialogs. Several of them should eventually remember recent values without duplicates: when the same value is submitted again, it should become the most recent item.

The application already has command-line history completion and panel quick search overlays. This change should reuse the same rendering conventions: a compact framed list, single border, no padding, and the shared right-border scrollbar style.

## Scope

Implement reusable history support for single-line text input:

1. Keep `SingleLineTextInput` usable without history by default.
2. Add a small state type for optional input history suggestions.
3. Store suggestion items without duplicates, ordered from most recent to oldest.
4. Adding an existing item moves it to the top.
5. Do not store empty or whitespace-only values.
6. Render a dropdown arrow as the rightmost input-field character when history is enabled and the field is wide enough.
7. Clicking or otherwise requesting the arrow opens the history dropdown.
8. Typing the first characters in a history-enabled field opens the dropdown when matching items exist.
9. Suggestions match by prefix against the current input text.
10. Use up to 10 content rows, reduced by available screen height.
11. If there is not enough room for a single-border frame plus at least one item, do not open the dropdown.
12. Draw the dropdown as a single-border popup without padding and with the shared right-border scrollbar.
13. Keep keyboard handling predictable: Up/Down move selection when the dropdown is open, Enter accepts the selected value, Escape closes it.

Create a controls registry:

1. Add a repository document that lists current user-interface controls.
2. For each control, record purpose, owning type/file, important functions, supported interaction features, and known reuse points.
3. Include existing text input, command completion, quick search, list dialogs, button bar, popup/frame renderers, scrollbars, panels, menus, and progress/message dialogs.

## Non-goals

Do not implement now:

1. persistent per-field history storage;
2. wiring history into every dialog field;
3. fuzzy, substring, or case-sensitive mode switches;
4. a new UI framework;
5. mouse dragging for the new dropdown scrollbar;
6. compatibility shims for older input APIs.

## Plan

1. Inspect `SingleLineTextInput`, command completion rendering, quick search rendering, and dialog field usage.
2. Add reusable history state and dropdown rendering in `CSharpFar.Ui`.
3. Extend `SingleLineTextInput` rendering and key handling with optional history state while preserving existing default behavior.
4. Add focused unit tests for duplicate-free recency, arrow rendering, prefix opening, selection, accepting, Escape, and short-console hiding.
5. Add the controls registry document.
6. Run repository checks.

## Pause points

- Stop before changing public dialog behavior broadly; this spec adds reusable support, not mass adoption.
- Stop before adding persistence if no existing per-control storage contract is available.
- Stop before replacing the existing command-line completion flow unless the new reusable code can be adopted without behavior changes.

## Acceptance criteria

- Existing calls to `SingleLineTextInput.HandleKey` and `Render` keep working.
- History-enabled input can keep a duplicate-free recency list.
- Re-adding an existing history item moves it to the top.
- Empty and whitespace-only items are ignored.
- History-enabled rendering shows a down arrow at the right edge and reserves that character from text display.
- Typing a prefix opens the dropdown when matching history entries exist.
- The dropdown does not open when the screen cannot fit a frame and at least one item.
- The dropdown shows at most 10 content rows and shrinks to available height.
- Up and Down move dropdown selection without editing text.
- Enter accepts the selected suggestion into the input buffer.
- Escape closes the dropdown without changing text.
- The dropdown uses a single border, no shadow, no padding, and the shared right-border scrollbar when scrollable.
- A controls registry document exists and describes the current controls and supported functions.

## Outcome

Implementation summary:
- Added `SingleLineTextHistoryState` for duplicate-free, most-recent-first input
  history. Empty and whitespace-only values are ignored, and adding an existing
  value moves it to the top.
- Extended `SingleLineTextInput` with history-aware overloads. Existing callers
  can keep using the old API; history-enabled callers get prefix opening,
  Up/Down selection, Enter accept, Escape close, arrow rendering, arrow hit
  testing, and explicit dropdown opening.
- Added history dropdown rendering through the shared `PopupRenderer` with a
  single border, no shadow, no padding, and right-border scrollbar state.
- Added focused tests for recency ordering, prefix opening, selection, accepting,
  Escape, arrow/dropdown rendering, short-console hiding, and the 10-row cap.
- Added `docs/controls-registry.md` with current controls, responsibilities,
  supported functions, and known reuse points.
- Recorded the durable control-registry rule in `AGENTS.local.md`.
